> [!IMPORTANT] PRD 生命周期状态机（遵循 `rule://prd-change-management`）
> 草稿 → 评审中 → 已评审 → 已发布 → 已归档/已替换；已废弃为任意阶段的终态分支。
> **硬约束**：`已评审` 不可回退 `草稿`；`已发布` 后新需求**只能新开 PRD + supersedes 链**，禁止往已发布 PRD 堆叠新版本功能。变更类型判据（A 实现偏差 / B v1 内微调 / C 跨版本叠加）与决策树见 `rule://prd-change-management`。

## 0. 目标声明

为 sdd-pack 仓库构建一个 **`sdd` 命令行工具**（TypeScript + bun），定位为 sdd-pack **文档生命周期的权威入口**。提供 `propose` / `validate` / `archive` 三个核心子命令（MVP）+ 4 个辅助子命令（完整集），将 `prd-change-management` rule 的判断逻辑程序化，将 `sdd-core/sdd-prd` 技能的操作流程自动化。

## 1. 背景与目标

### 1.1 业务背景

sdd-pack 仓库 v1.2.3 的 PRD（`docs/prd/2026-06-24-sdd-pack.md`）状态行堆叠了 v1.2.0–v1.2.3 共 5 个版本的功能描述在一条状态行，暴露了 sdd-pack 文档生命周期管理的两大核心痛点：

1. **PRD 状态行堆叠**：多个版本功能挤在一条状态行，状态机状态（`已发布`）与版本号（`1.2.3`）与功能描述混排，难以程序化校验
2. **PRD 生命周期操作纯手工**：起草、评审、归档、supersedes 链维护全靠人记，agent 可能遗漏步骤（忘记更新 `docs/index.md`、漏掉双向引用、未规范化状态行）

commit 7c0af19 通过新增 `prd-change-management` rule 解决了"判断走哪条路"（决策树 + 状态机），但**未解决"走完怎么落地、怎么合并"**——rule 是判断指南，不执行文件操作。

### 1.2 产品目标

构建 `sdd` CLI 工具，作为 sdd-pack 文档生命周期的**唯一权威入口**：

| 目标               | 衡量标准                                                                                     |
| ------------------ | -------------------------------------------------------------------------------------------- |
| 文档操作零遗漏     | 模板填充 / index 同步 / supersedes 双向引用 / lore commit 封装全部自动化，agent 不需手动维护 |
| 状态机违规硬拦截   | `sdd validate` 状态机检查 severity=block，违规时拒绝 commit                                  |
| 与现有守门体系正交 | CLI 做结构化检查（可程序化，无需 LLM），三层守门 agent 做语义检查（需 LLM 推理）             |
| 借鉴成熟方案       | 借鉴单一职责子命令 / 文档驱动约束 / schema 校验 / fluid DAG 设计思想（不复制产品）           |

### 1.3 成功指标

- **指标 1**：`docs/prd/2026-06-24-sdd-pack.md` 状态行被 `sdd migrate` 自动清理为单行 `> 状态：已发布 | 发布日期：2026-06-25 | 版本：1.2.3`
- **指标 2**：Phase A（CLI 骨架）实施后，新增 PRD 时**默认**走 `sdd propose` 而非手动 `write` —— 100% 新 PRD 通过 CLI 创建
- **指标 3**：状态机违规测试覆盖 `已评审 → 草稿` / `已发布 → 编辑` / 状态行堆叠 三种 case，全部 block

### 2.2 使用场景

**场景 1：v1.3 新功能起新 PRD（v1 已发布场景）**

```
v1.2.3 已发布(2026-06-25)
↓
开发者: sdd propose --title "sdd-pack-v1.3-cli"
CLI: 创建 docs/prd/2026-06-29-sdd-cli.md（状态：草稿）
开发者: 填写 §1 业务背景 / §3 功能清单 / §8 验收标准
↓
评审: sdd validate → 状态机 + 结构 + 交叉引用全部通过 → 状态改为 评审中
↓
发布: 评审通过 → 状态改为 已发布
归档: v1.4 新需求来时 → sdd archive --reason completed
```

**场景 2：已发布 PRD 增量变更（v1.x 内部微调）**

```
v1.2.3 已发布
↓
开发者: sdd propose --supersedes docs/prd/2026-06-24-sdd-pack.md --title "sdd-pack-v1.2.4-bugfix"
CLI: 创建 docs/prd/2026-06-24-sdd-pack-v1.2.4-bugfix.md（状态：草稿 + Δ 段 + > 替代: 旧 PRD）
开发者: 填写 Δ 段（仅 ADDED/MODIFIED/REMOVED 三表）
↓
sdd archive --merge-delta: Δ 段合并到完整 PRD 后归档
```

**场景 3：状态行堆叠清理（历史欠账）**

```
当前: docs/prd/2026-06-24-sdd-pack.md 状态行堆叠 5 个版本
↓
开发者: sdd migrate docs/prd/2026-06-24-sdd-pack.md --dry-run
CLI: 预览将生成 CHANGELOG-2026-06-24-sdd-pack.md + 规范化状态行
开发者: 确认
↓
sdd migrate: 自动备份 → 生成 CHANGELOG → 规范化状态行 → 验证 → 输出建议 commit 命令
```

### 3.2 详细功能描述

#### 3.2.1 `sdd validate`（核心子命令）

**功能说明**：校验文档结构完整性 + 状态机合规性 + 交叉引用一致性。是 `docs-check.sh` 的超集 + 状态机校验。

**输入/前置条件**：

- 仓库根目录存在 `docs/` 目录
- 可选参数：`[path]`（文件/目录路径，省略则全量 `docs/`）、`--staged`（仅 staged 变更）、`--severity <warn|error|block>`、`--json`

**处理逻辑**：

1. 解析参数，确定校验范围
2. 加载所有 PRD/Phase/Spec 文件
3. 执行 10 项检查（见验收标准）
4. 按 severity 聚合错误与警告
5. 输出结果（人读或 JSON）

**输出/后置条件**：

- 退出码：0（pass/warn）/ 1（error）/ 2（block）
- stdout：人读格式或 JSON 格式

**异常处理**：

- 文档解析失败 → 报 error（结构问题）
- 状态机非法迁移 → 报 block（硬违规）
- 文件不存在 → 报 error（路径问题）

#### 3.2.2 `sdd propose`

**功能说明**：创建新 PRD 文件或为已发布 PRD 创建 delta 变更。

**输入**：`--spec <path>`（从 spec 提纯）/ `--supersedes <prd-path>`（标记替代旧 PRD）/ `--title <name>` / `--type <full|delta>`（默认 full，`--supersedes` 时默认 delta）/ `--dry-run`

**处理逻辑**：

1. 校验前置条件（`--supersedes` 指向的文件存在且状态为 `已发布`）
2. 选择模板（full 11 节 / delta 仅 Δ 段 + header）
3. 填充模板（`> 替代:` / `> 对应阶段:` 字段自动生成）
4. 写入文件（`docs/prd/YYYY-MM-DD-<name>.md`）
5. 触发 `sdd validate <new-file>`（仅 error 及以上阻止保存）

**输出**：新文件路径 + 下一步建议

#### 3.2.3 `sdd archive`

**功能说明**：归档 PRD。对 delta 型 PRD 可选合并 delta 到完整 PRD。同步更新 `docs/index.md`、supersedes 链、Phase 引用。

**输入**：`<prd-path>` / `--reason <completed|replaced|abandoned>` / `--merge-delta` / `--dry-run` / `--no-commit` / `--new-prd <path>`

**处理逻辑**：

1. 校验前置条件（PRD 存在 / 状态合法 / `--new-prd` 指向的文件存在）
2. 自动执行 `sdd validate`（不通过则拒绝归档）
3. 按 reason 执行归档：
   - `completed`：移动到 `docs/prd/archive/`
   - `replaced`：原地不动，状态改 `已替换`，设置双向 supersedes
   - `abandoned`：原地不动，状态改 `已废弃`
4. 更新所有交叉引用（Phase 的 `> 对应 PRD:`、其他 PRD 的 `> 替代:`）
5. 更新 `docs/index.md`（活跃表 → 归档分区）
6. 封装 `lore commit`（除非 `--no-commit`）

## 4. 非功能需求

### 4.1 性能要求

- `sdd validate` 在 sdd-pack 仓库规模（~30 docs 文件）下 < 100ms
- `sdd propose` 创建文件 < 500ms
- `sdd archive` 完整归档流程（含 lore commit）< 2s

### 4.2 安全要求

- CLI 不修改 git 历史（不执行 `git reset` / `git rebase`）
- CLI 不向网络发送数据（纯本地文件操作）
- CLI 输出不包含敏感信息（lore commit trailer 不含 token / credential）

### 4.3 可用性要求

- CLI 子命令有 `--help` 帮助文本
- 错误信息人类可读（不只是退出码 + JSON）
- `--dry-run` 模式可预览所有破坏性操作

## 5. 数据需求

### 5.1 数据模型

CLI 不引入新数据结构，复用现有：

- `docs/prd/*.md`（PRD 文件）—— 状态机迁移的源数据
- `docs/phase/*.md`（Phase 文件）—— 交叉引用校验的源数据
- `docs/spec/*.md`（Spec 文件）—— `sdd propose --spec` 的输入
- `docs/index.md`（文档索引）—— 自动更新
- `docs/prd/archive/*.md`（归档 PRD）—— 归档操作的目标位置

### 5.2 数据迁移

不涉及。CLI 操作现有文件结构，不迁移数据（除 `sdd migrate` 子命令处理状态行堆叠的局部清理）。

## 6. 界面需求

### 6.1 命令结构

```
sdd <command> [options] [args]
```

命令通过 `arg` 风格的子命令树实现。

### 6.2 输出格式

- **人读格式**（默认）：彩色终端输出 + 错误列表
- **JSON 格式**（`--json`）：结构化输出，供 hook/CI 解析

### 6.3 帮助系统

每个子命令支持 `--help`，输出 usage + options + examples。

## 7. 集成需求

### 7.1 内部系统集成

| 系统名称                     | 集成方式           | 数据流向                | 说明                                                               |
| ---------------------------- | ------------------ | ----------------------- | ------------------------------------------------------------------ |
| `lore` CLI                   | spawn subprocess   | 双向（调用 + 解析输出） | `lore-wrapper.ts` 封装 lore commit / lore why                      |
| omp hook runtime             | 子进程调用         | 双向                    | `hooks/index.ts` 在 commit 拦截中调 `sdd validate --staged --json` |
| `prd-change-management` rule | 程序化（TS 实现）  | 单向（rule → CLI）      | 状态机逻辑从 rule §3 翻译为 TS 代码                                |
| `docs-check.sh`              | 内部调用或移植     | 单向（CLI 调用 shell）  | 4 项检查保留；CLI 是超集                                           |
| `lore-protocol` rule         | 调用 `lore commit` | 单向（CLI → lore）      | 遵守 lore trailer JSON schema                                      |

### 7.2 外部系统集成

不涉及外部 API。CLI 是纯本地工具。

## 8. 验收标准

### 8.1 功能验收

#### `sdd validate` 校验项

- [ ] PRD ↔ Phase 双向引用（docs-check.sh §1，severity=error）
- [ ] 回指格式规范（docs-check.sh §2，severity=error）
- [ ] index.md 覆盖度（docs-check.sh §3，severity=error）
- [ ] 相对路径链接有效性（docs-check.sh §4，severity=warn）
- [ ] 状态机合规性（severity=block）
- [ ] supersedes 链完整性（severity=error）
- [ ] 命名规范（severity=warn）
- [ ] 状态行格式（severity=error）—— 禁止堆叠多版本
- [ ] 归档文件位置（severity=warn）
- [ ] 必需章节完整性（severity=error）—— PRD 含 §1-§5

#### `sdd propose` 验收

- [ ] `sdd propose --spec docs/spec/<name>.md` 创建 full PRD
- [ ] `sdd propose --supersedes <path> --title X` 创建 delta PRD（含 Δ 段 + `> 替代:` 字段）
- [ ] `sdd propose --dry-run` 仅打印不写入
- [ ] `--supersedes` 指向非 `已发布` PRD 时报错退出码 1

#### `sdd archive` 验收

- [ ] `sdd archive <path> --reason completed` 移动到 `docs/prd/archive/` + 更新状态行 + 更新 index + 封装 lore commit
- [ ] `sdd archive <path> --reason replaced --new-prd <new>` 设置双向 supersedes 引用
- [ ] `sdd archive --merge-delta` 将 delta 段合并到完整 PRD 后归档
- [ ] 归档前自动 `sdd validate` 失败时拒绝归档

### 8.2 非功能验收

- [ ] `sdd validate` 在 sdd-pack 仓库 < 100ms
- [ ] 状态机 `prd-state-machine.ts` 与 `prd-change-management` rule §3 状态机表 100% 一致（单元测试覆盖）
- [ ] CLI 入口是 bash 薄壳（10 行内），转发到 bun + TS
- [ ] docs-check.sh 不删除（CLI 内部可调用其逻辑）
- [ ] hook 修改是增量（仅增加 `sdd validate` 子进程调用，不改其他逻辑）

| ------------------------------------------------ | ---- | ---- | ---------------------------------------------------------------------------------- |
| 范围爆炸：CLI 子命令越加越多                     | 高   | 中   | MVP 严格限定 propose/validate/archive；完整集标记为后续独立任务                    |
| 与 hook 冲突：CLI 校验逻辑与 hook 拦截逻辑不一致 | 高   | 低   | CLI 是 hook 调用的下游（hook → CLI），若不一致以 CLI 为准（单一事实源）            |
| 用户认知负担：又学一个工具                       | 中   | 中   | CLI 是 opt-in（不配 alias 完全不受影响）；README 提供「不用 CLI 怎么做」手动对照表 |
| bun 兼容性：用户环境 bun 版本不兼容              | 中   | 低   | bun 是 omp 运行时依赖；`bin/sdd` wrapper 检查 bun 可用性                           |
| `sdd migrate` 解析失败                           | 中   | 中   | 自动备份 + dry-run + 不可识别格式跳过                                              |
| 性能：大仓库中 `sdd validate` 慢                 | 低   | 低   | 当前仓库 ~30 文件，校验 < 100ms；后续大仓库可加缓存                                |

### 10.2 约束条件

- **不破坏现有结构**：兼容 `docs/prd/` `docs/phase/` `docs/architecture/` 现状
- **与 lore 共存**：CLI 不要求用户放弃 `lore commit`，应能 wrap 它
- **与 hook 共存**：ADR-006 标注 hook 保留（因 `lore-commit-guard` 硬拦截能力），CLI 接入不能扩大 hook 范围
- **不引入 OpenSpec 产品**：仅借鉴其设计思想（单一职责子命令 / 文档驱动 / schema 校验 / fluid DAG）
- **Simplicity First**：MVP 限定 propose/validate/archive 三件套，apply 留给手写代码
- **Brownfield-first**：`sdd migrate` 为现有状态行堆叠提供迁移路径

### 11.1 术语表

### 11.2 设计借鉴来源

本 CLI 借鉴以下设计思想（**不复制产品，不引入外部依赖**）：

| 设计思想         | 说明                                                      |
| ---------------- | --------------------------------------------------------- |
| 单一职责子命令   | 每个子命令职责清晰，参数与退出码可预测                    |
| 文档驱动约束     | spec 是源真理，change 是提议，delta 是桥                  |
| schema 校验      | 通过 `validate` CLI 在提交/归档前做结构检查               |
| fluid DAG        | 无 phase gate，artifacts（proposal/design/tasks）任意顺序 |
| 三级 severity    | warn（提示）/ error（阻塞）/ block（硬违规拒绝）          |
| 渐进严格度       | MVP lite（默认）/ full（高风险场景升级）                  |
| brownfield-first | 提供迁移工具（`sdd migrate`）处理历史欠账                 |
