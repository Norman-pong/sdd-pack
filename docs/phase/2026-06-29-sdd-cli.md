## 1. 阶段目标

### 1.1 阶段定位

本阶段为 sdd-pack 构建 `sdd` CLI 工具（TypeScript + bun），将 `prd-change-management` rule 的判断逻辑程序化，将 `sdd-core`/`sdd-prd` 技能的操作流程自动化，成为 sdd-pack **文档生命周期的权威入口**。

分 3 个 Phase 渐进交付：

| Phase | 定位 | 交付物 | 灰度策略 |
|-------|------|--------|---------|
| **Phase A** | CLI 骨架 | `sdd validate` + `sdd propose` + 核心库 | 内部 dogfooding，hook 未接入 |
| **Phase B** | 门禁集成 | `sdd validate --staged` + hook 拦截 | hook 接入，severity=warn（仅警告不阻塞） |
| **Phase C** | 闭环完善 | `sdd archive` + `sdd migrate` + 完整集命令 | 正式发布，severity=error+block |

### 1.2 阶段目标

- 完成 CLI 全部 7 个子命令（3 MVP + 4 完整集），覆盖 PRD §3.1 全部功能点
- 满足 PRD §0 目标验收开关的 14 条业务/技术/文档验收标准
- 解决 PRD §10.1 列出的 6 项已知风险中的高影响项（范围爆炸、hook 冲突、bun 兼容性）
- 实现「文档操作零遗漏」：模板填充、index 同步、supersedes 双向引用、lore commit 封装全部自动化

### 1.3 完成标准

- [ ] `sdd validate` 对现有仓库 `docs/` 运行无错误（10 项检查全部通过或可解释 warn）
- [ ] 状态机违规测试覆盖 `已评审 → 草稿` / `已发布 → 编辑` / 状态行堆叠 三种 case，全部 block
- [ ] `sdd propose --supersedes` 创建的 PRD 文件包含 Δ 变更摘要段（ADDED/MODIFIED/REMOVED 三表）
- [ ] `sdd archive` 归档 PRD 时正确移动文件、更新状态行、同步 `docs/index.md`、封装 `lore commit`
- [ ] `sdd migrate` 能将 `docs/prd/2026-06-24-sdd-pack.md` 的状态行堆叠自动清理为单行 + CHANGELOG
- [ ] hook 集成后 commit 时 `sdd validate --staged --json` 自动运行，校验失败时 `lore-commit-guard` 拒绝 commit
- [ ] `docs-check.sh` 不删除（CLI 内部可调用其逻辑或移植为 TS）
- [ ] ADR-008 状态从 Proposed → Accepted
- [ ] `docs/prd/_template.md` 新增 Δ 变更摘要段模板
- [ ] `plugins/sdd-pack/README.md` 新增 CLI 安装/使用章节

#### T002: 实现校验引擎（validator.ts）

**任务描述**：
实现 `validator.ts` 校验引擎框架（severity 分级 + 错误聚合），移植 `docs-check.sh` 的 4 项检查逻辑为 TS，新增 6 项检查。共 10 项检查，按 severity 分级。

**10 项检查清单**：

| # | 检查项 | severity | 来源 | 说明 |
|---|--------|----------|------|------|
| 1 | PRD ↔ Phase 双向引用 | error | docs-check.sh §1 | PRD 含 `> 对应阶段:` 链接，Phase 含 `> 对应 PRD:` 链接，互相指向存在 |
| 2 | 回指格式规范 | error | docs-check.sh §2 | `> 对应阶段:` / `> 对应 PRD:` 行格式必须匹配 `^> *对应 (阶段|PRD): *\[` |
| 3 | index.md 覆盖度 | error | docs-check.sh §3 | 所有非模板 PRD/Spec/Phase 已入 `docs/index.md` 对应表格 |
| 4 | 相对路径链接有效性 | warn | docs-check.sh §4 | 文档内 `[text](./relative/path.md)` 链接指向的文件存在 |
| 5 | 状态机合规性 | block | prd-change-management rule §3 | PRD 状态行声明的状态合法，状态迁移不违反状态机表 |
| 6 | supersedes 链完整性 | error | prd-change-management rule §4 | `> 替代:` 指向的文件存在且状态为 `已替换`，双向引用不缺失 |
| 7 | 命名规范 | warn | conventions.md §2 | 文件名格式 `YYYY-MM-DD-<kebab-case>.md` |
| 8 | 状态行格式 | error | 新增 | 禁止堆叠多版本/多状态；强制单行单状态格式 `> 状态：<状态> [\| 发布日期：...] [\| 版本：...]` |
| 9 | 归档文件位置 | warn | archival-mechanism.md | `docs/prd/archive/` 下文件状态必须为 `已归档`；非归档文件不在 archive/ 目录 |
| 10 | 必需章节完整性 | error | conventions.md §3-5 | PRD 至少含 `## 0. 目标声明`、`## 1.`、`## 3. 功能需求`、`## 8. 验收标准` |

**验收标准**：
- [ ] severity 分级正确：warn→exit 0, error→exit 1, block→exit 2
- [ ] docs-check.sh 的 4 项逻辑完整移植为 TS（行为等价，不改变检查语义）
- [ ] 状态机检查：`已评审 → 草稿` 报 block，`已发布 → 任何编辑` 报 block
- [ ] 状态行堆叠检测（`docs/prd/2026-06-24-sdd-pack.md` 当前堆叠行）报 error
- [ ] 每项检查独立可开关（`--rules-only` / `--structure-only`）

#### T003: 实现 `sdd validate` 命令

**任务描述**：
实现 `sdd validate` 子命令：参数解析 → 确定校验范围 → 调用 validator.ts → 输出结果。

**参数**：

| 参数 | 说明 |
|------|------|
| `[path]` | 校验范围（文件/目录路径，省略则全量 `docs/`） |
| `--staged` | 仅校验 git staged 变更涉及的 docs 文件（Phase B 实现） |
| `--severity <warn\|error\|block>` | 校验严格度；默认 `error` |
| `--json` | JSON 输出（供 hook/CI 解析） |
| `--rules-only` | 仅状态机校验（check #5） |
| `--structure-only` | 仅文档结构校验（check #1-#4, #6-#10） |

**退出码**：
- `0`：pass 或仅有 warn（`--severity error` 时 warn 不改变退出码）
- `1`：有 error 级别违规
- `2`：有 block 级别违规

**验收标准**：
- [ ] `sdd validate` 对现有 `docs/` 目录运行，输出 10 项检查结果（人读格式，彩色终端）
- [ ] `sdd validate --json` 输出合法 JSON（含 `status`/`errors`/`warnings`/`checks` 字段）
- [ ] `sdd validate docs/prd/` 仅校验 prd/ 下文件
- [ ] `sdd validate --severity warn` 时所有违规均为警告，exit 0
- [ ] `sdd validate --rules-only` 仅执行 check #5
- [ ] `sdd validate --structure-only` 不执行 check #5
- [ ] 文件不存在时报 error（退出码 1），并给出明确的路径提示

#### T004: 实现 `sdd propose` 命令

**任务描述**：
实现 `sdd propose` 子命令 + `template-engine.ts`。模板引擎根据 `--type` 选择模板（full 11 节 / delta 仅 Δ 段 + header），填充自动字段后写入文件。

**参数**：

| 参数 | 必须 | 说明 |
|------|------|------|
| `--spec <path>` | 否 | 从 spec 文件提纯，创建 full PRD |
| `--supersedes <prd-path>` | 否 | 标记替代旧 PRD（旧 PRD 必须状态为 `已发布`） |
| `--title <name>` | 否* | PRD 名称（未提供则交互式询问或报错） |
| `--type <full\|delta>` | 否 | full（完整 11 节）或 delta（仅 Δ 段）；默认 full，`--supersedes` 时默认 delta |
| `--dry-run` | 否 | 仅打印模板内容不写入 |

**前置校验**：
- `--supersedes` 指向的文件存在且 `doc-parser` 解析的状态为 `已发布`，否则报错退出码 1
- `--title` 生成的路径不冲突（`docs/prd/YYYY-MM-DD-<name>.md` 不存在），否则报错退出码 2

**自动填充字段**：
- `> 替代: [旧 PRD](../prd/<old>.md)`（`--supersedes` 时）
- `> 对应阶段: TBD - 待设计评审后由 sdd-phase 补全`
- 日期：当日 `YYYY-MM-DD`

**验收标准**：
- [ ] `sdd propose --spec docs/spec/<name>.md --title X` 创建 full PRD（含 11 节模板）
- [ ] `sdd propose --supersedes docs/prd/2026-06-24-sdd-pack.md --title "v1.3-cli"` 创建 delta PRD（含 Δ 段 + `> 替代:`）
- [ ] `--supersedes` 指向非 `已发布` PRD 时报错退出码 1（含具体原因说明）
- [ ] `sdd propose --dry-run` 仅打印不写入
- [ ] 创建后自动触发 `sdd validate <new-file>`，有 error 时警告但不阻止（允许草稿有未完成项）
- [ ] delta 型 PRD 模板含 `## Δ 变更摘要` 段，三表（ADDED/MODIFIED/REMOVED）已空但结构完整

#### T007: `hooks/index.ts` 集成 `sdd validate --staged --json`

**任务描述**：
在 `hooks/index.ts` 的 `lore-commit-guard` 拦截逻辑中，增加 `sdd validate --staged --json` 子进程调用。解析 JSON 输出，按 severity 处理：
- `block`（exit 2）→ 硬拦截 commit，`return { block: true, reason: "sdd validate 硬拦截:\n" + errors }`
- `error`（exit 1）→ Phase B 阶段仅警告（灰度），`pi.sendMessage` 提示修复
- `warn` / `pass`（exit 0）→ 无动作，继续

**技术方案**：
```typescript
import { spawnSync } from "bun";

function runSddValidate(): { status: string; errors: string[]; warnings: string[] } {
  const sddBin = "bun";  // 或插件内路径
  const result = spawnSync([
    sddBin, "run", sddCliPath,
    "validate", "--staged", "--json"
  ]);
  return JSON.parse(result.stdout.toString());
}
```

**验收标准**：
- [ ] `lore commit` 时自动触发 `sdd validate --staged --json`
- [ ] 正常文档（无违规）→ 无额外消息，正常继续
- [ ] 有 warn → 打印警告但不阻塞
- [ ] 有 error → Phase B 灰度阶段打印警告（不阻塞），可升级到 block
- [ ] 有 block → 硬拦截，`lore commit` 被拒绝
- [ ] 现有 hook 的其他逻辑（session_start reminder, doc-edit-guard）不受影响
- [ ] 修改是增量式，不改现有 hook 的 4 工厂聚合结构

#### T010: 实现 `sdd archive` 命令

**任务描述**：
实现 `sdd archive` 子命令，依赖 `index-sync.ts`（`docs/index.md` 自动更新）和 `lore-wrapper.ts`（`lore commit` 封装）。

**`index-sync.ts` 职责**：
- 解析 `docs/index.md` 的表格结构（PRD 表、Phase 表、归档分区）
- 添加/移动/更新条目（新增 PRD → 活跃表；归档 PRD → 移到归档分区或标注状态）
- 更新文档统计数字

**`lore-wrapper.ts` 职责**：
- spawn `lore commit` 子进程，传入 commit message + JSON trailer
- 遵守 `lore-protocol` rule 的 trailer JSON schema
- 不调用 `git commit` 裸命令

**archive 参数**：

| 参数 | 必须 | 说明 |
|------|------|------|
| `<prd-path>` | 是 | 目标 PRD 文件路径 |
| `--reason <completed\|replaced\|abandoned>` | 否 | 归档原因（默认 completed） |
| `--merge-delta` | 否 | delta 段合并到完整 PRD（仅 supersedes 型 PRD） |
| `--dry-run` | 否 | 仅打印操作清单不执行 |
| `--no-commit` | 否 | 跳过 lore commit（手动提交） |
| `--new-prd <path>` | 否 | （`--reason replaced` 时）替代本 PRD 的新 PRD 路径 |

**归档操作**：

| reason | 文件位置 | 状态变更 | supersedes |
|--------|---------|---------|------------|
| completed | 移动到 `docs/prd/archive/` | `已归档` | — |
| replaced | 原地不动 | `已替换` | 双向引用（本 PRD `> 已被:` + 新 PRD `> 替代:`） |
| abandoned | 原地不动 | `已废弃` | — |

**自动步骤序列**：
1. 校验前置条件（PRD 存在 / 状态合法 / `--new-prd` 文件存在）
2. 自动执行 `sdd validate <prd-path>`（不通过则拒绝归档，exit 1）
3. 按 reason 修改状态行
4. 移动文件（仅 completed）
5. 更新所有交叉引用（Phase `> 对应 PRD:` / 其他 PRD `> 替代:` 链接）
6. 更新 `docs/index.md`（活跃表 → 归档分区）
7. 封装 `lore commit`（commit message 包含归档原因 + 目标文件）

**验收标准**：
- [ ] `sdd archive docs/prd/<name>.md --reason completed` 移动到 `archive/` + 状态行更新 + index 更新 + lore commit
- [ ] `sdd archive docs/prd/<name>.md --reason replaced --new-prd docs/prd/<new>.md` 设置双向 supersedes（旧 `> 已被:` + 新 `> 替代:`）
- [ ] `sdd archive docs/prd/<name>.md --reason abandoned` 状态改 `已废弃`，文件原地
- [ ] 归档前 validate 不通过 → 拒绝归档并打印 validate 输出
- [ ] `--dry-run` 仅打印操作清单不执行任何写操作
- [ ] `--no-commit` 跳过 lore commit，其他步骤正常执行
- [ ] `--merge-delta`（Phase C 附加验收）合并 delta 段到完整 PRD：消费 ADDED/MODIFIED/REMOVED 行到对应章节，移除 Δ 段

**涉及文档**：
- [PRD §3.2.3 sdd archive](../prd/2026-06-29-sdd-cli.md#323-sdd-archive)
- [设计文档 §2.1 sdd archive](../architecture/sdd-cli-design.md#sdd-archive)
- [设计文档 §4.1 Δ 变更摘要段](../architecture/sdd-cli-design.md#41-新增-Δ-变更摘要段)（merge-delta 逻辑）
- [archival-mechanism.md](../../plugins/sdd-pack/skills/sdd-prd/references/archival-mechanism.md)

**备注**：
`--merge-delta` 是 PRD §3.2.3 的功能，但实现复杂度高（需解析 Δ 段 → 定位目标章节 → 合并内容 → 移除 Δ 段）。如 Phase C 时间紧张，可标为 P1 延后，但本任务默认实现。
