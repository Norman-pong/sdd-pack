# 约束集(阶段 3 产物 · sdd)

> 用法:列出实施时会卡住的点,每点写"必须/禁止/优先"约束,附理由。按 P0/P1/P2 排序。
> 约束只锁**机制**,不锁**实现细节**。例如:"i18n 用 JSON 字段存"(锁机制),不写"字段名是 `title_i18n`"(这是实施阶段的事)。
>
> **路径约定**:阶段 3 产物 `docs/prd/.working/YYYY-MM-DD-<name>/constraint-set.md`,阶段 4 后 P0 约束并入 PRD §4 非功能需求,P1/P2 约束放入 PRD 附录。

## P0 约束（不做就阻塞——MVP 前必修）

| #   | 卡点          | 约束                                                                                                                          | 理由                                                                                   |
| --- | ------------- | ----------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------- |
| 1   | i18n 存储方案 | **必须**：i18n 用 JSON 字段存储（`{ "zh": "...", "en": "..." }`），禁止按语言拆表。**优先**：嵌套 fallback（zh-CN → zh → en） | 按语言拆表会导致 N 语言 × M 表的笛卡尔积；嵌套 fallback 解决区域变体（zh-CN vs zh-TW） |
| 2   | Slug 生成     | **必须**：Slug 用英文标题生成，唯一约束在 DB 层。**禁止**：用自增 ID 当 slug（SEO 差）。**优先**：slug 可手动覆盖             | 唯一性靠 DB unique constraint 保证；手动覆盖用于特殊需求（如数字开头的 slug）          |
| 3   | 并发编辑冲突  | **必须**：用乐观锁（version 字段），冲突时返回 diff 让用户手动合并。**禁止**：悲观锁（长时间锁定影响体验）                    | 实际并发编辑概率低；乐观锁的用户体验更好（不会"卡住"）                                 |
| 4   | 权限模型      | **必须**：用 RBAC（角色-权限表），角色可组合。**禁止**：硬编码权限在业务代码里。**优先**：权限检查中间件统一                  | RBAC 适合中型团队；硬编码权限是技术债（每次加角色都要改代码）                          |
|     |               |                                                                                                                               |                                                                                        |

## P1 约束（早期做——第一个迭代周期内完成）

| #   | 卡点              | 约束                                                                                                                  | 理由                                                                       |
| --- | ----------------- | --------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------- |
| 5   | 错误码统一        | **必须**：业务错误码用 `DOMAIN_CODE` 格式（如 `AUTH_003`），映射→ HTTP status。**禁止**：裸 500 抛给前端              | 前端需要根据错误码做不同处理（重试、跳转、提示）；裸 500 不可 debug        |
| 6   | Design Token 存储 | **必须**：Design Token 用单个 JSON 源，CSS 变量和 Figma 都从它生成。**禁止**：设计稿手动同步→代码                     | 单一源避免 Figma 和代码不同步；设计改一次，token JSON 改一次，两端自动更新 |
| 7   | API 版本化        | **必须**：URL 路径版本（`/api/v1/...`）。**禁止**：Header 版本（debug 困难）。**优先**：v1 → v2 过渡期共存 2 个大版本 | 路径版本调试直观（curl 一眼看出版本）；共存期让前端分批迁移                |
|     |                   |                                                                                                                       |                                                                            |

## P2 约束（可推迟——非 MVP 或依赖 P0/P1 完成后再定）

| #   | 卡点          | 约束                                                                              | 理由                                                                   |
| --- | ------------- | --------------------------------------------------------------------------------- | ---------------------------------------------------------------------- |
| 8   | 版本历史/审计 | **必须**：操作日志写入 event stream，不耦合→业务数据表。**优先**：异步写入        | 审计日志读少写多，独立 event stream 不影响业务表性能；异步避免写锁竞争 |
| 9   | 全文搜索      | **优先**：用 PostgreSQL FTS 而非外部搜索引擎（如 ES）。**禁止**：LIKE '%keyword%' | PG FTS 适合中等数据量（百万级）；ES 引入运维复杂度；LIKE 全表扫        |
| 10  | 媒体文件处理  | **必须**：上传时自动 resize 为多种尺寸 + WebP 转换。**禁止**：原图直接返回给前端  | 前端请求 2000×2000 的图会卡顿；WebP 压缩率高；尺寸按需加载             |
|     |               |                                                                                   |                                                                        |

## 约束的写法规则

### 3 种约束词

| 词       | 含义                         | 示例                                           |
| -------- | ---------------------------- | ---------------------------------------------- |
| **必须** | 硬约束，不讨论               | **必须**：slug 唯一约束在 DB 层                |
| **禁止** | 不能碰的线，碰了就是 bug     | **禁止**：按语言拆 i18n 表                     |
| **优先** | 倾向方案，有更好替代时可讨论 | **优先**：用 PostgreSQL FTS 而非 Elasticsearch |

### 约束 vs 功能的区别(sdd)

- **P0 约束**写入 PRD §4 非功能需求:不可妥协,实施阶段必须遵守
- **P1/P2 约束**写入 PRD 附录:可推迟,版本迭代时可以增删改
- **功能**写入 PRD §3 功能需求:可调整,版本迭代时可以增删改
- 不要把"支持深色模式"写成约束——这是功能,v1 不做可以推迟

### 识别卡点的方法

如果实施阶段会在某个问题上**停下来问"这个怎么做"**——这就是卡点，需要写约束。
典型信号：

- "这里该用什么方案？" → 缺约束
- "这两个方案选哪个？" → 缺约束 + 缺优先说明
- "这个实现对吗？" → 缺"是否做对"的判断标准（也是缺约束）
