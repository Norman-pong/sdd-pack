# 假设清单模板（阶段 3 产物）

> 用法：阶段 3 显性化假设时使用。每条假设必须分类为 **必假定 / 必验证 / 不成立**。
> 这个清单会进入 spec 文档的"未决假设"章节--PRD 工作流阶段 2 会优先质疑这些。

## 假设类型的判别标准

### 必假定（必须接受为前提）

- 推翻后**整个项目重构**（不是改一处，而是重新规划）
- 例子：项目在客户私有化环境部署（推翻 → 改 SaaS → 完全不同的技术栈）
- 例子：v1 必须支持 10 万用户（推翻 → 砍一半用户范围）

### 必验证（实施前必须去确认）

- 推翻后**部分功能/选型**要调整，但不是整个项目重构
- 例子：用户量级（验证后调整容量，不调整方向）
- 例子：客户内部有 SMTP 服务器（验证后调整邮件方案）
- 例子：团队有 React 经验（验证后调整前端选型）

### 不成立（问清楚后发现假设是错的）

- 已经被对话本身证伪了——直接更新 v1 范围
- 例子：用户最初说"我们要做实时协作"，追问后发现"其实 5 分钟延迟也行" → 假设不成立 → 不需要实时协作
- 例子：客户说"必须有 Web 端"，追问后说"其实移动端优先" → 假设不成立 → v1 可以先做移动端

## 挖掘假设的提问技巧

### "如果 X 错了会怎样"

**问法**："如果 {X} 其实不是这样呢？"

适用：用户用了"当然"、"肯定"、"必须"等强确定性词

例：

- "如果用户其实不想用这个功能呢？" → 假设：用户有需求
- "如果客户环境没外网呢？" → 假设：客户有外网
- "如果团队没人会 React 呢？" → 假设：团队能写 React

### "什么会让这件事失败"

**问法**："什么样的情况会让你说'这个项目不该上'？"

适用：用户说"没问题"或"肯定能行"时

例：

- "如果产品上线后没人用，你会觉得是什么导致的？" → 暴露对用户的假设
- "如果项目延后 3 个月，会出什么问题？" → 暴露对时间的假设

### "反过来想"

**问法**："你说 X，那 X 的反面 Y 你怎么看？"

适用：用户表达偏好或方向

例：

- "你说要快速上线——那'慢但稳'的方案你怎么看？"
- "你说要做 SaaS——那'私有化'你会考虑吗？"

### "具体到数字"

**问法**："你说'很多用户'——具体是 100、1 万、还是 100 万？"

适用：用户用了量词但没具体数字

例：

- "性能要快" → "多快？100ms 还是 1s？"
- "用户量大" → "日活 1 千还是 10 万？"

## 假设清单的常见分类

按"假设来源"分类，可以分批挖：

### 用户类假设

- 用户真的有这个需求
- 用户愿意改变当前行为
- 用户会按我们设想的方式使用
- 用户的付费意愿

### 技术类假设

- 团队技能匹配
- 第三方服务可获得（API、库、SDK）
- 部署环境的限制（外网/内网、资源限制）
- 性能要求能达到

### 业务类假设

- 客户/老板的期望
- 竞品的反应
- 商业模式的有效性
- 市场窗口期

### 时间类假设

- 团队投入的时间
- 上下游依赖的时间
- 试错周期

---

## 反模式

- **假设 = "我们没想过"** —— 显性化才是目的，没想过的就是"盲区"
- **所有假设都"必假定"** —— 很多假设其实是"必验证"或"不成立"
- **不验证就假定** —— 必验证的假设如果不验证，到实施阶段会爆发
- **跳过阶段 3** —— 不显性化的假设在实施阶段会突然暴露

---

## 阶段 3 完成标志

- 至少 5-10 个假设被显性化
- 每条假设有类型和后果说明
- "不成立"的假设已更新到 v1 范围
- "必验证"的假设已列入 spec 的"未决假设"章节
- "必假定"的假设已和用户确认接受为前提
