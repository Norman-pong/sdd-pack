# PRD 工作流：spec -> PRD 4 阶段方法论

---

## 1. 自审 spec（阶段 1）

### 1.1 为什么自审比其他阶段都重要

自审是整个流程的**根基**。如果内部矛盾没找出来，后续所有阶段都在"自相矛盾的 spec"基础上构建——阶段 2 的选型质疑会失去目标，阶段 3 的约束补全会方向模糊，阶段 4 的精简会变成"精简一份有 bug 的文档"。

核心洞察：**不是找新东西，是找已写出来但互相打架的东西。** 一段说 v1 做 A，另一段说 v1 不做 A——这就是矛盾。不需要"灵感"，只需要**逐条交叉比对**。

### 1.2 交叉对比的技术

#### 方法：对偶压力法

挑两段互相依赖的章节，逐条对照。如果一段承诺了但另一段没兑现，就是问题。

#### 标准对比对

| 章节 A   | 章节 B   | 找什么                                                            |
| -------- | -------- | ----------------------------------------------------------------- |
| 功能需求 | API 契约 | 功能说的操作 API 都有端点吗？API 有的端点功能里都提了吗？         |
| 数据模型 | 业务规则 | 业务规则引用的字段都在数据模型里有定义吗？类型/约束一致吗？       |
| 部署说明 | 技术选型 | 部署说的环境选型支持吗？（私有化 vs 云服务、容器 vs 裸机）        |
| 角色权限 | 功能需求 | 每个角色能操作的功能都明确了吗？管理员权限和普通用户有重叠吗？     |
| 版本规划 | 功能需求 | v1 做 vs v1 不做的边界清晰吗？有没有"v1 功能"用到了"v2 才有的组件"？ |
| 性能指标 | 技术选型 | 选型能撑住指标吗？（QPS、延迟、数据量）                           |
| 安全要求 | 数据模型 | 加密字段、脱敏规则、审计日志——哪层负责？                          |

#### 实操步骤

1. **先把两个章节并排打开**（对比阅读，不要来回滚动）
2. **建一个"承诺清单"**：章节 A 承诺了什么操作/能力
3. **逐一在章节 B 里找对应**
4. **找不到 → 矛盾或缺定义；找到但不一致 → 矛盾**

#### 示例

**spec 片段 A（功能需求）：**

> v1 支持用户批量导入（CSV、Excel）、单个创建、按角色搜索。

**spec 片段 B（API 契约）：**

> POST /api/v1/users - 创建单个用户
> GET /api/v1/users - 搜索用户（支持 role 过滤）

**交叉对比结果：**

- 功能说"批量导入"，API 只有单条 POST，标为**矛盾**（必修）
- 功能说"CSV、Excel"，API 没提文件上传格式，标为**缺定义**（应修）
- 角色搜索，API 有 `role` 过滤，标为**一致**

### 1.3 4 类问题的判断标准

#### 矛盾（必修级）

**判断**：两个地方说的不一致，会导致实施时做不下去。

```
特征："v1 做 X" vs "v1 不做 X"
     "用 A 方案" vs "用 B 方案"（两处说的不同）
     "管理员可以删除" vs "删除权限只给超级管理员"（角色边界打架）
```

**为什么矛盾几乎总该是必修？** 矛盾不修，后面做的决策都在自相矛盾的地基上。即使看起来是"小矛盾"（如措辞不同），很可能反映更深的认知不一致。记录矛盾时**必须引用两个位置**（位置 A vs 位置 B）。

#### 缺定义（应修级，但可能是必修）

**判断**：重要概念提了名但没定义，实施时**无法确定怎么做**。

```
特征："P0 必填"但字段类型没说
     "支持多语言"但没写存储方案
     "高性能"但没写 QPS/延迟目标
```

**升为必修的条件**：

- 缺定义阻塞了阶段 2/3 的判断（不知道字段类型，连数据库选型都定不了）
- 缺定义让两个实施方向都"通"但方向相反（"多语言支持"到底按语言拆表还是 JSON 字段？）
- 缺定义有安全/合规影响（"加密传输"但没说 TLS 版本）

#### 悬空引用（应修）

**判断**：引用了不存在的章节、表、API、接口。

在阶段 4 的引用扫描中会全部暴露——阶段 1 可以先标记，阶段 4 再统一处理。

#### 模糊（应修/可选）

**判断**：多个理解方向都合理，需要拍板定一种。

```
特征："管理员"和"超级管理员"权限边界重叠
     说了"多语言支持"但未说明 fallback 策略
     "中小企业"未定义规模边界
```

**处理**：暂可用合理默认值（如 fallback 取 en），标注"后续需拍板"。

### 1.4 自审的深度自检

#### "我审得够深吗？"的 5 个信号

1. 至少找到 **5 个问题**（如果少于 5 个，大概率是审得不够——真实文档很少有完美一致的）
2. 至少有 **1 个矛盾**（纯缺定义和模糊 → 可能只是表面审，没做交叉对比）
3. 每个问题都引用**两个具体位置**（交叉对比的标志）
4. 必修/应修/可选分类合理——至少三分之一应该是必修或应修
5. 自审结束后，你对 spec 的**理解比自审前更深**——你找到了 spec 作者可能都没意识到的矛盾

#### 如果问题清单是空的

两种可能：

- spec 真的完美（极端罕见，只在极简 spec 中出现）
- 审得不够深（默认假设）

处理方式是：**用子代理再审一遍**。子代理没有主上下文的认知偏差（你刚才读了 spec，已经有了"它是对的"的印象），更能抓到矛盾。子代理独立读取 spec，生成它的问题清单，然后和你的问题清单合并。具体子代理使用技术见 §4.3。

#### 自审清单（自己检查是否充分）

- [ ] 功能章和 API 章交叉对比过了？
- [ ] 部署章和技术选型章交叉对比过了？
- [ ] 数据模型和业务规则交叉对比过了？
- [ ] 所有"v1 做 / v1 不做"的决策互相打架吗？
- [ ] 所有 P0/P1/P2 字段都有类型和约束定义吗？
- [ ] 所有"参见 §X"的引用都能指向真实存在的章节吗？
- [ ] 如果问题清单是空的——是 spec 真的完美，还是我审得不够深？考虑让子代理再审一遍。

### 1.5 常见漏审的模式

这些是容易漏的地方——因为它们不在"核心内容"里，而在边角：

- **假设章节**："我们假设用户已经有了 X"——如果后续章节没验证这个假设，就是矛盾
- **migration/迁移说明**：如果说了"从旧系统迁移"但数据转换方案没写 → 缺定义
- **glossary/术语表**：如果两个术语定义打架（如"项目"和"workspace"本质是同一个概念） → 矛盾
- **非功能需求**：安全/性能/可维护性——往往写在单独段落，和功能章节的隐含前提不一致

## 2. 深审选型（阶段 2）

### 2.1 深度审视的核心思维

阶段 2 的独特之处：它不是找 spec 写出了什么矛盾（那是阶段 1），而是问 spec 之外的"**有没有更好的路**"。

spec 是一种**关于"应该怎么做"的惯性**——它承载了某种路径依赖。也许 spec 作者因为历史原因选了某个框架，或"别人都这么写"所以写了某个方案。深度审视就是在不推翻 spec 意图的前提下，**挑战实现路径**。

核心问题永远只有一句：

> **"为什么必须这样做？有更轻、更独立、更适合团队和部署目标的方法吗？"**

### 2.2 决策原则先于选型

#### 为什么先定原则

很多选型纠结是因为"A 优势 X，B 优势 Y"——如果比较维度很多，就会陷入无限对比。**决策原则是"过滤器"——它把比较维度减少到真正重要的 2-3 个。**

#### 如何建立决策原则

从项目的实际情况中提炼：

**来源 1：部署约束**

- 私有化部署 → "必须能在无外网环境运行"
- 客户自有 IDC → "不能依赖云服务"
- 离线更新 → "必须支持自升级机制"

**来源 2：团队能力**

- 团队有 Rust 但没人熟 C++ → "栈必须在团队当前能力范围内"
- 前端 3 人 React → "非 React 方案必须有压倒性优势才考虑"

**来源 3：架构愿景**

- "任何组件可独立替换"
- "前端不绑定后端框架"
- "存储不绑定 ORM"

#### 原则的写法

原则是**测试**——用它去测每个选型：

> "如果换掉这个组件，其他组件受影响吗？"

原则不要超过 5 个。多了等于没原则。

### 2.3 质疑的 4 个切入点

#### 1. 质疑"全家桶"

**信号**：spec 选了某个全家桶（Modern.js、Next.js、Spring Boot 全家桶、Django）
**质疑**：全家桶里哪些是真正需要的？拆开用散件 + 约定，是否更轻？

**示例**：
spec 选 Modern.js → 实际上 Hono 做 API、React Router 做路由 → Modern.js 只是上层封装
→ 去掉 Modern.js，用 Vite + 少量约定 → 更轻、组件更独立

#### 2. 质疑"看起来理所当然的"

**信号**：选型中没有解释"为什么"——仿佛这就是唯一合理答案
**质疑**：有没有反直觉的替代方案？

**示例**：
spec 说"用 MySQL 因为数据量大" → 实际预期数据 ≤ 100W 行
→ SQLite（含 WAL 模式）在百万级数据完全够用 → 部署更简单，零运维

#### 3. 质疑"规模不匹配"

**信号**：用了"大规模"方案处理"小规模"问题，或反之
**质疑**：现在和 2 年后的规模分别是什么？方案什么时候会瓶颈？

**示例**：
spec 选 Elasticsearch 做全文搜索（10K 条文章）→ PG FTS 完全够
→ 去掉 ES，减少运维复杂度

#### 4. 质疑"非功能性瓶颈"

**信号**：方案的运维/升级/调试路径含糊
**质疑**：如果出问题怎么定位？怎么回滚？怎么升级？

**示例**：
spec 说"微服务"但没有服务发现/追踪/日志聚合方案
→ 如果团队只有 5 人，"微服务"可能是过度工程 → modular monolith 更合适

### 2.4 需要质疑的选型类别（自检清单）

对 spec 中每个选型，用以下问题过一遍：

#### 框架/库选型

- [ ] 这个框架的"全家桶"特性是否把不该绑定的组件绑在一起了？
- [ ] 去掉这个框架，用更轻的东西——缺了什么？能否自己补？
- [ ] 团队有人实际用过吗？还是只是 spec 写了？

#### 存储选型

- [ ] 读写模式是什么？（OLTP/OLAP/混合）
- [ ] 10x 数据量增长后还撑得住吗？
- [ ] 私有化部署能跑吗？依赖外部服务（如 S3、Cloud SQL）吗？
- [ ] 备份、迁移、回滚方案考虑了吗？

#### 通信选型

- [ ] REST/GraphQL/gRPC/WebSocket——为什么这个？场景匹配吗？
- [ ] 断网、重连、超时处理有方案吗？
- [ ] 版本化策略有吗？（API v1 → v2 怎么过渡？）

#### 部署选型

- [ ] 只有 Docker 方案？裸机部署考虑了吗？
- [ ] CI/CD 依赖外部服务吗？私有化部署时 CI/CD 怎么跑？
- [ ] 自升级机制有吗？（尤其是无外网环境）

### 2.5 ADR 的写作心法

#### ADR 不是作文——是决策的"证据链"

一份好的 ADR，读者只有 2 个问题：

1. "为什么选了 X？" - 看决策原因
2. "代价是什么？" - 看影响

不需要文学修辞，不需要背景铺垫。简明即可。

#### 推翻选型的 ADR 必须写

**推翻一个 spec 继承的选型是正常操作**——但不记录推翻原因的 ADR，下一轮又会重新考虑被推翻的选型。

ADR 的存在意义之一就是 **"防止决策重做"**。

#### ADR 的编号和生命周期

- 用递增序号：ADR-001, ADR-002, ...
- 一旦写定，不再修改内容
- 如果后来推翻 → 新建 ADR-00X，状态为"已推翻"，覆盖旧 ADR 状态
- 永远可以追溯"为什么那个时候做了那个决策"——这才是 ADR 的价值

### 2.6 团队经验和外部依赖的考量

除了纯技术对比，还应该考量：

#### 团队经验

- "我们团队有人写过吗？"——如果 0 人用过 Rust，选 Rust 写核心服务是巨大风险
- "出了问题谁修？"——选小众框架意味着社区帮助少，所有 bug 都是自己的 bug

#### 外部依赖风险

- "如果这个 SaaS/PaaS 挂了/涨价了/下线了，方案还跑得动吗？"
- "版本兼容性承诺多久？"——依赖库的 breaking change 频率

#### 部署路径

- "私有化部署需要什么？"——是否需要 Docker、K8s、外部镜像仓库？
- "离线升级怎么搞？"——如果是私有化部署，升级包怎么传递？

## 3. 增量约束（阶段 3）

### 3.1 增量设计的本质

阶段 3 做的事情可以用一句话概括：

> **把"实施时会卡住"的隐性决策提前显性化。**

实施团队拿到 spec，会发现大量细节 spec 没说。不是 spec 写得差——而是**spec 不该写那么细**。spec 在意的是"做什么"，阶段 3 在意的是"怎么做才不跑偏"。

阶段 3 不写代码，不写实现细节，只做一件事：**定约束**——也就是"必须做什么、禁止做什么、优先做什么"。

### 3.2 识别卡点的系统方法

#### 方法 1：走一遍"实施第一天"

假设你是开发第一天进项目的工程师：

- 打开 spec，读到"需要一个用户系统"——好，建表。字段类型？string? int? uuid?
- 读到"支持多语言"——好，怎么存？一张表所有语言？N 张表每语言？
- 读到"文件上传"——好，存哪？本地文件系统？S3？MinIO？文件大小限制？
- 读到"搜索"——好，怎么搜？LIKE？全文索引？搜索引擎？

**每个"这里该怎么做？" = 一个卡点 = 需要写约束。**

#### 方法 2：从历史中学习

回顾你的团队上次做项目时：

- 哪些决策拖了很久没定？（→ 持续性的卡点）
- 哪些决策定了又推翻重来？（→ 缺原则的决策）
- 哪些实施后发现"原来是这个意思"？（→ 约束不清晰的后果）

这些几乎一定会在新项目中重现——现在就写约束。

#### 方法 3：分类穷举法

把"所有可能的卡点"分类，逐一检视：

| 领域     | 典型卡点                                                                 |
| -------- | ------------------------------------------------------------------------ |
| **数据** | 字段类型/约束、ID 策略（自增/uuid/snowflake）、软删除/硬删除、时间戳精度 |
| **API**  | 版本策略、错误码体系、分页方式、认证方式、限流方案                       |
| **前端** | 路由策略、状态管理、i18n、design token 管理、构建方案                    |
| **存储** | 文件存储路径、图片处理策略、CDN 方案、备份策略                           |
| **部署** | 容器化/裸机、CI/CD 方案、自升级机制、监控告警                            |
| **安全** | 认证（JWT/Session）、授权（RBAC/ABAC）、CSRF/CORS、加密存储              |
| **运维** | 日志方案、审计策略、错误追踪、性能监控                                   |

不是每个领域都要写——但每类至少扫一眼："这个领域在我们项目里有卡点吗？"

#### 示例：从 spec 片段推导卡点

**spec 说**："用户可以上传文章封面图。"

**推导卡点：**

1. 存哪？（本地 / S3 / MinIO） → 约束："必须用 S3-compatible storage"
2. 大小限制？（1MB? 10MB? 没有限制？） → 约束："必须限制为 5MB"
3. 格式？（jpg? png? webp? 都接受？） → 约束："必须转换为 WebP，保留原图 30 天"
4. 尺寸？（原图直接返回给前端？） → 约束："必须生成 thumbnail(200x200) + medium(800x800)"
5. 命名？（用户原始文件名？hash？） → 约束："必须用 content-hash 命名，防止冲突"

**一条 spec 语句 → 产生了 5 条约束。** 这就是增量设计的价值。

### 3.3 约束的优先级判定

#### P0：不做就阻塞

**判断标准**：如果这个约束不定，实施团队就**无法开始**对应领域的工作。

```
信号：
- 依赖链：P0 约束定了，下游设计才能开始
- 一旦实施就不可逆（如数据库 schema、API 版本策略）
- 影响所有功能模块（如权限模型）
```

#### P1：第一个迭代周期内做

**判断标准**：可以暂时用合理默认值或简化方案，但不能拖过第一个迭代。

```
信号：
- 临时方案不影响功能正确性，但影响质量/可维护性
- 可以推迟 1-2 周，但再晚就会累积技术债
```

#### P2：可推迟——非 MVP

**判断标准**：MVP 不需要它也能完整交付价值。

```
信号：
- 非 MVP 功能依赖这个约束
- 约束只在规模化后才有意义（如全局搜索、审计日志）
- 约束可以通过"先不做"来规避
```

### 3.4 约束的写法规则

#### 3 种约束词

| 词       | 含义                         | 示例                                           |
| -------- | ---------------------------- | ---------------------------------------------- |
| **必须** | 硬约束，不讨论               | **必须**：slug 唯一约束在 DB 层                |
| **禁止** | 不能碰的线，碰了就是 bug     | **禁止**：按语言拆 i18n 表                     |
| **优先** | 倾向方案，有更好替代时可讨论 | **优先**：用 PostgreSQL FTS 而非 Elasticsearch |

#### 约束 vs 功能的区别

- **P0 约束**写入 PRD §4 非功能需求：不可妥协，实施阶段必须遵守
- **P1/P2 约束**写入 PRD 附录：可推迟，版本迭代时可以增删改
- **功能**写入 PRD §3 功能需求：可调整，版本迭代时可以增删改
- 不要把"支持深色模式"写成约束——这是功能，v1 不做可以推迟

#### 识别卡点的方法

如果实施阶段会在某个问题上**停下来问"这个怎么做"**——这就是卡点，需要写约束。

典型信号：

- "这里该用什么方案？" → 缺约束
- "这两个方案选哪个？" → 缺约束 + 缺优先说明
- "这个实现对吗？" → 缺"是否做对"的判断标准（也是缺约束）

### 3.5 约束 ≠ 实现

#### 核心分界

| 约束写什么                         | 实施阶段写什么                                                   |
| ---------------------------------- | ---------------------------------------------------------------- |
| "i18n 用 JSON 字段存，fallback en" | "字段名叫 `title_i18n`，结构是 `{ zh: string, en: string }`"     |
| "Slug 唯一约束在 DB 层"            | "migration 里加 `unique_index :articles, :slug`"                 |
| "错误码用 DOMAIN_CODE 格式"        | "错误码枚举：AUTH_001, AUTH_002, ..."                            |
| "用乐观锁处理并发编辑"             | "articles 表加 `lock_version` 字段，update 时 WHERE version = ?" |

#### 为什么不能写太细

- 写太细 → 实施团队没有判断空间 → PRD 变成"能不能照抄"而不是"能不能理解"
- 写太细 → 边界条件多了就必须全关联 → 改一处就崩全盘
- 写太细 → 读者分不清"哪段是可以商量的" → 约束和功能失去分层价值

### 3.6 常见约束的类型导引

#### 数据层约束

- ID 策略、软删除/硬删除、时间戳格式（UTC/Epoch/ISO 8601）
- JSON 字段的使用范围（"只用于配置数据，不用于关系数据"）
- Migration 方向（只增不删？允许回滚？）

#### API 层约束

- 版本策略、分页方式、错误格式、认证 token 方式
- 限流策略（Per-IP? Per-User? Token-bucket?）
- API 文档生成方式（代码注释驱动 vs 手动维护 → 约束"必须用代码注释生成规范"）

#### 前端层约束

- 状态管理位置（Client State vs Server State 的边界）
- i18n 键名格式、Design Token 源
- 测试策略（E2E vs 集成 vs 单元 的分工）

#### 部署层约束

- 容器化方案、环境变量管理、密钥管理
- CI/CD 触发条件、制品存储、回滚策略

### 3.7 自检清单

- [ ] 至少覆盖了 8 个实施卡点？
- [ ] 每个约束都附了理由？
- [ ] P0/P1/P2 排序合理？（P0 = 不做就阻塞）
- [ ] 约束只锁机制，不锁实现细节？
- [ ] 约束和功能已分离（两个独立文件/章节）？

## 4. 精简提交（阶段 4）

### 4.1 为什么精简比丰富更重要

一个常见误解：**"文档越长越详细越好。"**

现实是：文档越长，读者越不会读。信息密度（重要信息 / 总字数）才是衡量文档质量的核心指标。

阶段 4 的目标不是删到"短"——是删到**"每段话都有不可替代的作用"**。

如果一段话删掉后文档还能正常运作，它就**应该被删掉或移到 reference**——不是因为它没用，而是因为它的位置不对。

### 4.2 删除 4 类内容的实操技术

#### 4.2.1 删具体代码

**判断标准**：这段内容在执行时**会发生变化**（字段名会改、参数会增、配置会调）。

**移到哪**：

- 完整 Schema → `docs/architecture/api-reference.md`
- 配置文件 → `docs/architecture/config-reference.md`
- 部署脚本 → `docs/architecture/deployment-guide.md`

**PRD 里留什么**：

- 不是"API Schema 的完整内容"，而是"API Schema 的位置"
- 不是"配置文件的每项说明"，而是"配置文件的路径和设计理念"

**示例**：

```markdown
# 反例：PRD 里不该写

## 用户 API

POST /api/v1/users
Body: { "name": "string (required)", "email": "string (required)", ... }
Response: { "id": "uuid", "name": "string", ... }

# 正例：PRD 里只写

完整 API Schema → `docs/architecture/api-reference.md`
```

#### 4.2.2 删决策过程

**判断标准**：段落在描述"我们讨论了什么"而非"我们决定了什么"。

**移到哪**：`docs/architecture/decisions.md`（ADR 集）

**PRD 里留什么**：

- 不是"为什么选 A 不选 B"，而是"我们用 A"
- 读者不需要重新做一遍你的决策——他们需要执行你的决策

#### 4.2.3 删重复

**判断标准**：同一条信息在 ≥2 个地方出现，且内容因复制粘贴而可能不同步。

**处理规则**：

- 保留最权威的那处（通常是"定义处"而非"引用处"）
- 其余用引用代替（短引用："参见 §X"；跨文档引用："详见 `docs/architecture/decisions.md#ADR-003`"）

**不要做的事**：

- 不要"为了读者方便"保留重复——方便的代价是"改一处忘改另一处"
- 不要"摘要 + 正文"都保留——要么摘要+引用，要么只有正文

#### 4.2.4 删过时引用

**操作**：用 grep 扫描所有引用标记（`见 §` / `参见 §` / `refer to` / `详见`），逐一检查：

- 目标章节/表/文件存在吗？
- 引用到的内容是当前内容吗？还是旧版本？

**在阶段 4 需要跨文档校验**：

- PRD 引用了 `decisions.md` 的 ADR-003？检查 ADR-003 是否存在、状态是否已更新
- PRD 引用了 `phase/2026-06-23-xxx.md`？——这是占位，允许存在
- 任一文档改了 → grep 全部 → 确认所有引用都同步

### 4.3 子代理审查机制

#### 4.3.1 为什么需要子代理

主上下文有**认知偏差**：你花了几个小时写这份文档，大脑已经形成了"它是对的"的预期。主上下文再审一遍，很大概率会**漏掉**矛盾——不是不够认真，而是认知框架已经定了。

子代理没有这个偏差——它第一次看到文档，是**干净视角**。

#### 4.3.2 子代理的 5 项审查任务

把以下任务交给子代理，要求它**逐条执行并报告**：

1. **内部一致性检查**
   - 通读 PRD 和 `decisions.md`
   - 交叉对比，找矛盾（类似阶段 1 的逻辑）
   - 报告：不一致的具体位置 + 建议修复

2. **过时引用扫描**
   - grep 所有章节引用、文件引用、ADR 引用
   - 逐一验证目标存在且正确
   - 报告：悬空引用列表

3. **产物完整性检查**
   - 阶段 1 的问题清单——所有"必修"问题已在最终 PRD 中解决了吗？
   - 阶段 2 的 ADR——所有 ADR 的"影响"已体现在 PRD 中了吗？
   - 阶段 3 的约束——所有 P0 约束都在 PRD §4 非功能需求中体现了吗？
   - 报告：遗漏项列表

4. **信号噪声比检查**
   - 在 PRD 中随机挑 3-5 段，问"删掉这段，文档还能运作吗？"
   - 如果能 → 标记为可精简或应移到 reference
   - 报告：建议删除/移动的段落

5. **规范一致性检查**
   - 对照 `conventions.md §3.1` 的 5 必填章节，逐条打勾
   - 命名是否符合 §2.1（`YYYY-MM-DD-<name>.md`）
   - 顶部是否含"对应阶段"反向链接（占位 TBD 可接受）
   - 报告：规范合规性评分（0-5）

#### 4.3.3 子代理的作业方式

用 Task 工具启动子代理（`agent_type="general-purpose"`），prompt 要点：

- 明确说"只审不改"——避免它自作主张去改文档
- 给出文档路径（PRD 路径、ADR 集路径）和 5 项检查清单
- 要求逐项报告结论和建议，不执行修改
- 主上下文收到反馈后修复问题，再复用同一个 prompt 让子代理复审（确保修正没引入新问题）

#### 4.3.4 处理子代理的反馈

- 子代理报告"没问题" → 大概率也是真的没问题了（两个独立视角都认为 OK）
- 子代理找到问题 → 修正后**让子代理再审一遍**（确保修正没有引入新问题）
- 子代理和你的判断不一致 → 讨论后定——子代理也可能错，但它的"错"是新鲜的视角

### 4.4 提交策略（lore commit）

#### 4.4.1 为什么单次提交

"PRD 全面定型"是一个**逻辑上不可分割的事件**——所有变更服务于同一个目标：把 PRD 从草稿提升到可执行基线。

拆成多个提交的问题：

- "加字段 A"→"加字段 B"→"修正矛盾"→ 每个提交都丢失了"为什么改"的上下文
- 后来的人看 git log 看到 5 个提交，不知道哪个是"最终的 PRD 基线"
- 如果有 Cherry-pick 需求，拆开才是对的——但 PRD 定型不需要 cherry-pick

#### 4.4.2 lore commit 格式

完整 schema 与字段说明见 `conventions.md §8`。PRD 定型提交的要点：

- `intent`（必填，≤72 字符）：一句话说明本次变更目的，如"提纯 {spec-name} 为 PRD v1.0"
- `body`：概述 4 阶段工作流成果（发现 N 个矛盾、确认 M 个选型、补全 K 个约束、精简后落盘）
- 关键 trailers：
  - `Constraint`：PRD 引入的硬规则（如"PRD 必含 conventions §3.1 的 5 章节 + §0 目标声明/验收开关"）
  - `Rejected`：被否决的方案，格式 `方案 | 原因`
  - `Directive`：团队约定（如"PRD 不写 Phase，Phase 留 TBD 由 phase-workflow 补全"）
  - `Tested`：已验证内容（如"子代理 5 项审查通过"、"grep 确认零悬空引用"）
  - `Not-tested`：未验证内容（如"PRD ↔ Phase 双向引用 Phase 留 TBD"）

提交时用 `lore commit`（禁止裸 `git commit`，由 `lore-commit-guard` 拦截）。

### 4.5 阶段 4 的最终检查

在说"完成"之前，确认：

- [ ] PRD 内容精炼，没有无谓重复
- [ ] 所有"为什么选 A"已移入 `docs/architecture/decisions.md`
- [ ] 所有具体代码/配置已移入 `docs/architecture/`
- [ ] `grep` 确认零悬空引用
- [ ] 子代理 5 项审查完成，反馈已处理
- [ ] **PRD 顶部有"目标声明"和"目标验收开关"**（归档触发器，详见 `archival.md`）
- [ ] **PRD 顶部有"对应阶段: [TBD]"占位行**（规范合规）
- [ ] **命名符合 conventions §2.1**：`YYYY-MM-DD-<name>.md`
- [ ] **工作产物已清理**：`docs/prd/.working/<name>/` 已删除
- [ ] **lore commit 已提交**（不是 `git commit`）
- [ ] 你愿意让一个新来的团队成员**只读 PRD + decisions.md 就开始实施**——如果可以，基线就完成了
