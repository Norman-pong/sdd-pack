# PRD -> Phase 4 阶段工作流（PRD 边界 -> 任务分解 -> 里程碑 -> 精简提交）

---

## 1. 自审 PRD 边界（阶段 1）

阶段 2 的"任务分解"完全建立在 PRD 边界之上。如果 PRD 内部矛盾没找出来：

- 任务拆解时把矛盾两边都拆了——下游工程师不知道听哪边
- 任务依赖识别错——并行任务被串行化，工期翻倍
- 验收标准缺失——任务"完成"无客观判定，反复返工

核心洞察：**不是找新东西，是找 PRD 已写出来但互相打架的东西**。一段说 v1 做 A，另一段说 v1 不做 A——这就是矛盾，会直接影响任务拆解。

### 1.1 PRD 边界审查的 4 个维度

#### 范围维度（影响任务清单）

**信号**：
- "v1 做 X" vs "v1 不做 X" 同时存在
- 功能清单 P0/P1/P2 分级不明确
- "可选功能"和"必须功能"混在一起

**影响**：任务拆解时不知道是否要给某些功能排期。

#### 依赖维度（影响任务依赖链）

**信号**：
- PRD 提到外部系统但未说集成方式
- 跨章节引用的接口/字段定义不一致
- "假设已有 X" 但没说 X 由谁提供

**影响**：任务拆解时无法规划"集成外部系统"任务。

#### 验收维度（影响任务完成判定）

**信号**：
- "性能良好"、"界面友好"等不可证伪的形容词
- NFR 阈值缺失（无 P99 数字、无 SLO 数字）
- 验收标准散落在多章节，口径不一致

**影响**：任务验收时无客观判定。

#### 引用维度（影响 Phase 文档引用完整性）

**信号**：
- PRD 引用了不存在的章节
- Phase 任务会引用 PRD 内容，但 PRD 内部就有矛盾引用

**影响**：任务"实现异常处理"无法定位需求来源。

### 1.2 3 类问题的 phase 应用（通用定义见 prd-workflow.md 阶段 1）

3 类问题（矛盾 / 缺定义 / 悬空引用）的通用判断标准见 `prd-workflow.md §1.3`。phase 阶段的差异不在"什么是矛盾"，而在**矛盾如何影响任务拆解**--审 PRD 而非 spec，落点是"任务边界"和"依赖链"：

| 类别 | phase 差异（影响任务拆解） | 升为必修的条件 |
|---|---|---|
| **矛盾** | 导致**任务拆解方向相反**（两边都拆 -> 下游不知听谁） | 几乎总该是必修（任务拆错，下游返工成本是阶段 1 的 5-10 倍） |
| **缺定义** | **任务无法开始**（任务边界无依据） | 阻塞任务粒度切分 / 让两个任务方向相反 / 有安全合规影响（如"加密传输"但没说 TLS 版本） |
| **悬空引用** | Phase 写完后任务会引用不存在的章节/接口 | 应修 |

### 1.3 自审清单（phase 专属，4 项 PRD 边界交叉验证）

> 通用自审项（"v1 做/不做"矛盾互查、"参见 §X"引用核验、空清单处理）见 `prd-workflow.md §1.4`。phase 阶段的差异是 4 项 PRD 边界 -> 任务拆解的交叉验证：

- [ ] PRD §3 全部 P0/P1 功能都交叉对比过了？（每个功能 -> ≥1 任务）
- [ ] PRD §4 NFR 与 §3 任务的依赖关系审查了？（NFR 阈值 -> 任务验收标准）
- [ ] PRD §7 集成需求与 §3 任务的外部依赖审查了？（外部系统 -> `[外部:X 已就绪]` 任务依赖）
- [ ] 所有任务验收标准都有可测阈值？（无"友好/流畅"，须有客观判定）

### 1.4 自审的深度自检

> "5 个深度信号"（≥5 问题、≥1 矛盾、双位置引用、分类比例）和"空清单 -> 子代理复审"的通用处理见 `prd-workflow.md §1.4`。phase 阶段的差异信号：

- 自审结束后，你**对任务拆解的边界比自审前更清晰**--能找到 PRD 作者没意识到的边界问题（不只是"理解更深"，而是"任务边界更明确"）

如果达不到这个信号，说明自审还停在"找缺定义"层面，没深入到"矛盾如何影响任务拆解"--回到 §1.1 的 4 个维度重新审。

### 1.5 常见漏审的模式

这些是容易漏的地方——因为它们不在"核心内容"里，而在边角：

- **PRD §1.3 成功指标**：往往用模糊词汇（"提升效率"），与 §3 任务无直接对应
- **PRD §4 NFR 阈值**：经常散落，与 §3 任务验收不对应
- **PRD §7 集成需求**：常被忽略，但 Phase 任务依赖外部系统
- **PRD §10 风险与约束**：已被识别为风险但未说"应对任务由谁做"

### 1.6 与 prd 阶段 1（spec 自审）的差异

| 维度 | prd 阶段 1（spec 自审） | phase 阶段 1（PRD 边界审查） |
|---|---|---|
| 输入 | spec（用户故事/客户需求） | PRD（已写好） |
| 目标 | 找 spec 内部矛盾 | 找 PRD 影响任务拆解的矛盾 |
| 产物 | problem-list.md -> ADR / 约束 | problem-list.md -> 任务分解 |
| 重点 | 范围/数据/部署 | 范围/依赖/验收/引用 |

**关键差异**：prd 是从 spec 提纯，矛盾越早找越好；phase 是从 PRD 拆任务，矛盾影响的是"任务边界"和"依赖链"。

## 2. 深审任务分解（阶段 2）

### 2.1 任务分解的本质

> **把 PRD 边界的"功能需求"拆解为"可执行任务"，每条任务 = 1 个工程师 1-3 天可完成 + 客观可验收。**

任务粒度不是越细越好，也不是越粗越好。**合适的粒度 = 1-3 天可完成 + 验收标准可判定**。

### 2.2 任务粒度的 3 个级别

#### 太粗（> 3 天）

**示例**：`T001: 实现用户系统`

**问题**：
- 无法追踪进度（做了 50% 算什么状态？）
- 无法识别依赖（数据库 schema？API 设计？前端？测试？没分）
- 无法判定"完成"（用户系统完成 = 什么？）

#### 太细（< 0.5 天）

**示例**：
```
T001: 写 users 表的 migration
T002: 写 users 表的 schema
T003: 写 users 表的索引
```

**问题**：
- 文档臃肿（10 个任务合并为 1 个更易管理）
- 工程师疲于"切任务"而非"做任务"
- 依赖链过度细化，无意义

#### 合适（1-3 天）

**示例**：
```
T001: 实现用户注册 API + 单元测试
- 包含：POST /api/v1/users 端点、参数校验、密码哈希、邮箱唯一性检查
- 验收：201/400/409 + bcrypt + 单元测试覆盖 ≥ 80%
- 工时：2 人天
```

**优势**：
- 进度可追踪（0%/50%/100%）
- 依赖可识别（数据库 schema 已完成 / [外部:邮件服务已就绪]）
- 验收可判定（201/400/409 是客观 HTTP 状态码）

### 2.3 任务分解的方法

#### 方法 1：从 PRD §3 功能需求映射

1. 列出 PRD §3 全部 P0/P1 功能点
2. 每个 P0 功能 -> ≥1 个任务
3. 每个 P1 功能 -> 1 个或多个任务（视复杂度）
4. P2 功能可选拆，若拆则单独标记

#### 方法 2：从技术层次切分

有些功能需要横切多个层次（数据库、API、前端、测试）。**任务按层次切分**比按功能切分更清晰。

```
PRD §3.2.1 用户注册:
-> 任务（按层次）:
T001: 设计 users 表 schema + migration（0.5 人天）
T002: 实现邮箱唯一性索引（0.5 人天，依赖 T001）
T003: 实现 POST /api/v1/users 端点（1.5 人天，依赖 T002）
T004: 实现邮件验证邮件发送（1 人天，依赖 T003）
T005: 实现前端注册表单（1 人天，依赖 T003）
T006: 集成测试 + E2E 测试（1 人天，依赖 T004, T005）
```

#### 方法 3：从依赖链倒推

1. 列出"必须最后做"的任务（部署、集成测试、上线）
2. 倒推"必须先做"的任务
3. 中间填充"可并行"的任务

### 2.4 任务属性设计

每条任务必须有 6 个属性（对齐 `conventions.md §4.1` Phase 必填章节）：

| 属性 | 取值 | 示例 |
|---|---|---|
| **任务 ID** | `T001`、`T002`... | `T001` |
| **任务名称** | 动词开头 | "实现用户注册 API" |
| **负责人** | 占位（待定） | `@待分配` |
| **预估工时** | 人天 | `2 人天` |
| **依赖** | 任务 ID / 外部依赖 | `T001` / `[外部:邮件服务已就绪]` |
| **状态** | `conventions.md §4.4` 三态 | `未开始` |

### 2.5 任务依赖识别

#### 依赖类型

| 类型 | 示例 | 标识 |
|---|---|---|
| **强依赖** | DB schema 必须先于 API | `T001 -> T003` |
| **弱依赖** | 文档可后于实现 | `T003 -> T010` |
| **外部依赖** | 邮件服务、第三方 API | `[外部:邮件服务已就绪]` |
| **并行任务** | 前端/后端各自独立 | `T005 ‖ T003` |

#### 依赖识别清单

- **数据库 schema** 必须先于 API
- **API 实现** 必须先于前端调用
- **单元测试** 可与实现并行，但**集成测试**必须后于实现
- **部署脚本** 必须后于所有功能完成
- **文档更新** 可后于实现（但归档前必须完成）

#### 依赖闭环检查

每条任务的依赖必须：

- 存在（任务 ID 在清单中）
- 合理（无循环依赖 A->B->A）
- 显性化（写在属性中，不靠"心照不宣"）

### 2.6 任务粒度反模式

#### 反模式 1：任务粒度不统一

```
T001: 设计数据库 schema（0.5 人天）
T002: 实现用户系统（5 人天）
T003: 写一个 if 分支（0.1 人天）
```

**问题**：无法估算整体工期；无法识别关键路径。
**修复**：统一粒度在 1-3 天，T002 拆为 5 个子任务。

#### 反模式 2：验收标准不可测

```
T001: 优化用户系统性能
验收：系统性能良好
```

**问题**："良好"无客观判定；工程师和 reviewer 认知不同。
**修复**：
```
T001: 优化用户系统性能
验收：
- [ ] POST /api/v1/users P99 延迟 ≤ 200ms（对齐 PRD §4.1）
- [ ] 100 并发用户下，无 5xx 错误
- [ ] 数据库查询 P99 ≤ 50ms
```

#### 反模式 3：任务依赖未识别

```
T001: 实现用户注册 API
T002: 编写注册接口的单元测试
```

**问题**：T002 依赖 T001，但未显性化——下游工程师不知道要先等 T001 完成。
**修复**：
```
T002: 编写注册接口的单元测试
依赖：T001
```

#### 反模式 4：任务覆盖度不够

```
PRD §3 功能需求：8 个 P0 功能
任务清单：5 个任务
```

**问题**：8 个功能只拆了 5 个任务——必有遗漏。
**修复**：阶段 2 完成前，**必须交叉验证** PRD §3 全部 P0/P1 都有对应任务。

### 2.7 任务粒度自检清单

- [ ] 每条任务预估 1-3 人天？
- [ ] 任务依赖已显性化？
- [ ] 验收标准全部可测（无"友好/流畅"）？
- [ ] 任务覆盖 PRD §3 全部 P0/P1？
- [ ] 关键路径已识别（M0 -> M3 必走任务）？

### 2.8 任务粒度反模式检查

- [ ] 无粒度不统一（混用 0.1 天和 5 天）？
- [ ] 无验收标准空泛？
- [ ] 无依赖未识别？
- [ ] 无重复任务？

### 2.9 与 prd 阶段 2（深审选型）的差异

| 维度 | prd 阶段 2 | phase 阶段 2 |
|---|---|---|
| 输入 | spec + 矛盾清单 | PRD + 边界问题清单 |
| 目标 | 质疑继承选型，产出 ADR | 拆任务，产出任务清单 |
| 产物 | adr-set.md | task-breakdown.md |
| 关注 | 选型是否最优 | 任务粒度/依赖/验收 |
| 涉及 | 架构/技术决策 | 工程实施 |

**关键差异**：prd 阶段 2 选"做什么技术"，phase 阶段 2 选"怎么拆任务"。两者正交，不重复。

## 3. 增量里程碑（阶段 3）

### 3.1 增量设计的本质

> **把"任务清单"组织为"可推进的阶段节奏"——里程碑给出时间锚点，风险暴露不确定性，协作明确责任。**

任务清单回答"做什么"；阶段 3 回答"什么节奏做、谁配合、有什么风险"。

### 3.2 里程碑设计

#### 里程碑的本质

**里程碑 = 一组任务完成的检查点 + 可交付物**。
不是任务的简单聚合，而是**节奏锚点**——团队在某个时间点可以"暂停"并评估进度。

#### 里程碑数量

**推荐 3-5 个**：

| 数量 | 适用 |
|---|---|
| < 3 | 阶段任务极少（警告：可能粒度太粗） |
| 3-5 | **标准**（推荐） |
| > 5 | 节奏碎片化（警告：可能粒度太细） |

#### 里程碑设计的方法

**方法 1：从依赖链关键路径推**——关键路径上的关键任务完成点 = 里程碑。

```
任务依赖图:
T001(T0+0.5) -> T003(T0+2) -> T006(T0+4) -> T010(T0+6) -> T020(上线 T0+7)

-> 关键路径 = 7 天
-> 里程碑:
M0(T0+0.5): T001 完成
M1(T0+4): T006 完成（后端 API ready）
M2(T0+6): T010 完成（集成测试 ready）
M3(T0+7): T020 完成（上线）
```

**方法 2：从交付物角度推**——每个可交付物（DB schema ready / 后端 API ready / 前端 ready / 集成测试 ready / 上线）= 1 个里程碑。

### 3.3 风险识别

#### 风险的 3 个类型

| 类型 | 示例 | 应对 |
|---|---|---|
| **技术风险** | 关键人员未掌握 Rust / 第三方 API 不稳定 | 培训 / 备选方案 / 缓冲时间 |
| **资源风险** | 关键人员离职 / 工期冲突 | 备份人员 / 任务并行 / 优先级重排 |
| **业务风险** | PRD 业务规则理解偏差 | 早期 demo / 评审会议 / 文档澄清 |

#### 风险属性

每条风险必须有 4 个属性（对齐 `conventions.md §4` 风险表）：风险、影响、概率、应对措施、责任人。

#### 风险识别的反模式

**反模式 1：空泛风险**
```
| 可能延期 | 高 | 中 | 加快进度 |
```
**问题**："可能延期"不可操作；"加快进度"无具体动作。
**修复**：写具体风险 + 具体应对 + 责任人。

**反模式 2：风险无责任人**
**问题**：无责任人，风险无人跟进。
**修复**：加 `@Tech Lead` 或具体人名。

### 3.4 协作需求

协作需求表：协作方（@人）、协作内容、时间节点、状态。

#### 协作需求的反模式

**反模式：协作未点名**
```
| 后端 | 提供 API |
```
**问题**："后端"是团队，不是个人；无时间节点。
**修复**：
```
| @李四（后端） | 提供 POST /api/v1/users API | M1（2026-07-05）前 | 待开始 |
```

### 3.5 阶段验收设计

阶段验收回答"**整个阶段**是否完成"，而非"单任务是否完成"。

#### 阶段级完成标准

- [ ] 全部 P0 任务已完成且验收标准通过
- [ ] 全部 P1 任务已完成或明确推迟理由
- [ ] 集成测试（E2E）通过率 100%
- [ ] 性能指标达到 PRD §4 NFR 阈值
- [ ] 安全审计通过（对齐 §4.2）
- [ ] 文档（API 文档、用户手册）已发布
- [ ] 部署到生产环境并灰度通过

#### 与 PRD 验收的对应关系

- 阶段验收 -> PRD §5 验收标准的子集
- 阶段未涵盖的验收项 -> 需新阶段或下个 Phase

### 3.6 增量设计反模式自检清单

#### 里程碑反模式

- [ ] 里程碑数量 3-5 个（无 < 3 或 > 5）？
- [ ] 里程碑均匀分布（无聚集在一周内）？
- [ ] 每个里程碑有日期+交付物+状态？

#### 风险反模式

- [ ] 风险数量 ≥ 3？
- [ ] 风险有空泛（如"可能延期"）？——应改为具体风险
- [ ] 风险有责任人（@人）？
- [ ] 风险有应对措施（可操作）？

#### 协作反模式

- [ ] 协作有具体负责人（@人），不是"后端/前端"？
- [ ] 协作有时间节点（可追踪）？

### 3.7 与 prd 阶段 3（增量约束）的差异

| 维度 | prd 阶段 3 | phase 阶段 3 |
|---|---|---|
| 输入 | spec + ADR | PRD + 任务清单 |
| 目标 | 补全约束 | 组织里程碑/风险/协作 |
| 产物 | constraint-set.md | milestone-set.md |
| 关注 | 实施卡点 | 节奏锚点 |
| 涉及 | 机制/约束 | 协作/时间 |

**关键差异**：prd 阶段 3 选"用什么机制"；phase 阶段 3 选"什么节奏做"。

## 4. 精简提交（阶段 4）

### 4.1 为什么精简比丰富更重要

一个常见误解：**"Phase 文档越详细越好"**。

现实是：文档越长，读者越不会读。**信息密度（重要信息 / 总字数）**才是衡量 Phase 文档质量的核心指标。

阶段 4 的目标不是删到"短"——是删到**"每段话都有不可替代的作用"**。

如果一段话删掉后 Phase 还能正常运作，它就**应该被删掉或移到 reference**——不是因为它没用，而是因为它的位置不对。

### 4.2 合并工作产物到 Phase 文档

阶段 1-3 的工作产物是**分文件**的（`.working/` 临时区，见 `conventions.md §1.4`）：problem-list.md、task-breakdown.md、milestone-set.md。

阶段 4 需要**合并**到最终 Phase 文档（`conventions.md §4.1` 的 5 必填章节）：

| 工作产物 | 合并到 Phase 章节 |
|---|---|
| `problem-list.md`（已通过的不修的） | 附录（留痕） |
| `problem-list.md`（必修已修的） | §4 风险与问题 |
| `task-breakdown.md` | §2 任务分解 |
| `milestone-set.md` | §3 里程碑 + §4 风险 + §6 依赖与协作 |
| 阶段验收设计 | §5 验收 |

**操作**：
- 任务详情直接合并到 §2.2
- 里程碑表合并到 §3
- 风险表合并到 §4.1
- 协作需求表合并到 §6.2
- 阶段验收清单合并到 §5.1
- 验收记录表留空（待实施时填）

### 4.3 删除 4 类内容（通用框架见 prd-workflow.md 阶段 4）

删除 4 类内容（与 PRD 重复 / 决策过程 / 重复 / 过时引用）的通用判断标准见 `prd-workflow.md` 阶段 4。phase 阶段的差异：

#### 删与 PRD 重复的内容（phase 重点，PRD 是 phase 输入）

PRD 是 phase 的输入，所以"与 PRD 重复"是 phase 阶段 4 的重点。

**判断标准**：Phase 文档的段落与 PRD §3/§4 重复。

**移到哪**：不移动——只留"引用"。

**Phase 里留什么**：
- 不是 PRD §3.2.1 的全文，而是 `T001 实现用户注册 API（详见 PRD §3.2.1）`
- 不是 PRD §4.1 的 NFR 阈值，而是"§4.1 性能要求：P99 ≤ 200ms"

**反例**：把 PRD §3.2.1 全文复制到 Phase T001 任务描述。
**正例**：Phase T001 只写"实现用户注册 API。接口定义详见 PRD §3.2.1"。

#### 删决策过程（phase 角度）

**判断标准**：段落在描述"我们讨论了什么"而非"我们决定做什么任务"。

**移到哪**：`docs/architecture/decisions.md`（由 prd 阶段 2 维护）。

**Phase 里留什么**：
- 不是"为什么用 React 而非 Vue"，而是"前端任务基于 React（详见 ADR-001）"
- 不是"任务粒度为什么是 1-3 天"，而是任务列表（粒度由阶段 2 决定）

#### 删重复 + 删过时引用

通用规则见 `prd-workflow.md` 阶段 4。phase 阶段 4 需要跨文档校验：
- Phase 任务引用 PRD §X？检查 §X 是否存在
- Phase 引用 ADR-001？检查 ADR-001 是否存在（可能在 `decisions.md`）
- Phase 依赖 `[外部:Y]`？——这是占位，允许暂未实现

### 4.4 子代理审查（5 项 phase 检查，通用机制见 prd-workflow.md 阶段 4）

子代理审查的通用机制（启动方式、prompt 模板、"只审不改"约束、主上下文修复后复审）见 `prd-workflow.md` 阶段 4。phase 阶段的差异是 **5 项 phase-specific 检查**：

1. **PRD ↔ Phase 一致性检查**
   - 对照 PRD §3 全部 P0/P1 功能，逐条检查 Phase §2 任务是否覆盖
   - 对照 PRD §4 NFR，逐条检查 Phase 任务验收是否对齐
   - 报告：遗漏项列表

2. **任务依赖闭环检查**
   - 扫描 Phase §2 全部任务的"依赖"属性
   - 每个依赖要么是 Phase 内的任务 ID，要么是外部依赖（明确写出）
   - 检查无循环依赖（A->B->A）
   - 报告：悬空依赖/循环依赖列表

3. **验收标准可测性检查**
   - 扫描 Phase §2 全部任务的"验收标准"
   - 每条验收标准必须有客观判定（P99 ≤ 200ms、HTTP 状态码、测试覆盖率）
   - 报告：不可测验收标准列表

4. **里程碑合理性检查**
   - 检查里程碑 3-5 个，均匀分布，无聚集
   - 每个里程碑有日期+交付物+状态
   - 报告：不合理里程碑

5. **sdd 一致性检查（phase 独家）**
   - 对照 `conventions.md §4.1` 的 5 必填章节，逐条打勾
   - 命名是否符合 `conventions.md §2.2`（`<prd-id>/<seq>-<phase-name>.md`）
   - 顶部 `> 对应 PRD:` 反向链接是否指向真实 PRD 文件
   - 顶部 `> 状态:` 字段是否存在
   - 报告：合规性评分（0-5）

### 4.5 PRD 占位回填（phase 独家）

> 这是 phase 技能的独家能力——prd 阶段 4 会在 PRD 顶部写入 `[TBD - 由其他技能补全]` 占位，phase 阶段 4 完成后回填真实路径。

#### TBD 占位的识别

由 prd 阶段 4 在 PRD 顶部写入：

```markdown
> 对应阶段: [TBD - 由其他技能补全](../phase/<prd-id>/<seq>-<phase-name>.md)
```

**关键特征**：
- 包含 `[TBD` 字符串
- 路径指向 Phase 文档（即使文件不存在）

#### 回填条件

只在以下情况**同时满足**时回填：
- Phase 文档完成阶段 4（子代理审查通过）
- Phase 顶部 `> 对应 PRD:` 反向链接已建立
- 对应 PRD 顶部 `> 对应阶段:` 仍为 `TBD` 占位

#### 回填操作

**手动回填（推荐，避免 sed 误改）**：

1. 打开对应 PRD 文件
2. 找到顶部 `> 对应阶段:` 行
3. 替换为真实路径：
   - Before: `> 对应阶段: [TBD - 由其他技能补全](../phase/<prd-id>/<seq>-<phase-name>.md)`
   - After: `> 对应阶段: [<phase 名称>](../phase/<prd-id>/<seq>-<phase-name>.md)`
4. （可选）在 Phase 名称前加描述，便于阅读

**sed 批量回填（谨慎使用）**：仅当占位格式完全一致时使用。
- 风险：sed 不区分文件，会替换所有匹配行——若多个 PRD 都有相似占位，会误改
- 单个 PRD 回填 -> 手动
- 多个 PRD 批量回填 -> sed + 严格匹配

#### 回填后验证（三项必查）

1. **TBD 已清除**：`grep -rn "对应阶段: \[TBD" docs/prd/` 期望无输出
2. **Phase 文件存在**：`ls docs/phase/<prd-id>/<seq>-<phase-name>.md` 期望文件存在
3. **双向引用**：
   - Phase 文件含 `> 对应 PRD: [...](../prd/...)`
   - PRD 文件含 `> 对应阶段: [...](../phase/...)`

#### 回填失败的修复

| 失败情况 | 原因 | 修复 |
|---|---|---|
| TBD 仍在 | 回填未生效 | 重新执行回填 |
| Phase 文件不存在 | Phase 文档未完成 | 重新走阶段 4 |
| 双向引用缺失 | 一方未更新 | 检查 Phase §1 顶部和 PRD 顶部 |
| 链接路径错 | 命名不一致 | 重新检查 `conventions.md §2.2` 命名规范 |

#### 回填的反模式

- **禁止** Phase 文档未完成阶段 4 时回填 PRD 占位（PRD 顶部会指向不完整的 Phase）
- **禁止** 直接修改 PRD 顶部而不创建 Phase 文档
- **禁止** 回填后不复查（必须 grep 确认 TBD 已清除）
- **禁止** 回填时顺手改 PRD 其他章节（只允许改 `> 对应阶段:` 这一行，其他内容由 prd 技能维护，破坏 prd 4 阶段产物的完整性）

#### 多 PRD 对应多 Phase

**一对多场景**（一个 Phase 文档对应多个 PRD，罕见）：
- PRD-A 顶部 TBD 替换为 Phase 路径
- PRD-B 顶部 TBD 替换为同一 Phase 路径

**多对一场景**（一个 PRD 拆为多个 Phase，更常见）：
- 每个 Phase 文档顶部 `> 对应 PRD:` 指向同一 PRD
- PRD 顶部"对应阶段"列表合并所有 Phase 路径
- 注意：需先建多个 Phase 占位，然后回填 PRD 顶部的多 Phase 列表

#### 与 prd 归档的配合

PRD 归档前（由 prd 技能触发），必须确认：
- 检查对应 Phase 已建立（`grep "对应阶段" docs/prd/...` 期望不是 `[TBD]`）

若 Phase 仍为 TBD，询问用户：
- "PRD 顶部对应阶段仍为 TBD。是否要现在补全 Phase，再归档？"
- "若不补全，归档时 TBD 占位保留（可能违反 `conventions.md` 完整性）。"

PRD 归档后，Phase 文档的 `> 对应 PRD:` 反向链接**不应修改**——指向已归档 PRD 是合法的（归档 PRD 是历史记录）。

但若归档后**重写** PRD（创建新 PRD 替代），需要：
- 新 PRD 顶部"对应阶段"指向原 Phase
- 原 Phase 顶部"对应 PRD"指向新 PRD

这是"目标替换"场景，本技能不主动处理（由用户决策）。

### 4.6 提交策略（lore commit）

#### 单次提交原则

"Phase 编写 + PRD 占位回填"是一个**逻辑上不可分割的事件**——必须同时落地，否则双向引用断裂。

#### lore commit 格式

```bash
echo '{
  "intent": "补全 PRD 阶段占位并产出 Phase 文档",
  "body": "4 阶段工作流完成：自审 PRD 边界 N 个问题，深审拆解 M 个任务，增量识别 K 个里程碑，精简后 Phase 落盘到 docs/phase/<prd-id>/<seq>-<phase-name>.md，并回填对应 PRD 顶部 TBD 占位。",
  "trailers": {
    "Constraint": ["Phase 必含 conventions.md §4.1 的 5 章节 + 顶部 PRD 反向链接"],
    "Rejected": ["独立路径 prd-outputs/phase/ | 与 conventions.md §1.1 冲突"],
    "Directive": ["PRD ↔ Phase 双向引用由 conventions.md §3.3/§4.3 强制"],
    "Confidence": "high",
    "Tested": ["PRD 顶部 TBD 已被替换", "Phase 顶部含 PRD 反向链接", "子代理 5 项审查通过"],
    "Not-tested": []
  }
}' | lore commit
```

### 4.7 阶段 4 的最终检查

在说"完成"之前，确认：

- [ ] 工作产物已合并到 Phase 主文档（分文件已删除）
- [ ] 所有与 PRD 重复的内容已删（只留引用）
- [ ] 所有决策过程已移入 `docs/architecture/decisions.md`
- [ ] `grep` 确认零悬空引用
- [ ] 子代理 5 项审查完成，反馈已处理
- [ ] **PRD 顶部 TBD 已被替换为真实路径**
- [ ] **Phase 顶部 `> 对应 PRD:` 反向链接已建立**
- [ ] **命名符合 `conventions.md §2.2`**：`<prd-id>/<seq>-<phase-name>.md`
- [ ] **工作产物已清理**：`docs/phase/.working/<phase>/` 已删除
- [ ] **lore commit 已提交**（不是 `git commit`）
- [ ] 你愿意让一个新来的工程师**只读 Phase + PRD 就开始执行**——如果可以，基线就完成了
