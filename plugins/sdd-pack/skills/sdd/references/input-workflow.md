# idea -> spec 工作流（4 阶段方法论）

---

## 总览

```
澄清（对话追问具象化）-> 定边界（v1 范围）-> 显性化假设（找出隐藏前提）-> 交付（结构化 spec）
```

每阶段的本质：

- **阶段 1 澄清**：对话而非生成--问对问题让用户想清楚，不替用户做产品决定
- **阶段 2 定边界**：砍--把"什么都是 v1 必做"砍成"v1 必做 / v1 推迟 / 永远不做"
- **阶段 3 显性化假设**：挖--把没说出来的前提显性化、分类、定后果
- **阶段 4 交付**：结构化输出--把前三阶段对话收集到的信息填入模板，不替用户做技术选型

---

## 1. 澄清（阶段 1）

### 1.1 阶段本质

阶段 1 是**对话而非生成**--你（主上下文）的工作不是写 spec，而是**问对问题让用户想清楚**。

很多技能在阶段 1 容易"替用户做决定"--"我理解你想要 X 对吧？"--这种做法看似高效，但**会丢失用户的真实需求**。

更好的做法是：**让用户用自己的话说**。如果用户说不出来，要么是用户没想清楚（需要继续追问），要么是用户没需求（需要砍范围）。

### 1.2 5 类核心追问模式

#### 模式 1：反问"你说的 X 具体指什么"

适用于用户用模糊的词（如"好用"、"友好"、"专业"、"稳定"）。

例：

- 用户："我希望界面友好"
- 你："你说的'友好'，是指什么？视觉上简洁？还是操作步骤少？还是反馈信息清楚？能举个例子吗？"

#### 模式 2：问"不做会怎样"

适用于用户表达"我们想做 X"。

例：

- 用户："我们想做实时协作"
- 你："如果不做实时协作，只用 5 分钟同步，用户的体验会差到无法接受？还是只是'没那么即时'？"

这个反问很关键--它把"必须做"和"可推迟"区分开。

#### 模式 3：问"成功的具体样子"

适用于用户说"做出来"或"上线"。

例：

- 用户："上线后应该很受欢迎"
- 你："能不能描述一下，上线后第一天你打开产品看到什么会觉得'成功了'？"

成功的具体样子可以反向推导验收标准。

#### 模式 4：问"如果反过来"

适用于用户表达方向或偏好。

例：

- 用户："我们必须用云服务"
- 你："什么情况下你会考虑私有化？反过来，私有化的什么点你最不能接受？"

这个反问能挖出假设（"为什么必须用云服务"背后的真实原因）。

#### 模式 5：问"具体数字"

适用于用户用量词但没具体数字。

例：

- 用户："会有很多用户"
- 你："你说的'很多'，是 100、1 千、1 万、还是 10 万？预估每天活跃用户？"

数字会改变很多技术决策--不能含糊。

### 1.3 3 个必问反问

无论什么项目，下面 3 个反问都应该问--它们能挖出"用户没明说但实施时会撞上"的假设。

#### "如果用户其实不想用呢？"

- **目的**：验证"需求真实"--不是"我觉得用户需要"，而是"用户真的会来用"
- **回答模式**：
  - "不会" / "肯定有需求" -> 必假定，但要追问"为什么这么确定"
  - "可能会" / "看推广" -> 必验证
  - "其实……" -> 不成立，调整范围

#### "什么会让这件事失败？"

- **目的**：让用户主动暴露"红线"--这些红线就是永远不做
- **回答模式**：
  - "如果 X" -> 提取为"必验证"或"必假定"
  - "永远不会 X" -> 提取为"永远不做"

#### "上线后第一个月看到什么算成功？"

- **目的**：把"成功"具象化--是验收标准的源头
- **回答模式**：
  - 具体指标（"1 万用户"、"100 单交易"）-> 提取为业务验收标准
  - "看到用户反馈" -> 追问"具体什么反馈"
  - "看到用户继续用" -> 追问"如何衡量继续用"

### 1.4 不同类型输入的追问侧重

#### 输入：口语化一句话

- **例**："我想做一个给独立开发者用的项目管理工具"
- **重点追问**："做给谁"、"解决什么"、"做成什么样"（一句话想法这两点最弱）+ "如果独立开发者其实不想用项目管理工具呢？"（寻找真痛点）
- **难点**：用户对自己想要的可能也模糊

#### 输入：客户会议纪要

- **例**：一段 2000 字的会议纪要
- **重点追问**："做给谁"、"解决什么"、"不做会怎样"（纪要可能含混）+ "v1 必做/推迟/永远不做"（纪要经常列很多功能但没说优先级）+ 部署目标、团队、预算、合规（纪要常常不写约束）
- **难点**：纪要可能含混或者多 stakeholder 诉求冲突

#### 输入：用户访谈笔记

- **例**：5 段用户访谈的纪要
- **重点追问**："解决什么"、"做成什么样"（从用户原话里抓痛点）+ "v1 必做/推迟"（用户说要的不等于真做）+ "如果反过来"、"用户怎么看"（验证假设）
- **难点**：用户访谈中说"想要"不等于"真用"--要验证

#### 输入：竞品分析报告

- **例**：分析 Notion、Airtable、Linear 的功能对比
- **重点追问**："做给谁"、"解决什么"、"不做会怎样"（竞品分析里这两个最弱）+ "永远不做"（和竞品不做什么）+ 团队能力（决定能不能和竞品拼）
- **难点**：分析容易"我要做 X 竞品的全部"--要砍

### 1.5 怎么判断"问够了"

阶段 1 完成的标志是：你能用 2-3 句话向第三方**清楚**说明以下 4 件事：

1. **做的是什么**（功能）--一句话说清
2. **给谁用**（用户）--明确用户角色
3. **解决什么**（价值）--具体痛点
4. **不做什么**（边界）--反边界

如果 4 件都说清楚了，可以进阶段 2。

如果其中任一项说"看情况"、"差不多"、"都可以"--继续追问。

### 1.6 反模式清单

- ❌ **替用户做产品决定**："我理解你想要 Notion 风格的工具对吧？"--用户可能根本没想好，你替他"框定"了
- ❌ **一次问 10+ 个问题**：甩一大串问题给用户--用户疲劳，回答会敷衍。正确做法：分批问，每批 2-3 个
- ❌ **跳过阶段 1 直接给 spec**：用户给一段输入，主上下文 30 秒后就给 spec--背景信息严重缺失，spec 质量差
- ❌ **把"用户想法"当"用户需求"**：用户说"我们想加 AI"，主上下文就把 AI 当需求--前者是"我觉得 X"，后者是"用户痛 Y"。要追问"用户怎么用 AI"
- ❌ **把"目标"当"功能"**：用户说"提升客户满意度"，主上下文写"加满意度调查功能"--满意度是结果不是功能。要追问"做什么能让满意度提升"
- ❌ **替用户做选择**："你是不是想用 React？"
- ❌ **问实现细节**：数据库、API 设计、UI 库--这些不是阶段 1 的事
- ❌ **接受模糊回答**："差不多"+"看情况"--要追问到能拍板

### 1.7 与阶段 2 的衔接

阶段 1 完成后，**核心 5 问**的回答会成为阶段 2 的"v1 必做"的基础：

- "做给谁" -> 阶段 2 的"为谁服务"
- "解决什么" -> 阶段 2 的"价值判定"
- "做成什么样" -> 阶段 2 的"完成标准"
- "不做会怎样" -> 阶段 2 的"永远不做"
- "何时上线" -> 阶段 2 的"v1/v2 时间窗口"

阶段 1 答得越清楚，阶段 2 的范围划定越快。

## 2. 定边界（阶段 2）

### 2.1 阶段本质

阶段 2 的核心动作是**砍**--把"什么都是 v1 必做"砍成"v1 必做 / v1 推迟 / 永远不做"。

砍范围比加范围难--需要明确**为什么不做**，需要识别**什么对 v1 完整不必要**。

如果 v1 必做列超过 12 个，几乎一定砍错了。

### 2.2 三分法的实际运用

#### v1 必做（v1 Must-Have）

**判断标准**：删掉任何一个都让 v1 失去"完整产品"的感觉。

- 例子：登录功能--没有登录用户不能保存自己的数据
- 例子：核心交易流程--没有这个 v1 就不是这个产品

**反例**（不该是 v1 必做）：

- "个性化推荐"--v1 没有用户也能用
- "导出 PDF"--v1 不导出也能用
- "深色模式"--v1 浅色模式也能用

#### v1 推迟（v1 Defer）

**判断标准**：删掉后 v1 仍然完整（不那么完美，但能用）。

**必须有明确的时间窗口**：

- "v1.x 看数据"--具体是 v1.1 还是 v1.2？
- "v2 评估"--v2 启动时再决定
- "等用户量过 10 万"--具体阈值是什么？

**反例**（不该是 v1 推迟）：

- "v1 推迟" 没有时间窗口 -> 实际是"没决定"
- "v1 推迟" 后面列了一堆 -> 实际上是"v1 不做"但不敢说

#### 永远不做（Never）

**判断标准**：明确砍掉--不是"以后做"，是"做了反而稀释核心价值"。

**真正的"永远不做"**：

- 与产品定位冲突的（如"我们做项目管理，不做团队聊天"）
- 价值过低但成本高的（如"我们不做客户端"）
- 用户不感兴趣的（如"我们不做 AI 集成"--用户访谈显示用户不要）

**反例**（不该是"永远不做"）：

- "暂时不做" -> 实际是"v1 推迟"
- "等团队变大再做" -> 实际是"v1 推迟"
- "看用户需求" -> 实际是"v1 推迟"

### 2.3 砍范围的 4 个方法

#### 方法 1：极简主义（"如果 v1 只能保留 3 个功能"）

**问法**："如果 v1 只能保留 3 个功能，会是哪 3 个？"

**为什么有效**：强迫用户从"什么都重要"变成"什么最重要"。

#### 方法 2：MVP 思维（"最小可用"是什么）

**问法**："MVP（最小可用产品）能跑起来吗？"

**判断标准**：MVP 完成后用户能"完成一个完整任务"--即使不是最优体验。

#### 方法 3：价值-成本矩阵

让用户对每个功能做 2 维评估：

- 价值（高 / 中 / 低）
- 实施成本（高 / 中 / 低）

|        | 价值高  | 价值中   | 价值低   |
| ------ | ------- | -------- | -------- |
| 成本低 | v1 必做 | v1 必做  | v1 推迟  |
| 成本中 | v1 必做 | v1 推迟  | 永远不做 |
| 成本高 | v1 推迟 | 永远不做 | 永远不做 |

#### 方法 4：删除法（"如果不做 X 会怎样"）

**问法**：对每个候选 v1 必做功能问"如果不做会怎样"。

**判断标准**：

- "用户会无法完成核心任务" -> 必做
- "用户体验差" -> 推迟
- "做了用户也没注意" -> 永远不做

### 2.4 范围蔓延的早期信号

砍范围时关注这些信号--出现时警觉：

- "v1 必做列越来越长" -> 已经蔓延
- "v1 推迟变成了'没想好'" -> 没决定
- "永远不做列空白" -> 没做反边界识别
- "v1 必做里出现'未来'" -> 描述模糊
- "v1 必做里有'等 X 团队搞定'" -> 跨团队依赖未管理

### 2.5 范围的可测试性

每个 v1 必做必须**可测试**--"做完了用户能验证"。

**可测试**：

- "用户能创建文章" -> 验证：用户点"创建"按钮，看到表单
- "搜索响应时间 < 500ms" -> 验证：测速

**不可测试**（要改写）：

- "界面要友好" -> 改写为"完成主流程不超过 3 次点击"
- "性能要好" -> 改写为"P95 < 500ms"

不可测试的功能定义 = 实施时各做各的 = 范围失控。

### 2.6 范围与验收标准、非功能需求的关系

**与验收标准**：

- 每个 v1 必做至少有 1-2 条验收标准
- 验收标准是"完成的定义"--"做完了吗"看这个
- 阶段 4 的 spec 模板里"验收标准"章节就是把这里的列表搬到文档里

**与非功能需求**：

非功能需求（性能、安全、合规）**不是"v1 必做"**--它们是"v1 必满足"。

处理方式：

- 在 spec 模板"非功能需求"章节单独列
- 不混在 v1 必做里（否则"性能要快"会和"用户能创建文章"并列--层次混乱）

### 2.7 反模式

#### 反模式 1：v1 必做列 20+ 个

**为什么错**：列越长，每个都越不"必"--实施时全部要排优先级，列的价值消失。

**正确做法**：v1 必做 ≤ 12 个。多出来的进 v1 推迟。

#### 反模式 2：没有"永远不做"

**为什么错**：一个产品如果什么都做，必然什么都做不好。"永远不做"是定位的镜子。

**正确做法**：至少 1-3 项永远不做。和"v1 推迟"区分开。

#### 反模式 3：v1 推迟没有时间窗口

**为什么错**："v1 推迟"是延迟决定，不是决定--后期会变成"待定"。

**正确做法**：每项 v1 推迟明确说"什么时候再说"。

#### 反模式 4：用户说"都要"时妥协

**为什么错**：用户说"都要" = 没做减法 = 范围失控的开始。

**正确做法**：用"3 个功能"反问强迫减法。

### 2.8 阶段完成标志

- v1 必做 ≤ 12 个
- v1 推迟都有时间窗口
- 永远不做至少 1 项
- 每个 v1 必做都有验收标准

完成后可进入阶段 3。

## 3. 显性化假设（阶段 3）

### 3.1 阶段本质

阶段 1 和 2 解决"是什么"和"做多少"。阶段 3 解决"基于什么"--所有没说出来的前提。

每个项目都有几十个"我觉得是 X"的假设。它们藏在用户的语气里、对话的省略里、共识的沉默里。

阶段 3 的工作是把它们挖出来、分类、定后果。

### 3.2 假设的 3 种类型

#### 必假定（不可质疑的边界）

**特征**：推翻后**整个项目重构**。

**例子**：

- "客户在私有化环境部署" -> 推翻 -> 改 SaaS -> 完全不同的技术栈
- "v1 必须支持 10 万用户" -> 推翻 -> 砍一半用户范围
- "必须支持中文" -> 推翻 -> 改英文/加 i18n

**处理方式**：

- 和用户**明确确认**为前提
- 写进 spec 文档"约束"章节
- PRD 工作流阶段 2 不会质疑必假定

#### 必验证（实施前必须确认）

**特征**：推翻后**部分功能/选型**调整，但不重构。

**例子**：

- "团队 6 人有 React 经验" -> 验证 -> 没经验则改 Vue 或培训
- "客户有 AWS 账号" -> 验证 -> 没有则改其他云
- "数据量 < 1 亿行" -> 验证 -> 多了则改 OLAP 引擎
- "客户内部有 SMTP 服务器" -> 验证 -> 调整邮件方案

**处理方式**：

- 列入 spec 文档"未决假设"章节
- PRD 工作流阶段 2 会**优先质疑**这些
- 实施前必须有一个验证步骤

#### 不成立（被对话证伪）

**特征**：问清楚后发现**用户原本说错了**。

**例子**：

- 用户说"我们必须做实时协作" -> 追问"5 分钟延迟用户能接受吗" -> "其实也行" -> 假设不成立 -> 不需要实时协作
- 用户说"必须有 Web 端" -> 追问"用户场景" -> "其实主要是移动端" -> 假设不成立 -> v1 可以先做移动端

**处理方式**：

- **更新 v1 范围**（而不是写进假设清单）
- 阶段 2 的范围划定可能要回调

### 3.3 挖掘假设的 4 个工具

#### 工具 1："如果 X 错了会怎样"

**问法**："如果 {用户说的话} 其实不是这样呢？"

**适用**：用户用了"当然"、"肯定"、"必须"、"绝对"等强词

**例**：

- 用户："客户肯定有外网"
- 你："如果客户环境其实没外网呢？我们会怎么办？"
- 用户："用户肯定有这个需求"
- 你："如果用户其实不想用这个功能呢？"
- 用户："团队肯定能写 React"
- 你："如果团队没人会 React 呢？"

**答案分析**：

- "不可能没外网" -> 必假定
- "如果没外网，我们就改私有化部署" -> 必验证 + 改方案
- "其实我也不知道" -> 必验证

#### 工具 2："什么会让这件事失败"

**问法**："什么样的情况会让你说'这个项目不该上'？"

**适用**：用户说"没问题"或"肯定能行"

**例**：

- "如果产品上线后没人用，你会觉得是什么导致的？" -> 暴露对用户的假设
- "如果项目延后 3 个月，会出什么问题？" -> 暴露对时间的假设

**答案分析**：

- 具体场景（"用户不上线"）-> 必验证
- 永远不发生（"用户必然会用"）-> 必假定
- "我想不到" -> 必验证（因为用户没想到）

#### 工具 3："反过来想"

**问法**："你说的 X，反过来 Y 你怎么看？"

**适用**：用户表达偏好或方向

**例**：

- 用户："我们必须用云服务"
- 你："什么情况下你会考虑私有化？反过来，私有化的什么点你最不能接受？"
- 用户："要快速上线"
- 你："那'慢但稳'的方案你怎么看？"
- 用户："要做 SaaS"
- 你："那'私有化'你会考虑吗？"

**答案分析**：

- 用户能说清楚反例 -> 必验证（"我想要云的弹性"）
- 用户坚持 -> 必假定

#### 工具 4："具体到数字"

**问法**："你说'很多'/'很快'/'很大'--具体是多少？"

**适用**：用户用量词但没具体数字

**例**：

- "性能要快" -> "多快？100ms 还是 1s？"
- "用户量大" -> "日活 1 千还是 10 万？"
- "预算有限" -> "上限是 1 万、10 万、还是 100 万？"

数字会改变很多决策--不能含糊。

### 3.4 常见假设分类

按"假设来源"分类，可以分批挖：

#### 用户类假设

- "用户真的有这个需求"
- "用户愿意改变当前行为"
- "用户会按我们设想的方式使用"
- "用户的付费意愿"
- "用户能接受 {某种体验}"

#### 技术类假设

- "团队技能匹配"（React？Go？K8s？）
- "第三方服务可获得"（API、库、SDK 是否稳定）
- "部署环境的限制"（外网/内网、资源限制）
- "性能要求能达到"
- "数据规模在可处理范围"

#### 业务类假设

- "客户/老板的期望"
- "竞品的反应"
- "商业模式的有效性"
- "市场窗口期"
- "合作伙伴的支持"

#### 时间类假设

- "团队投入的时间"
- "上下游依赖的时间"
- "试错周期"

#### 法律合规类假设

- "数据合规要求"（GDPR、HIPAA、PCI）
- "行业监管要求"
- "跨境数据传输"
- "未成年人保护"

### 3.5 假设清单的字段

每条假设应该包含：

| 字段     | 含义                     |
| -------- | ------------------------ |
| #        | 编号                     |
| 假设     | 一句话说清               |
| 类型     | 必假定 / 必验证 / 不成立 |
| 验证方法 | 怎么确认（仅必验证）     |
| 推翻后果 | 如果假设错了会怎样       |

> 填空式模板见 `templates/assumptions.md`。

### 3.6 反模式

#### 反模式 1：假设 = "我们没想过"

**为什么错**：显性化才是目的--"没想过"是盲区，必须挖出来。

**正确做法**：每个"我们没想过"挖出背后的真正问题。

#### 反模式 2：所有假设都"必假定"

**为什么错**：很多假设其实是"必验证"或"不成立"--把它们当必假定 = 假装项目没风险。

**正确做法**：分类时严格--推翻后**真**重构才是必假定。

#### 反模式 2b：不验证就假定

**为什么错**：必验证的假设如果不验证，到实施阶段会爆发--正确分类了但没执行验证步骤。

**正确做法**：必验证假设在实施前必须有明确的验证动作，不能停留在"已标注"。

#### 反模式 3：假设清单过长（> 20 条）

**为什么错**：清单失去重点--后期没人看 20 条清单。

**正确做法**：聚焦"5-10 条最重要的"--其余在阶段 1/2 已经处理。

#### 反模式 4：跳过阶段 3

**为什么错**：不显性化的假设在实施阶段会突然爆发--代价是返工。

**正确做法**：阶段 3 是 4 个阶段里"看起来最没用"但"最不能省"的。

### 3.7 阶段完成标志

- 至少 5-10 个假设被显性化
- 每条假设有类型和后果说明
- "不成立"的假设已更新到 v1 范围
- "必验证"的假设已列入 spec 的"未决假设"章节
- "必假定"的假设已和用户确认接受为前提

### 3.8 与 PRD 工作流阶段 2 的衔接

`PRD 工作流` 阶段 2（深审）会**优先质疑**阶段 3 的"必验证"假设：

- 阶段 3 说"必验证：客户有 AWS 账号"
- PRD 工作流阶段 2 会说"如果客户没 AWS 账号会怎样？需要重选部署吗？"
- 这会变成 ADR 中的一个"质疑-决策"对

因此，阶段 3 的"必验证"假设清单是 PRD 工作流阶段 2 的**重点扫描对象**--不是普通的文档章节。

## 4. 交付 spec（阶段 4）

### 4.1 阶段本质

阶段 4 是**结构化输出**--把前三阶段对话收集到的信息填入模板。

但"填模板"不等于"ctrl-C ctrl-V"--每个章节都要**严格基于用户回答**。

最大的风险：**替用户做技术选型**。spec 文档不是 PRD，技术选型不在此阶段。一旦写了"前端用 React"--PRD 工作流阶段 2 的质疑空间就被压缩了。

### 4.2 模板填充的 3 个原则

#### 原则 1：严格基于用户回答

**不替用户做决定**--用户说"X"，就写"X"，不写"X 类似 Y"。

例：

- 用户说"我们想做给独立开发者用的项目管理工具"
- ✅ spec 写"目标用户：独立开发者"
- ❌ spec 写"目标用户：类似 Notion 的中小型团队"（加了用户没说过的内容）

#### 原则 2：选型未决

**任何"应该用 X"的判断都不写**--这是 PRD 工作流阶段 2 的工作。

例：

- 用户说"我们想做实时协作" -> 写"v1 必做：实时协作"
- 不写"实时协作可以用 WebSocket 实现"（这是选型）
- 不写"实时协作需要后端支持长连接"（这是实施细节）

#### 原则 3：每个章节不空白

**模板的每个章节都要填**--空白章节是 PRD 工作流阶段 1 自审的隐患。

如果某章节真的"待定"--写"待定"并解释**为什么待定**和**什么时候确定**。

### 4.3 9 个章节的填写要点

#### 1. 问题陈述

**来源**：阶段 1 的"解决什么"

填写要点：

- 一句话主陈述（不超过 30 字）
- 2-3 句背景（现状 + 目标状态）
- 不要写愿景（"赋能企业、引领行业"是噪声）

#### 2. 目标用户

**来源**：阶段 1 的"做给谁"

填写要点：

- 主要用户（必须）
- 次要用户（如适用）
- 反例用户（防止 scope creep）-- 容易漏的

#### 3. v1 功能边界

**来源**：阶段 2 的三分法

填写要点：

- v1 必做 ≤ 12 个
- 每个 v1 必做都带验收标准
- v1 推迟必须有"什么时候再说"
- 永远不做至少 1 项

#### 4. 核心场景

**来源**：阶段 1 的"成功的具体样子" + 阶段 2 的 v1 必做

填写要点：

- 用**用户语言**描述（不是技术语言）
- 2-3 个场景--不要堆 10 个
- 每个场景 1 段话，含步骤和结果

#### 5. 验收标准

**来源**：阶段 2 的 v1 必做 + 阶段 1 的"成功的样子"

填写要点：

- 每条都可测试（不要写"界面要友好"）
- 区分功能验收 / 业务验收 / 时间验收
- 时间验收明确节点

#### 6. 非功能需求

**来源**：阶段 1 的约束追问 + 阶段 3 的必假定

填写要点：

- 性能 / 可用性 / 安全 / 合规 / 部署 / 国际化 / 可维护性
- 每项有具体值或"待定"标注
- 不混在 v1 必做里（层次不同）

#### 7. 未决假设

**来源**：阶段 3 的"必验证"假设

填写要点：

- 至少 5-10 条
- 每条有类型（必假定 / 必验证）和后果说明
- 这是 PRD 工作流阶段 2 的**重点扫描对象**

#### 8. 选型未决

**来源**：显式声明

填写要点：

- 一段话明确"本 spec 不锁任何技术选型"
- 说明选型由 PRD 工作流阶段 2 决定

#### 9. 参考材料

**来源**：用户的原始输入

填写要点：

- 列出所有原始材料（笔记、纪要、邮件、聊天记录）
- 保留可追溯性
- 后续 PRD 工作流阶段 1 自审可能需要回溯

### 4.4 子代理交叉检查（可选但推荐）

用 Task 工具启动子代理（agent_type="general-purpose"），prompt 模板：

```markdown
请审查以下 spec 文档的完整性和一致性：

- {path}/spec.md

执行 4 项检查：

1. 章节完整性：9 个章节都有内容吗？空白章节 = 隐患
2. 选型是否泄露：检查是否写了技术选型（前端框架、数据库、部署方案）--如果有，必须移到 PRD 工作流阶段 2
3. 假设可验证性：未决假设的"验证方法"具体吗？抽象的 = 实施时无法验证
4. 验收标准可测试性：每条验收标准都"能直接测"吗？"用户能验证"吗？

报告每个检查项的结论和建议。不执行修改（只审不改）。
```

子代理的反馈用来补全 spec--不修改"不替用户做决定"原则。

### 4.5 反模式

#### 反模式 1：替用户做选型

**症状**：spec 里出现"应该用 X"、推荐某框架、某数据库

**为什么错**：选型是 PRD 工作流阶段 2 的事

**正确做法**：所有选型写"未决"或直接不写

#### 反模式 2：写 PR 风格的成功愿景

**症状**："赋能企业、引领行业、打造 X 标杆"

**为什么错**：稀释信号噪声比，浪费读者时间

**正确做法**：用 30 字主陈述说清问题

#### 反模式 3：章节不全

**症状**：只写功能章节，跳过非功能需求、跳过未决假设

**为什么错**：PRD 工作流阶段 3 的"实施约束"会无依据

**正确做法**：9 章节全填

#### 反模式 4：验收标准不可测试

**症状**："界面要友好"、"性能要好"、"系统要稳定"

**为什么错**：实施时各做各的 = 范围失控

**正确做法**：每条都有具体数字或可观察标准

#### 反模式 5：参考材料缺失

**症状**：只写最终 spec，不附原始输入

**为什么错**：失去可追溯性，后期回溯困难

**正确做法**：列出所有原始材料

### 4.6 阶段完成标志

- 9 个章节全部有内容
- 没有写具体技术选型
- v1 必做 ≤ 12 个，永远不做 ≥ 1
- 未决假设 ≥ 5
- 验收标准每条都可测试
- 子代理审查（如有）已处理
- 用户确认 spec 完整

### 4.7 与下游的衔接

#### 完成后做的事

- 展示 spec 给用户确认
- 提示用户："可以继续 PRD 工作流--指向 `docs/spec/YYYY-MM-DD-<name>.md` 即可"

#### 接力时可能遇到的问题

- 用户说"我先改改"--支持
- 用户说"再加一项"--回到阶段 2 重新划范围
- 用户说"某处不要了"--回到对应阶段（视改动大小）
