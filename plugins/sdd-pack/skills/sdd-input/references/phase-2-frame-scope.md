# 阶段 2 深度参考：定边界（Frame Scope）

> 加载时机：阶段 1 完成、用户能描述"做的是什么"但说不清"做多少"、需要砍范围时。

> **sdd-core 适配说明**:本阶段产物(三分法 v1 必做/推迟/不做)写入 `docs/spec/YYYY-MM-DD-<name>.md` §3 v1 功能边界章节。模板见 `templates/spec-template.md`。

---

## 阶段 2 的本质

阶段 2 的核心动作是**砍**——把"什么都是 v1 必做"砍成"v1 必做 / v1 推迟 / 永远不做"。

砍范围比加范围难——需要明确**为什么不做**，需要识别**什么对 v1 完整不必要**。

如果 v1 必做列超过 12 个，几乎一定砍错了。

---

## 三分法的实际运用

### v1 必做（v1 Must-Have）

**判断标准**：删掉任何一个都让 v1 失去"完整产品"的感觉。

- 例子：登录功能——没有登录用户不能保存自己的数据
- 例子：核心交易流程——没有这个 v1 就不是这个产品

**反例**（不该是 v1 必做）：

- "个性化推荐"——v1 没有用户也能用
- "导出 PDF"——v1 不导出也能用
- "深色模式"——v1 浅色模式也能用

### v1 推迟（v1 Defer）

**判断标准**：删掉后 v1 仍然完整（不那么完美，但能用）。

**必须有明确的时间窗口**：

- "v1.x 看数据"——具体是 v1.1 还是 v1.2？
- "v2 评估"——v2 启动时再决定
- "等用户量过 10 万"——具体阈值是什么？

**反例**（不该是 v1 推迟）：

- "v1 推迟" 没有时间窗口 → 实际是"没决定"
- "v1 推迟" 后面列了一堆 → 实际上是"v1 不做"但不敢说

### 永远不做（Never）

**判断标准**：明确砍掉——不是"以后做"，是"做了反而稀释核心价值"。

**真正的"永远不做"**：

- 与产品定位冲突的（如"我们做项目管理，不做团队聊天"）
- 价值过低但成本高的（如"我们不做客户端"）
- 用户不感兴趣的（如"我们不做 AI 集成"——用户访谈显示用户不要）

**反例**（不该是"永远不做"）：

- "暂时不做" → 实际是"v1 推迟"
- "等团队变大再做" → 实际是"v1 推迟"
- "看用户需求" → 实际是"v1 推迟"

---

## 砍范围的 4 个方法

### 方法 1：极简主义（"如果 v1 只能保留 3 个功能"）

**问法**："如果 v1 只能保留 3 个功能，会是哪 3 个？"
**为什么有效**：强迫用户从"什么都重要"变成"什么最重要"。

### 方法 2：MVP 思维（"最小可用"是什么）

**问法**："MVP（最小可用产品）能跑起来吗？"
**判断标准**：MVP 完成后用户能"完成一个完整任务"——即使不是最优体验。

### 方法 3：价值-成本矩阵

让用户对每个功能做 2 维评估：

- 价值（高 / 中 / 低）
- 实施成本（高 / 中 / 低）

|        | 价值高  | 价值中   | 价值低   |
| ------ | ------- | -------- | -------- |
| 成本低 | v1 必做 | v1 必做  | v1 推迟  |
| 成本中 | v1 必做 | v1 推迟  | 永远不做 |
| 成本高 | v1 推迟 | 永远不做 | 永远不做 |

### 方法 4：删除法（"如果不做 X 会怎样"）

**问法**：对每个候选 v1 必做功能问"如果不做会怎样"。
**判断标准**：

- "用户会无法完成核心任务" → 必做
- "用户体验差" → 推迟
- "做了用户也没注意" → 永远不做

---

## 范围蔓延的早期信号

砍范围时关注这些信号——出现时警觉：

- "v1 必做列越来越长" → 已经蔓延
- "v1 推迟变成了'没想好'" → 没决定
- "永远不做列空白" → 没做反边界识别
- "v1 必做里出现'未来'" → 描述模糊
- "v1 必做里有'等 X 团队搞定'" → 跨团队依赖未管理

---

## 范围的可测试性

每个 v1 必做必须**可测试**——"做完了用户能验证"。

**可测试**：

- "用户能创建文章" → 验证：用户点"创建"按钮，看到表单
- "搜索响应时间 < 500ms" → 验证：测速

**不可测试**（要改写）：

- "界面要友好" → 改写为"完成主流程不超过 3 次点击"
- "性能要好" → 改写为"P95 < 500ms"

不可测试的功能定义 = 实施时各做各的 = 范围失控。

---

## 范围与验收标准的关系

v1 必做 → 验收标准

- 每个 v1 必做至少有 1-2 条验收标准
- 验收标准是"完成的定义"——"做完了吗"看这个

阶段 4 的 spec 模板里"验收标准"章节就是把这里的列表搬到文档里。

---

## 范围与非功能需求的关系

非功能需求（性能、安全、合规）**不是"v1 必做"**——它们是"v1 必满足"。

处理方式：

- 在 spec 模板"非功能需求"章节单独列
- 不混在 v1 必做里（否则"性能要快"会和"用户能创建文章"并列——层次混乱）

---

## 阶段 2 的反模式

### 反模式 1：v1 必做列 20+ 个

**为什么错**：列越长，每个都越不"必"——实施时全部要排优先级，列的价值消失。
**正确做法**：v1 必做 ≤ 12 个。多出来的进 v1 推迟。

### 反模式 2：没有"永远不做"

**为什么错**：一个产品如果什么都做，必然什么都做不好。"永远不做"是定位的镜子。
**正确做法**：至少 1-3 项永远不做。和"v1 推迟"区分开。

### 反模式 3：v1 推迟没有时间窗口

**为什么错**："v1 推迟"是延迟决定，不是决定——后期会变成"待定"。
**正确做法**：每项 v1 推迟明确说"什么时候再说"。

### 反模式 4：用户说"都要"时妥协

**为什么错**：用户说"都要" = 没做减法 = 范围失控的开始。
**正确做法**：用"3 个功能"反问强迫减法。

---

## 阶段 2 完成标志

- v1 必做 ≤ 12 个
- v1 推迟都有时间窗口
- 永远不做至少 1 项
- 每个 v1 必做都有验收标准

完成后可进入阶段 3。
