# 阶段 3 深度参考：显性化假设（Surface Assumptions）

> 加载时机：用户表达非常肯定、范围已定但需要验证前提、阶段 2 完成后需要继续追问时。

> **sdd-core 适配说明**:本阶段产物(假设清单)写入 `docs/spec/.working/YYYY-MM-DD-<name>/assumptions.md`(留痕)+ spec §7 未决假设章节。模板见 `templates/assumptions-template.md`。

---

## 阶段 3 的本质

阶段 1 和 2 解决"是什么"和"做多少"。阶段 3 解决"基于什么"——所有没说出来的前提。

每个项目都有几十个"我觉得是 X"的假设。它们藏在用户的语气里、对话的省略里、共识的沉默里。

阶段 3 的工作是把它们挖出来、分类、定后果。

---

## 假设的 3 种类型

### 必假定（不可质疑的边界）
**特征**：推翻后**整个项目重构**

例子：
- "客户在私有化环境部署" → 推翻 → 改 SaaS → 完全不同的技术栈
- "v1 必须支持 10 万用户" → 推翻 → 砍一半用户范围
- "必须支持中文" → 推翻 → 改英文/加 i18n

**处理方式**：
- 和用户**明确确认**为前提
- 写进 spec 文档"约束"章节
- sdd-prd 阶段 2 不会质疑必假定

### 必验证（实施前必须确认）
**特征**：推翻后**部分功能/选型**调整，但不重构

例子：
- "团队 6 人有 React 经验" → 验证 → 没经验则改 Vue 或培训
- "客户有 AWS 账号" → 验证 → 没有则改其他云
- "数据量 < 1 亿行" → 验证 → 多了则改 OLAP 引擎

**处理方式**：
- 列入 spec 文档"未决假设"章节
- sdd-prd 阶段 2 会**优先质疑**这些
- 实施前必须有一个验证步骤

### 不成立（被对话证伪）
**特征**：问清楚后发现**用户原本说错了**

例子：
- 用户说"我们必须做实时协作" → 追问"5 分钟延迟用户能接受吗" → "其实也行" → 假设不成立 → 不需要实时协作
- 用户说"必须有 Web 端" → 追问"用户场景" → "其实主要是移动端" → 假设不成立 → v1 可以先做移动端

**处理方式**：
- **更新 v1 范围**（而不是写进假设清单）
- 阶段 2 的范围划定可能要回调

## 挖掘假设的 4 个工具

### 工具 1："如果 X 错了会怎样"
**问法**："如果 {用户说的话} 其实不是这样呢？"

适用：用户用了"当然"、"肯定"、"必须"、"绝对"等强词

例：
- 用户："客户肯定有外网"
- 你："如果客户环境其实没外网呢？我们会怎么办？"

答案分析：
- "不可能没外网" → 必假定
- "如果没外网，我们就改私有化部署" → 必验证 + 改方案
- "其实我也不知道" → 必验证

### 工具 2："什么会让这件事失败"
**问法**："什么样的情况会让你说'这个项目不该上'？"

适用：用户说"没问题"或"肯定能行"

答案分析：
- 具体场景（"用户不上线"）→ 必验证
- 永远不发生（"用户必然会用"）→ 必假定
- "我想不到" → 必验证（因为用户没想到）

### 工具 3："反过来想"
**问法**："你说的 X，反过来 Y 你怎么看？"

适用：用户表达偏好或方向

例：
- 用户："我们必须用云服务"
- 你："什么情况下你会考虑私有化？反过来，私有化的什么点你最不能接受？"

答案分析：
- 用户能说清楚反例 → 必验证（"我想要云的弹性"）
- 用户坚持 → 必假定

### 工具 4："具体到数字"
**问法**："你说'很多'/'很快'/'很大'——具体是多少？"

适用：用户用量词但没具体数字

例：
- "性能要快" → "多快？100ms 还是 1s？"
- "用户量大" → "日活 1 千还是 10 万？"
- "预算有限" → "上限是 1 万、10 万、还是 100 万？"

数字会改变很多决策——不能含糊。

## 按来源分类的常见假设

### 用户类假设
- "用户真的有这个需求"
- "用户愿意改变当前行为"
- "用户会按我们设想的方式使用"
- "用户的付费意愿"
- "用户能接受 {某种体验}"

### 技术类假设
- "团队技能匹配"（React？Go？K8s？）
- "第三方服务可获得"（API、库、SDK 是否稳定）
- "部署环境的限制"（外网/内网、资源限制）
- "性能要求能达到"
- "数据规模在可处理范围"

### 业务类假设
- "客户/老板的期望"
- "竞品的反应"
- "商业模式的有效性"
- "市场窗口期"
- "合作伙伴的支持"

### 时间类假设
- "团队投入的时间"
- "上下游依赖的时间"
- "试错周期"

### 法律合规类假设
- "数据合规要求"（GDPR、HIPAA、PCI）
- "行业监管要求"
- "跨境数据传输"
- "未成年人保护"

## 假设清单的形态

`templates/assumptions-template.md` 提供了完整模板。每条假设应该包含：

| 字段 | 含义 |
|------|------|
| # | 编号 |
| 假设 | 一句话说清 |
| 类型 | 必假定 / 必验证 / 不成立 |
| 验证方法 | 怎么确认（仅必验证） |
| 推翻后果 | 如果假设错了会怎样 |

## 阶段 3 的反模式

### 反模式 1：假设 = "我们没想过"
**为什么错**：显性化才是目的——"没想过"是盲区，必须挖出来。
**正确做法**：每个"我们没想过"挖出背后的真正问题。

### 反模式 2：所有假设都"必假定"
**为什么错**：很多假设其实是"必验证"或"不成立"——把它们当必假定 = 假装项目没风险。
**正确做法**：分类时严格——推翻后**真**重构才是必假定。

### 反模式 3：假设清单过长（> 20 条）
**为什么错**：清单失去重点——后期没人看 20 条清单。
**正确做法**：聚焦"5-10 条最重要的"——其余在阶段 1/2 已经处理。

### 反模式 4：跳过阶段 3
**为什么错**：不显性化的假设在实施阶段会突然爆发——代价是返工。
**正确做法**：阶段 3 是 4 个阶段里"看起来最没用"但"最不能省"的。

---

## 与 sdd-prd 阶段 2 的衔接

`sdd-prd` 技能阶段 2（深审）会**优先质疑**阶段 3 的"必验证"假设：

- 阶段 3 说"必验证：客户有 AWS 账号"
- sdd-prd 阶段 2 会说"如果客户没 AWS 账号会怎样？需要重选部署吗？"
- 这会变成 ADR 中的一个"质疑-决策"对

因此，阶段 3 的"必验证"假设清单是 sdd-prd 阶段 2 的**重点扫描对象**——不是普通的文档章节。

## 阶段 3 完成标志

- 至少 5-10 个假设被显性化
- 每条假设有类型和后果说明
- "不成立"的假设已更新到 v1 范围
- "必验证"的假设已列入 spec 的"未决假设"章节
- "必假定"的假设已和用户确认接受为前提

完成后可进入阶段 4。
