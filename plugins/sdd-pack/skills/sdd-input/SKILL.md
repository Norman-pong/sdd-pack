# idea → spec 对话式工作流(sdd-core 接力版)

> 本技能是 sdd-core 文档体系的**上游预处理器**——把口语化想法/零散需求,通过对话式追问,产出**结构化 spec** 作为 sdd-prd 的输入。
> 与 sdd-core 的协作边界见 `references/sdd-collaboration.md`。

---

## 1. 这个技能做什么

你有一个**模糊的、零散的、未成文**的需求——可能是一句话的想法、一份会议纪要、一堆用户访谈笔记、客户的微信消息、甚至"我也不知道要啥,就是觉得这件事值得做"。

本技能帮你**通过追问和重构**,把它变成一份**结构化的 spec 文档**——直接作为 `sdd-prd` 技能的输入。

### 1.1 核心原则

- **对话驱动**——每阶段必等用户回答,不能跳
- **不替用户做技术选型**——选型留给 sdd-prd 阶段 2
- **显性化假设**——所有"我以为是"必须摊出来
- **产物可被 sdd-prd 直接消费**——spec 章节结构与 sdd-prd 阶段 1 自审所需的"输入"对齐

### 1.2 与 sdd-core 体系的关系

| 维度 | sdd-core 管 | sdd-input 管 | sdd-prd 管 | sdd-phase 管 |
|---|---|---|---|---|
| 路径 | 整个 docs/ 体系 | `docs/spec/` | `docs/prd/` | `docs/phase/` |
| 产物形态 | 目录结构、索引、提交协议 | 口语化 → 结构化 spec | spec → 目标驱动 PRD | PRD → 阶段任务 |
| 提交 | lore commit | lore commit(同 sdd-core) | lore commit | lore commit |
| 命名 | YYYY-MM-DD | YYYY-MM-DD-<name>.md | YYYY-MM-DD-<name>.md | YYYY-MM-DD-<phase>.md |

### 1.3 与 sdd-prd 的接力流程

```mermaid
flowchart LR
    A["用户口语化想法"] --> B["sdd-input<br/>对话式追问 4 阶段"]
    B --> C["docs/spec/YYYY-MM-DD-name.md<br/>结构化 spec"]
    C --> D["sdd-prd 读取 spec"]
    D --> E["docs/prd/YYYY-MM-DD-name.md<br/>目标驱动 PRD"]
    E --> F["sdd-phase"]
    F --> G["docs/phase/YYYY-MM-DD-phase.md<br/>阶段任务"]
    G --> H["回填 PRD 顶部对应阶段:TBD"]
```

接力时**不需要重新整理**——spec 章节结构与 sdd-prd 阶段 1 自审所需的"输入"对齐,直接调 sdd-prd 并指向 `docs/spec/YYYY-MM-DD-<name>.md`。

## 2. 适用与不适用

**适用:**
- 项目启动期:只有一个想法,还没有 spec
- 需求零散:会议纪要、用户访谈、聊天记录——需要被提炼
- 客户/老板说了一个方向:要系统化整理
- "我想做 X 但说不清楚"——从口语化中萃取出结构
- 跨团队对齐:要让多个 stakeholder 看到同一份需求描述
- sdd-core 体系已初始化(`docs/` 目录存在)

**不适用:**
- **已有完整 spec 文档**——直接用 `sdd-prd`
- 纯 bug 修复/小改动——直接写代码
- 纯实现任务(无新需求)——直接做
- "做技术调研"——这是另一个工作流(候选方案评估)
- sdd-core 未初始化——先用 `sdd-core` 场景 4 初始化 docs/ 体系

---

## 3. 与 sdd-core 的协作模式(强制)

### 3.1 路径与命名

- 产物写入 `docs/spec/YYYY-MM-DD-<name>.md`(对齐 conventions.md §2 命名)
- 命名 `<name>` 用小写字母和连字符,如 `user-auth`、`payment-flow`
- **禁止**:`Spec-001.md`(无日期)、`2026-6-23.md`(日期格式错)、`User_Auth.md`(下划线大写)
- 临时工作产物(questions-asked.md / assumptions.md)放 `docs/spec/.working/YYYY-MM-DD-<name>/`

### 3.2 提交协议

**强制走 lore commit**。完整 JSON trailer schema 见 `rule://lore-protocol` 和 sdd-core §核心工作流程→提交变更。本技能推荐的 trailers:

| Trailer | 推荐内容 |
|---|---|
| `intent` | "将口语化想法提纯为结构化 spec" |
| `Constraint` | `spec 必含 sdd-prd 阶段 1 所需的输入章节` |
| `Rejected` | `旧的 spec-outputs/ 路径 \| 与 conventions.md §1.1 冲突` |
| `Directive` | `spec 是 sdd-prd 的输入,Phase 由 sdd-phase 负责` |
| `Tested` | `用户已确认 v1 范围`, `假设清单已显性化` |

### 3.3 与 sdd-prd 的接力

完成后,提示用户:

> spec 已落盘到 `docs/spec/YYYY-MM-DD-<name>.md`,可直接调用 `sdd-prd` 技能,告知"基于 docs/spec/YYYY-MM-DD-<name>.md 提纯 PRD",sdd-prd 会自动读取并进入 4 阶段工作流。

**接力时序**(避免边界冲突):
- sdd-input **只产出** `docs/spec/`,**不写** `docs/prd/`
- sdd-prd **读取** `docs/spec/` 作为输入,**产出** `docs/prd/`
- 接力点 = `docs/spec/YYYY-MM-DD-<name>.md` 路径

## 4. 4 阶段工作流(对话驱动)

```mermaid
flowchart LR
    S1["阶段 1:澄清"] --"5 问确认"--> S2["阶段 2:定边界"]
    S2 --"v1 必做/推迟/不做"--> S3["阶段 3:显性化假设"]
    S3 --"5-10 个假设"--> S4["阶段 4:交付"]
    S4 --"结构化 spec.md"--> Done["sdd-prd 接力"]
```

**核心原则**:**默认对话驱动**——每阶段等待用户回答后再继续。若用户一次性提供充分信息,可切换批处理模式:将各阶段问题汇总为一条清单,用户一次性回答后直接产出 spec。两种模式都需要用户确认,不替用户决定。

---

### 阶段 1:澄清(Clarify)

**目标**:从模糊想法到能拍板的颗粒度——把"我觉得是 X"变成"X 是 Y"。

**进入条件**:
- 用户有了初始输入(一句话、笔记、纪要、邮件、聊天记录……任何形态)
- 用户愿意回答问题
- sdd-core 已初始化(`docs/spec/` 可写)

**操作步骤**:
1. **通读用户输入,识别 5-10 个最关键的不确定项**。
   - 不确定项 = 拍板时需要用户输入的事项
   - 优先追问"会影响功能范围"的不确定项(其他细节可推迟)
2. **分批提问**。一次问 2-3 个问题(不要一次甩 20 个):
   - **第一批**:核心问题——"做给谁"+"解决什么"+"不做会怎样"
   - **第二批**:范围问题——"v1 必含 vs 可推迟"
   - **第三批**:约束问题——"部署环境、团队、预算、时窗"
3. **让用户用自己话说**,不替用户做决定。常见的反模式是替用户写"标准答案"——必须等用户说。
4. **只问"什么"不问"怎么"**。这一阶段只关心"是什么"(功能、范围、用户、痛点),不关心"怎么做"(技术选型、API 怎么设计、数据库选什么)——后者留给 sdd-prd 阶段 2。

**关键问题模板**(参考 `references/phase-1-clarify.md` 完整清单):

1. **"做给谁"** —— 谁会用?目标用户群体是什么?
2. **"解决什么"** —— 现在他们的痛点是什么?没有这个会怎样?
3. **"做成什么样"** —— 成功的样子能描述一下吗?(让用户描述"完成了会看到什么")
4. **"不做会怎样"** —— 反过来问:哪些是绝对不做的?用户最不想看到的功能是什么?
5. **"何时上线"** —— 时间压力是什么?这会影响 v1/v2 的边界。

**反模式**:
- **替用户回答**——"你是不是想做一个类似 X 的产品?"这会让用户接受你的预设,丢失了真实需求
- **一次问太多**——超过 3-5 个问题用户会疲劳,建议分批
- **问实现细节**——"用 React 还是 Vue"——这种问题不该在阶段 1 出现
- **跳过阶段 1 直接给 spec**——看起来高效,实际上会基于错误预设产出

**完成标志**:
- 用户已回答核心 5 问
- 主上下文能用 2-3 句话向第三方说清"做的是什么、给谁用、解决什么、不做什么"

---

### 阶段 2:定边界(Frame Scope)

**目标**:把"什么都想做"砍成"v1 必做 / v1 推迟 / 永远不做"——避免范围蔓延。

**进入条件**:
- 阶段 1 完成,"是什么"已经清楚
- 用户能描述 v1 想要的样子

**操作步骤**:
1. **列出 v1 必做的 5-10 个核心功能**。每个功能用一句话说清"做了什么 / 解决了什么"。
2. **明确"v1 推迟"**——承认有些功能想做但 v1 不做。每个 v1 推迟项说清"什么时候再说"(v1.x? v2? 看数据?)。
3. **明确"永远不做"**——识别用户不感兴趣的、违反核心价值的方向。
4. **验证最小可用**——"如果 v1 只剩 3 个功能,能发布吗?"如果不能,要么是最小集不准确,要么 v1 范围太大。

**三分法的实际用法**:

| 类别 | 含义 | 决策原则 |
|---|---|---|
| **v1 必做** | 没它就不算 v1 | 删掉任何一个都让 v1 失去完整价值 |
| **v1 推迟** | 想做但 v1 不做 | 推迟后 v1 仍然完整(可能不那么完美) |
| **永远不做** | 明确砍掉 | 不是"以后再做",是"做了反而稀释核心价值" |

**反模式**:
- **v1 必做列太长**——超过 12 个就是"什么都是 v1",实际是范围不清
- **没有"永远不做"**——一个产品如果什么都做,必然什么都做不好
- **用"v1 推迟"代替"永远不做"**——推迟需要明确时间窗口("v2 看数据"),含糊的推迟就是没决定

**完成标志**:
- v1 必做 / v1 推迟 / 永远不做 三类有清晰边界
- 每个 v1 必做能用一句话说清价值

---

### 阶段 3:显性化假设(Surface Assumptions)

**目标**:找出隐藏在用户想法里的"我觉得是"——那些没明说但实施时会撞上的前提。

**进入条件**:
- 阶段 2 完成,v1 范围明确

**操作步骤**:
1. **用"如果 X 错了会怎样"问用户**。每问一遍,潜在的隐藏假设就浮现一个。
   - "如果用户其实不想用这个功能呢?" → 假设:用户有此需求
   - "如果客户环境没外网呢?" → 假设:有外网
   - "如果团队没人会 React 呢?" → 假设:团队能写 React
2. **对每个假设分类**:
   - **必假定**(必须接受为前提,否则重新规划)
   - **必验证**(实施前需要去确认或调研)——sdd-prd 阶段 2 优先质疑
   - **不成立**(问清楚后发现假设是错的,回去修正 v1 范围)
3. **把"必验证"项变成 spec 里的"未决项"清单**——这些会被 sdd-prd 阶段 2 优先质疑。

**假设清单的实际形态**:

| 假设 | 类型 | 验证方法 / 推翻后果 |
|---|---|---|
| 用户有持续稳定网络 | 必假定 | 否则需要离线模式 |
| 团队 6 人有 React 经验 | 必验证 | 没经验则改选 Vue 或培训 |
| 客户有 AWS 账号 | 必假定 | 推翻则改私有化部署 |
| 数据量 < 1 亿行 | 必验证 | 否则需重新选型 OLAP 引擎 |
| 用户能接受中文界面 | 必假定 | 推翻则需 i18n 投入 |

**反模式**:
- **假设 = "我们没想过"** —— 显性化才是目的,没想过的就是"盲区",必须挖出来
- **所有假设都"必假定"**——很多假设其实是"必验证"或"不成立"
- **跳过阶段 3** —— 不显性化的假设在实施阶段会突然爆发

**完成标志**:
- 至少 5-10 个假设被显性化
- 每条假设有类型(必假定/必验证/不成立)和后果说明

---

### 阶段 4:交付(Deliver)

**目标**:用结构化模板输出 spec 文档——让 sdd-prd 能直接消费。

**进入条件**:
- 阶段 3 完成,假设清单已有
- 用户已确认 v1 范围

**操作步骤**:
1. **用 `templates/spec-template.md` 模板填充**。模板的每个章节对应:
   - **问题陈述** — 从阶段 1 提炼
   - **目标用户** — 从阶段 1 提炼
   - **v1 功能边界** — 从阶段 2 提炼(必做/推迟/不做)
   - **核心场景** — 用用户原话描述 2-3 个关键使用流程
   - **验收标准** — 每个 v1 必做功能 1-2 条验收
   - **非功能需求** — 性能、可用性、安全、合规(如适用)
   - **未决假设** — 阶段 3 的"必验证"项
   - **选型未决** — 显式说明"技术选型不在本 spec 范围"
   - **参考材料** — 用户的笔记/纪要/原始需求

2. **每个章节严格基于用户回答**——不替用户做技术选型。即使是"显然应该用 X",也只写"未决:选型在 sdd-prd 阶段 2 决定"。

3. **子代理交叉检查**(可选但推荐)——启动子代理问"这个 spec 有什么没覆盖到?什么矛盾没暴露?"

4. **展示给用户确认**。可以问:"这份 spec 是不是你想要的?还是某些地方需要改?"

5. **写入 `docs/spec/YYYY-MM-DD-<name>.md`**,触发 lore commit。

6. **提示接力**(关键):
   > spec 已落盘,可直接调用 `sdd-prd` 技能继续,告知"基于 `docs/spec/YYYY-MM-DD-<name>.md` 提纯 PRD"。

**反模式**:
- **塞技术选型**——"前端应该用 React"——这种是 sdd-prd 阶段 2 的事
- **写 PR 风格的成功愿景**——"赋能企业、引领行业"——稀释信号噪声比
- **章节不全**——比如只写了功能没写非功能需求——下游会卡
- **不写选型未决**——下游 sdd-prd 不知道这是有意不锁还是漏写

**完成标志**:
- spec 文档已写入 `docs/spec/YYYY-MM-DD-<name>.md`
- 所有模板章节有内容
- 用户确认 spec 完整
- lore commit 已提交
- 接力提示已给出

## 5. 协作角色(建议,非强制)

| 角色 | 阶段 1 | 阶段 2 | 阶段 3 | 阶段 4 |
|---|---|---|---|---|
| 用户 | 主导:回答 5 问 | 主导:确认 v1 边界 | 主导:表态假设 | 主导:确认 spec |
| 助手(PM/BA) | 协助:捕捉用户原话 | 协助:质疑"v1 必做"清单 | 协助:挖出更深假设 | 协助:结构化整理 |
| 技术负责人 | (不参与) | 提供选型约束输入 | 提供"必验证"项的工程判断 | (不参与) |

## 6. 输入太短的情况

如果输入是 < 20 行的简短需求(而不是完整 spec):
- **阶段 1 简化**:核心 5 问必答,其他可推迟
- **阶段 2 简化**:v1 必做 3-5 个即可(避免空话)
- **阶段 3 仍要做**:简短需求背后假设更多,必须显性化
- **阶段 4 仍要做**:结构化输出是下游 sdd-prd 的输入

如果输入是几句话的纯想法(如"我想做个 X"):
- **先和用户对齐"输入模式"**——是"基于这一句话扩成完整 spec"(重新写),还是"已有 X 的细节请你提炼"(信息补全)
- 不要假装已有完整 spec 而编造内容

## 7. 跨阶段通用原则

| 原则 | 含义 |
|---|---|
| **默认对话驱动** | 默认每阶段等用户回答;用户信息充分时切批处理模式(单轮汇总)——但不替用户决定 |
| **不替用户做技术选型** | 选型留到 sdd-prd 阶段 2 质疑——这是下游技能的工作 |
| **显性化优于默认** | 所有"我以为是"的假设都必须摊出来 |
| **3 选 1 永远不做"可做"** | 分类用"必做/推迟/不做",不是"可做" |
| **结构化输出** | spec 必须有清晰章节——让 sdd-prd 阶段 1 能交叉对照 |
| **接力优先** | 完成后提示用户可直接调用 sdd-prd |
| **路径与 sdd-core 一致** | 产物写入 `docs/spec/`,命名遵循 conventions.md §2 |
| **走 lore commit** | 不走 `git commit`,走 sdd-core 标准 JSON trailer |

## 8. 使用流程总览

当用户说"我有个想法想做成产品"或"帮我把这个笔记整理成 spec":

1. **确认 sdd-core 已初始化**:`docs/spec/` 存在?若不存在则提示用户先用 `sdd-core` 场景 4 初始化
2. **走阶段 1**:问 5 个核心问题 → 等用户回答
3. **走阶段 2**:列 v1 必做 / 推迟 / 不做 → 等用户确认
4. **走阶段 3**:挖出 5-10 个假设 → 等用户表态
5. **走阶段 4**:写 spec 文档 → 用户确认 → lore commit
6. **(可选)接力**:提示用户用 sdd-prd 继续

每个阶段完成后,向用户展示产出,确认后再进入下一阶段。
如果用户说"跳过对话直接给我 spec"——切批处理模式:把各阶段问题汇总成一条清单,用户一次性回答后直接产出 spec(仍需用户确认)。
