# sdd-prd:PRD 质量审视 + 目标驱动归档

> 本技能是 `sdd-core` 的"PRD 编写辅助"——不替代 sdd-core 的文档体系,只填补"从 spec 提纯→ PRD"的质量审视缺口。
> 与 sdd-core 的协作边界见 `references/sdd-collaboration.md`。

---

## 1. 这个技能做什么

你有了一份 spec(用户故事、客户需求、技术提案),需要把它**提纯**为一份**目标驱动的、可落盘→ sdd-core 文档体系的 PRD**。本技能帮你分 4 阶段、有节奏地完成这件事——不是一次性写完再审,而是**边写边验**,每一步都产生可验证的产物。

### 1.1 核心原则

- **边写边验**——每写一段就立刻找矛盾和缺口。
- **目标驱动**——PRD 不是"持续演化的合同",而是"目标驱动的需求文档"。目标达成即归档,不再加载。
- **PRD 唯一交付**——不写 Phase(留给其他技能)、不写 Architecture 总览(由 sdd-core 管)、不写 Reference 索引(由 sdd-core 管)。

### 1.2 与 sdd-core 的边界

| 维度                                     | sdd-core 管                            | sdd-prd 管                                                                             |
| ---------------------------------------- | -------------------------------------- | -------------------------------------------------------------------------------------- |
| `docs/prd/YYYY-MM-DD-<name>.md` 编写     | **不主动写**                           | **本技能唯一交付**                                                                     |
| `docs/prd/YYYY-MM-DD-<name>.md` 命名     | **遵循 conventions.md §2.1**           | 遵循                                                                                   |
| `docs/prd/YYYY-MM-DD-<name>.md` 必填章节 | **遵循 conventions.md §3.1 强制 5 章** | 遵循 + 顶部追加 §0 目标声明/验收开关                                                   |
| `docs/prd/archive/` 归档目录             | **不主动建**                           | **本技能独家**                                                                         |
| `docs/architecture/decisions.md` 写 ADR  | **不主动写**                           | **阶段 2 产物合并入**                                                                  |
| `docs/phase/`                            | **sdd-core 管**                        | **不涉及**                                                                             |
| `docs/index.md` / `docs/CONTRIBUTING.md` | **sdd-core 管**                        | **阶段 4 同步必要索引**                                                                |
| lore commit 提交                         | **协议本身**                           | **本技能走此提交**                                                                     |
| 索引同步                                 | **规则与模板来源**                     | **按 `rule://docs-update-guard` 在提交前同步 `docs/index.md` 和 `docs/prd/README.md`** |

### 1.3 输出物

- **PRD 主文档**:`docs/prd/YYYY-MM-DD-<name>.md`(唯一交付)
- **ADR 集**:`docs/architecture/decisions.md`(阶段 2 合并入;若已存在则追加)

工作产物(阶段 4 后清理):

- `docs/prd/.working/YYYY-MM-DD-<name>/problem-list.md`(阶段 1)
- `docs/prd/.working/YYYY-MM-DD-<name>/adr-set.md`(阶段 2,合并后删除)
- `docs/prd/.working/YYYY-MM-DD-<name>/constraint-set.md`(阶段 3,内容并入 PRD 后删除)

## 2. 适用与不适用

**适用:**

- 中型项目(5-15 人)、需求→实施的过渡阶段
- spec/客户需求/用户故事已存在,需要系统化提纯为可执行 PRD
- 项目已用 sdd-core 初始化过（`docs/prd/` 目录存在）
- 需要"目标驱动 + 归档"防止 PRD 随时间被 agent 污染

**不适用:**

- **多文档初始化**——首次创建 `docs/` 体系,改用 `sdd-core`(场景 4)
- **纯架构专题**——只写架构决策,无 PRD 需求,改用 `sdd-core` 直接写 `docs/architecture/`
- **多角色需求推演**——评估一个功能对多角色的影响,改用 `multi-role-requirement-analysis`
- **纯原型/小工具**(< 3 人、< 1 周)——简化为阶段 1 + 阶段 4 即可
- **强依赖外部 spec 且不允许改动**——阶段 2 力度降低,但至少做一遍快速审查

## 3. 与 sdd-core 的协作模式(强制)

### 3.1 路径与命名

- PRD 写入 `docs/prd/YYYY-MM-DD-<name>.md`(遵循 conventions.md §2.1)
- 命名 `<name>` 用小写字母和连字符,如 `user-authentication`、`payment-integration`
- **禁止**:`PRD-001.md`(无日期)、`2026-6-23.md`(日期格式错)、`User_Auth.md`(下划线大写)

### 3.2 必填章节(基于 conventions.md §3.1)

PRD 必含 sdd-core 强制 5 章节(conventions.md §3.1) + sdd-prd 独家 2 章节:

| #   | 章节                   | 来源                     | 强制级别   |
| --- | ---------------------- | ------------------------ | ---------- |
| §0  | 目标声明               | sdd-prd 独家             | 归档触发器 |
| §0  | 目标验收开关           | sdd-prd 独家             | 归档触发器 |
| §1  | 背景与目标             | conventions.md §3.1 强制 | 必填       |
| §2  | 用户与场景             | conventions.md §3.1 强制 | 必填       |
| §3  | 功能需求               | conventions.md §3.1 强制 | 必填       |
| §4  | 非功能需求             | conventions.md §3.1 强制 | 必填       |
| §5  | 验收标准               | conventions.md §3.1 强制 | 必填       |
| §6+ | 数据/界面/集成/风险 等 | conventions.md §3.2 可选 | 按需       |

### 3.3 Phase 占位(明确边界)

PRD 顶部按 conventions.md §3.3 必须有"对应阶段"链接。但本技能**不写 Phase**——由其他技能完成。

模板:

```markdown
# {产品/功能名} PRD

> 修改记录:执行 `lore log docs/prd/<filename>.md`
> 对应阶段: [TBD - 由其他技能补全](../phase/YYYY-MM-DD-<phase-name>.md)
```

**约束**:

- 占位行**必须存在**(否则违反 conventions.md §3.3)
- 文件路径**必须**指向 `../phase/...`,即使文件暂不存在
- 阶段 4 完成时若 Phase 仍为 TBD,允许提交——其他技能会补全

### 3.4 模板来源（代码内联生成）

模板由 `sdd propose` CLI 命令（`src/cli/lib/template-engine.ts`）在代码内联生成，**不依赖项目内 `_template.md` 文件**。`references/templates.md` 和 `templates/prd-outline-template.md` 仅作为技能自身参考。

**实际操作**：

- 用 `sdd propose --title "<名称>"` CLI 生成 PRD 基础框架
- 在 `> 对应阶段:` 行后**追加** `## 0. 目标声明` 和 `## 0. 目标验收开关` 两章节
- 其余章节保持 CLI 生成的框架原样

### 3.5 提交协议

**强制走 lore commit**。完整 JSON trailer schema 见 `rule://lore-protocol` 和 sdd-core §核心工作流程→提交变更。本技能推荐的 trailers:

| Trailer      | 推荐内容                                                                |
| ------------ | ----------------------------------------------------------------------- |
| `intent`     | "提纯 {spec-name} 为 PRD v1.0"                                          |
| `Constraint` | `PRD 必含 conventions.md §3.1 的 5 章节 + sdd-prd §0 目标声明/验收开关` |
| `Rejected`   | `旧的 prd-outputs/ 路径 \| 与 conventions.md §1.1 冲突`                 |
| `Directive`  | `sdd-prd 不写 Phase,Phase 留 TBD`                                       |
| `Tested`     | `子代理 5 项审查通过`                                                   |
| `Not-tested` | `PRD ↔ Phase 双向引用(Phase 留 TBD)`                                    |

## 4. 4 阶段工作流

```mermaid
flowchart LR
    S1["阶段 1:自审"] --"问题清单·内部矛盾"--> S2["阶段 2:深审"]
    S2 --"ADR 集·选型质疑"--> S3["阶段 3:增量设计"]
    S3 --"约束集·补全约束"--> S4["阶段 4:精简提交"]
    S4 --"精炼 PRD·信号密度"--> Done["PRD 落盘 + lore commit"]
```

**进入下一阶段的前提:当前阶段产物已产生并被用户确认。**
不能跨阶段跳跃——跳过阶段 1 直接做阶段 2,可能会"推翻选型后才发现内部矛盾更多"。

---

### 阶段 1:自审

**目标**:在 spec 内部找→自相矛盾、缺定义、悬空引用。不找新问题,只找**已写出来但互相打架**的问题。

**进入条件**:

- spec(或等效输入)已完整存在
- spec 长度足够做交叉比对(至少 20-30 个独立陈述点;少于则按"输入太短"处理)

**操作步骤**:

1. 通读全文,建立心智地图——记住每个章节承诺了什么、在哪个位置
2. 制造对偶压力——把不同章节交叉对比。典型对照:
   - 功能需求章 ↔ API 契约章:功能说"支持 X",API 没体现 X——矛盾
   - 部署章 ↔ 技术选型章:部署说"私有化",选型用了"仅云服务"——矛盾
   - 数据模型章 ↔ 业务规则章:数据模型允许 NULL,业务规则说 P0 必填——缺定义
3. 找 3 类信号:
   - **矛盾**("v1 做 X" vs "v1 不做 X")
   - **缺定义**("P0 必填"但字段类型、约束没说)
   - **悬空引用**(指向不存在的章节、表、接口)

**产物**:`docs/prd/.working/YYYY-MM-DD-<name>/problem-list.md`

按优先级分类:

- **必修**:不修就无法进入阶段 2(如矛盾导致两个方向都走不通)
- **应修**:影响质量但不阻塞流程
- **可选**:锦上添花

模板:`templates/problem-list.md`(沿用)

**反模式**:

- 只看一遍就下结论——交叉比对必须反复交叉
- 用"看起来对"代替实际对照——必须逐条在两个章节间找→对应句子
- 把"缺定义"归→"应修"或"可选"——很多时候缺定义就是矛盾
- 主上下文自己审完就觉得没问题——如果问题清单为空,**大概率审视不够深**,启动子代理再审一遍

**完成标志**:

- 问题清单已生成,每条都有具体位置(章节/行引用)
- 每条已被分类为必修/应修/可选
- 用户确认清单完成

---

### 阶段 2:深度审视

**目标**:验证核心选型是否真的最优,不要被 spec 推着走。**深度审视 = 怀疑一切已确定的,尤其是"看起来理所当然的"**。

**进入条件**:

- 阶段 1 完成,矛盾已修或已标记可后续处理
- spec 中包含技术选型、架构方向、或框架选择

**操作步骤**:

1. **质疑每个继承来的选型**。spec 选了某框架/某库?问"为什么不能直接用更轻的东西"
2. **建立决策原则**。先定原则,再用原则过滤选型:
   - "任何组件可独立替换"——换掉这个组件,其他组件需要改吗?
   - "私有化部署可行"——没外网环境能部署和升级吗?
   - "团队能力覆盖"——当前团队的技术栈能不能维护?
3. **反驳自己的选型**。每次决定前先想最坏情况:"在客户实际环境跑得动吗?规模翻 10 倍还撑得住吗?"
4. **评估规模与方案是否匹配**。12 个静态页用 SSG 没问题;200+ 产品持续增长时 SSG 重建要几十秒

**产物**:

- 阶段中:`docs/prd/.working/YYYY-MM-DD-<name>/adr-set.md`
- 阶段 4 后:**合并入** `docs/architecture/decisions.md`(若文件不存在则创建;若已存在则在末尾追加,标注日期)

模板:`templates/adr-template.md`(微调:改"ARCHITECTURE.md"为"`docs/architecture/decisions.md`")

**反模式**:

- "spec 写了就照做"——spec 也是人写的,可能过时或不合理
- "先选型再找理由"——先确定原则,用原则过滤选型
- 只看技术合理性,忽略运维/部署/团队能力

**完成标志**:

- 每个继承来的关键选型都被质疑过(至少 3 个)
- 每个推翻/采纳都有 ADR 记录
- 用户确认 ADR 集完成

---

### 阶段 3:增量设计

**目标**:补全约束、覆盖边界、明确实施时的"是否做对"的判断标准。**方案指导 ≠ 具体设计**——PRD 锁"用什么机制",不锁"具体字段名"。

**进入条件**:

- 阶段 2 完成,核心选型已确定
- spec 的内核(功能边界、数据流)已清晰

**操作步骤**:

1. **列出实施时会卡住的 8-12 个点**。典型:
   - slug 怎么生成?(唯一性、迁移、SEO)
   - 媒体文件怎么处理?(存储、resize、限制)
   - 并发编辑冲突怎么办?(锁策略、合并策略)
   - Session 怎么过期?(JWT、refresh token、cookie)
   - 错误码怎么统一?(HTTP 状态码 vs 业务错误码)
   - i18n 字段怎么 fallback?
   - design token 怎么存储?
   - 跨平台怎么构建?
   - native binding 怎么部署?
   - 搜索怎么实现?
   - 权限模型怎么定义?(RBAC/ABAC/混合)
   - 审计日志怎么设计?
   - **SLO 阈值**——关键路径 P99 延迟、可用性 SLO、错误率上限
   - **数据分级与脱敏**——敏感字段识别、加密静态/传输、脱敏规则
   - **审计与合规边界**——审计日志保留期、合规法规(GDPR/等保/PCI)
2. **每个点写一段方案约束**——不写具体实现,只说:
   - **必须**做什么(硬约束)
   - **禁止**做什么(不能碰的线)
   - **优先**做什么(优先于其他选择)
   - 每个约束附带**理由**(为什么是约束)
3. **按 P0/P1/P2 优先级排**。i18n 和权限是 P0——先锁方案;审计日志是 P2——可推迟
4. **约束和功能分离**。约束是"不可妥协"(归入 PRD §4 非功能需求),功能是"可能调整"(归入 PRD §3 功能需求)
5. **业务规则显性化**。状态机、权限矩阵、计算公式、配额表——四选一表达,禁止散文描述

**产物**:

- 阶段中:`docs/prd/.working/YYYY-MM-DD-<name>/constraint-set.md`
- 阶段 4 后:**内容并入** PRD §4 非功能需求(若 P0 约束)或附录(若 P1/P2)

模板:`templates/constraint-set-template.md`(微调:改"写入 ARCHITECTURE.md"为"归入 PRD §4/附录")

**反模式**:

- 约束写太细——锁→具体字段名就失去了实施阶段的判断空间
- 约束和功能混杂——读者不知道哪段"必须遵守"哪段"可以商量"
- 约束没标 P0/P1/P2——团队不知道哪些要立刻做、哪些可推迟

**完成标志**:

- 至少覆盖 8 个实施卡点,每个都有约束 + 理由
- 约束已按 P0/P1/P2 优先级排序
- 用户确认约束集完成

---

### 阶段 4:精简与提交

**目标**:交付的 PRD **精炼、内部一致、可执行**。**信号噪声比 > 信息量**——删得越多,重要信息越突出。

**进入条件**:

- 阶段 3 完成,约束集已定
- 完整的 PRD + ADR 已生成
- 用户已确认**这份 PRD 的目标陈述**

**操作步骤**:

1. **删具体代码**。完整 schema 移至 `docs/architecture/api-reference.md` reference,完整 config 移至 `docs/architecture/config-reference.md`,完整部署脚本移至 `docs/architecture/deployment-guide.md`。PRD 只留"指向哪里"
2. **删决策过程**。"为什么选 A 不选 B"是 ADR 的事(在 `docs/architecture/decisions.md`),不是 PRD 的事。决策定了之后,PRD 只写"用什么"
3. **删重复**。同一条信息在多处出现时,保留最权威的那处,其他用引用
4. **删过时引用**。grep 整个 workspace + 跨文档检查(PRD ↔ `decisions.md`)。每改一处,**所有引用它的章节都要同步**
5. **在 PRD 顶部加"目标声明"和"目标验收开关"**(若模板未含):
   - 目标声明:这份 PRD 是为达成什么目标
   - 目标验收:业务验收 + 技术验收的明确开关
   - 这两个章节是归档的触发器——agent 加载时必读
6. **子代理做最终审查**。**主上下文写文档,子代理做最后一遍全面对照**——主上下文有认知偏差(自己写的觉得都对),子代理没有。子代理应执行 5 项审查:
   - 检查内部一致性(交叉验证)
   - 检查过时引用(grep 全 workspace)
   - 检查阶段 1-3 的产物是否都已整合
   - 评估信号噪声比(随机挑 3 段,删掉能否保持文档完整)
   - **sdd-core 一致性**——对照 conventions.md §3.1 的 5 必填章节,逐条打勾;命名是否符合 §2.1;顶部 frontmatter 是否含"对应 Phase"反向链接(占位可接受)
7. **同步索引**。按 `rule://docs-update-guard` 和 sdd-core 规则更新 `docs/index.md` 和 `docs/prd/README.md`(若存在),并把验证结果写入 lore `Tested`
8. **清理工作产物**。删除 `docs/prd/.working/YYYY-MM-DD-<name>/` 整个目录
9. **单次提交**。所有变更在一个 `lore commit` 里——"PRD 全面定型"是一个完整事件,不应拆成多个小提交

**产物**:

- **PRD 主文档**:`docs/prd/YYYY-MM-DD-<name>.md`(精炼版,含"目标声明"和"目标验收开关"必填 2 章节)
- **ADR 集**:`docs/architecture/decisions.md`(阶段 2 合并)

模板:

- 运行时来源:`sdd propose` CLI（`template-engine.ts` 内联生成）
- 本技能参考:`templates/prd-outline-template.md`(已含 §0 目标声明/验收开关,不参与运行时优先级)

**完成标志**:

- 子代理最终审查完成,无遗留问题
- workspace 中 grep 确认零悬空引用
- 用户确认基线完整
- **PRD 顶部有"目标声明"和"目标验收开关"**(这是归档前提)
- lore commit 单次提交完成

### 阶段 4 完成 ≠ 归档

**阶段 4 完成是"PRD 可被实施"的标志,不是"PRD 归档"的标志。**

归档的 4 个触发条件(任一)见 `references/archival-mechanism.md`:

1. 业务验收开关全部勾选(可按目标维度分项勾选)
2. 技术验收开关全部勾选
3. 用户明确说"目标达成,PRD 归档"
4. 目标替换——新 PRD 替代旧 PRD 的目标(即使开关未全勾也归档旧 PRD)

## 5. 归档机制

> 详见 `references/archival-mechanism.md`。这里是摘要。

### 5.1 为什么需要归档

PRD 不是"写一次就完"——它会被 agent 反复加载。每次加载都会让 agent 在以下场景中"补全"PRD。这些"补全"累积 5-10 次后,PRD 表面看起来"完整",实际已偏离原始意图。

**归档的本质是文件移动,不是标志位**:

- 活跃 PRD 位于 `docs/prd/`(agent 默认扫描区)
- 归档 PRD 位于 `docs/prd/archive/`(agent 默认不扫描)

### 5.2 归档触发器

| 触发条件                              | 状态变更                          | 物理动作                                       |
| ------------------------------------- | --------------------------------- | ---------------------------------------------- |
| 目标验收开关全勾选 + 用户确认目标达成 | `> 状态：已归档`                  | 移动到 `docs/prd/archive/YYYY-MM-DD-<name>.md` |
| 新 PRD 替代旧目标                     | `> 状态：已归档 \| 归档原因：已替换 by <新 PRD 路径>` | 不移动，原地标注                               |
| 用户明确"不做了"                      | `> 状态：已归档 \| 归档原因：已中止（原因：...）`     | 不移动，原地标注                               |

**对齐 conventions.md §3.4**：状态字段是唯一标记，不使用 `<!-- archived -->` 等 HTML 注释第二套标记。归档 = 状态变更 + 文件移动，二者同步。

### 5.3 归档后行为

- **索引同步**:触发 sdd-core 更新 `docs/index.md`,把活跃区移入"归档"分区
- **agent 加载检查**:加载→归档区 PRD 则视为历史记录,提示用户,不允许修改
- **Phase 占位**:归档时若 Phase 仍为 TBD,保留占位

## 7. 输入太短的情况

如果输入是 < 20 行的小段需求(而不是完整 spec):

- **阶段 1 降级**:重点找"显式矛盾"和"明显缺定义",无需"对偶压力"
- **阶段 2 仍要做**:哪怕是简短用户故事,也可能有继承假设
- **阶段 3 仍要做**:用户故事背后一定有实施卡点
- **阶段 4 仍要做**:精简和子代理审查同样适用

如果输入是几句话的纯想法(如"我想做个 X"):

- **先和用户对齐"输入模式"**——是"基于这一句话扩成完整 PRD"(重新写),还是"已有 X 的细节请你提炼"(信息补全)
- 不要假装已有完整 spec 而编造内容

## 8. 跨阶段通用原则

| 原则                         | 含义                                                           |
| ---------------------------- | -------------------------------------------------------------- |
| **任何组件可独立替换**       | 选型时用"换掉这个组件,其他改吗?"来测试解耦程度                 |
| **方案指导 vs 具体设计**     | PRD 锁机制,不锁具体值——具体值留给实施阶段的 schema/config 文件 |
| **代码与方案分离**           | 代码移→ `docs/architecture/`,PRD 只留"指向哪里"                |
| **子代理做最终审查**         | 主上下文负责写,子代理负责审——避免"我写的都对"                  |
| **单次 commit 表达完整事件** | PRD 全面定型是一次性事件,不拆成功能级小提交                    |
| **每阶段产物明确**           | 自审问题清单;深审 ADR;增量约束集;精简最终 PRD——不跨阶段跳跃    |
| **目标驱动而非持续演化**     | 每份 PRD 对应一个明确目标;目标达成后归档→ `docs/prd/archive/`  |
| **PRD 不写 Phase**           | Phase 由其他技能完成,PRD 顶部留 TBD 占位                       |

## 9. 使用流程总览

当用户说"我有个 spec 要做成 PRD"或其他触发短语时:

1. **确认 sdd-core 已初始化**:`docs/prd/` 存在?若不存在则调用 `sdd-core` 场景 4 初始化
2. **走阶段 1**:自审 spec,产出 `problem-list.md`,用户确认
3. **走阶段 2**:质疑选型,产出 `adr-set.md`,用户确认
4. **走阶段 3**:补全约束,产出 `constraint-set.md`,用户确认
5. **走阶段 4**:精简 + 顶部加"目标声明"+"目标验收开关" + 子代理 5 项审查 + 清理工作产物 + lore commit,交付 PRD

每个阶段完成后,向用户展示产物,确认后再进入下一阶段。
如果用户说"跳过某阶段",解释该阶段的价值,但若用户坚持则尊重决定。

### 阶段 4 之后:归档触发

PRD 阶段 4 完成后,agent 主动询问用户:

- "目标验收开关是否全部勾选?"
- "目标是否已达成?"
- 如果达成则引导归档流程(见 `references/archival-mechanism.md`)
