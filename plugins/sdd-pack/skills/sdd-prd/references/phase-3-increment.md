# 阶段 3 深度参考：增量设计（Increment）

> 加载时机：用户不确定哪些约束该写、卡点怎么识别、优先级怎么排列。

## 增量设计的本质

阶段 3 做的事情可以用一句话概括：

> **把"实施时会卡住"的隐性决策提前显性化。**

实施团队拿→ spec，会发现大量细节 spec 没说。不是 spec 写得差——而是**spec 不该写那么细**。spec 在意的是"做什么"，阶段 3 在意的是"怎么做才不跑偏"。

阶段 3 不写代码，不写实现细节，只做一件事：**定约束**——也就是"必须做什么、禁止做什么、优先做什么"。

---

## 识别卡点的系统方法

### 方法 1：走一遍"实施第一天"

假设你是开发第一天进项目的工程师：

- 打开 spec，读→"需要一个用户系统"——好，建表。字段类型？string? int? uuid?
- 读→"支持多语言"——好，怎么存？一张表所有语言？N 张表每语言？
- 读→"文件上传"——好，存哪？本地文件系统？S3？MinIO？文件大小限制？
- 读→"搜索"——好，怎么搜？LIKE？全文索引？搜索引擎？

**每个"这里该怎么做？" = 一个卡点 = 需要写约束。**

### 方法 2：从历史中学习

回顾你的团队上次做项目时：

- 哪些决策拖了很久没定？（→ 持续性的卡点）
- 哪些决策定了又推翻重来？（→ 缺原则的决策）
- 哪些实施后发现"原来是这个意思"？（→ 约束不清晰的后果）

这些几乎一定会在新项目中重现——现在就写约束。

### 方法 3：分类穷举法

把"所有可能的卡点"分类，逐一检视：

| 领域     | 典型卡点                                                                 |
| -------- | ------------------------------------------------------------------------ |
| **数据** | 字段类型/约束、ID 策略（自增/uuid/snowflake）、软删除/硬删除、时间戳精度 |
| **API**  | 版本策略、错误码体系、分页方式、认证方式、限流方案                       |
| **前端** | 路由策略、状态管理、i18n、design token 管理、构建方案                    |
| **存储** | 文件存储路径、图片处理策略、CDN 方案、备份策略                           |
| **部署** | 容器化/裸机、CI/CD 方案、自升级机制、监控告警                            |
| **安全** | 认证（JWT/Session）、授权（RBAC/ABAC）、CSRF/CORS、加密存储              |
| **运维** | 日志方案、审计策略、错误追踪、性能监控                                   |

不是每个领域都要写——但每类至少扫一眼："这个领域在我们项目里有卡点吗？"

### 示例：从 spec 片段推导卡点

**spec 说**："用户可以上传文章封面图。"

**推导卡点：**

1. 存哪？（本地 / S3 / MinIO） → 约束："必须用 S3-compatible storage"
2. 大小限制？（1MB? 10MB? 没有限制？） → 约束："必须限制为 5MB"
3. 格式？（jpg? png? webp? 都接受？） → 约束："必须转换为 WebP，保留原图 30 天"
4. 尺寸？（原图直接返回给前端？） → 约束："必须生成 thumbnail(200x200) + medium(800x800)"
5. 命名？（用户原始文件名？hash？） → 约束："必须用 content-hash 命名，防止冲突"

**一条 spec 语句 → 产生了 5 条约束。** 这就是增量设计的价值。

## 约束的优先级判定

### P0：不做就阻塞

**判断标准**：如果这个约束不定，实施团队就**无法开始**对应领域的工作。

```
信号：
- 依赖链：P0 约束定了，下游设计才能开始
- 一旦实施就不可逆（如数据库 schema、API 版本策略）
- 影响所有功能模块（如权限模型）
```

### P1：第一个迭代周期内做

**判断标准**：可以暂时用合理默认值或简化方案，但不能拖过第一个迭代。

```
信号：
- 临时方案不影响功能正确性，但影响质量/可维护性
- 可以推迟 1-2 周，但再晚就会累积技术债
```

### P2：可推迟→非 MVP

**判断标准**：MVP 不需要它也能完整交付价值。

```
信号：
- 非 MVP 功能依赖这个约束
- 约束只在规模化后才有意义（如全局搜索、审计日志）
- 约束可以通过"先不做"来规避
```

## 约束 ≠ 实现

### 核心分界

| 约束写什么                         | 实施阶段写什么                                                   |
| ---------------------------------- | ---------------------------------------------------------------- |
| "i18n 用 JSON 字段存，fallback en" | "字段名叫 `title_i18n`，结构是 `{ zh: string, en: string }`"     |
| "Slug 唯一约束在 DB 层"            | "migration 里加 `unique_index :articles, :slug`"                 |
| "错误码用 DOMAIN_CODE 格式"        | "错误码枚举：AUTH_001, AUTH_002, ..."                            |
| "用乐观锁处理并发编辑"             | "articles 表加 `lock_version` 字段，update 时 WHERE version = ?" |

### 为什么不能写太细

- 写太细 → 实施团队没有判断空间 → PRD 变成"能不能照抄"而不是"能不能理解"
- 写太细 → 边界条件多了就必须全关联 → 改一处就崩全盘
- 写太细 → 读者分不清"哪段是可以商量的" → 约束和功能失去分层价值

## 常见约束的类型导引

### 数据层约束

- ID 策略、软删除/硬删除、时间戳格式（UTC/Epoch/ISO 8601）
- JSON 字段的使用范围（"只用于配置数据，不用于关系数据"）
- Migration 方向（只增不删？允许回滚？）

### API 层约束

- 版本策略、分页方式、错误格式、认证 token 方式
- 限流策略（Per-IP? Per-User? Token-bucket?）
- API 文档生成方式（代码注释驱动 vs 手动维护 → 约束"必须用代码注释生成规范"）

### 前端层约束

- 状态管理位置（Client State vs Server State 的边界）
- i18n 键名格式、Design Token 源
- 测试策略（E2E vs 集成 vs 单元 的分工）

### 部署层约束

- 容器化方案、环境变量管理、密钥管理
- CI/CD 触发条件、制品存储、回滚策略
