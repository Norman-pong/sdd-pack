# 阶段 1 深度参考：自审（Self-Review）

> 加载时机：用户说→"自审怎么做"、"怎么看矛盾"、"问题清单怎么填"、或自审遇→困难时。
> **sdd-prd 适配说明**:本阶段产物 `problem-list.md` 写入 `docs/prd/.working/YYYY-MM-DD-<name>/problem-list.md`。阶段 4 后清理该目录。
> 本阶段不涉及 Phase——Phase 由其他技能完成,本技能不约束。

---

## 为什么自审比其他阶段都重要

自审是整个流程的**根基**。如果内部矛盾没找出来，后续所有阶段都在"自相矛盾的 spec"基础上构建——阶段 2 的选型质疑会失去目标，阶段 3 的约束补全会方向模糊，阶段 4 的精简会变成"精简一份有 bug 的文档"。

核心洞察：**不是找新东西，是找已写出来但互相打架的东西。** 一段说 v1 做 A，另一段说 v1 不做 A——这就是矛盾。不需要"灵感"，只需要**逐条交叉比对**。

---

## 交叉对比的技术

### 方法：对偶压力法

挑两段互相依赖的章节，逐条对照。如果一段承诺了但另一段没兑现 → 问题。

### 标准对比对

| 章节 A   | 章节 B   | 找什么                                                            |
| -------- | -------- | ----------------------------------------------------------------- |
| 功能需求 | API 契约 | 功能说的操作 API 都有端点吗？API 有的端点功能里都提了吗？         |
| 数据模型 | 业务规则 | 业务规则引用的字段都在数据模型里有定义吗？类型/约束一致吗？       |
| 部署说明 | 技术选型 | 部署说的环境选型支持吗？（私有化 vs 云服务、容器 vs 裸机）        |
| 角色权限 | 功能需求 | 每个角色能操作的功能都明确了吗？管理员权限和普通用户有重叠吗？    |
| 版本规划 | 功能需求 | v1 做 vs v1 不做的边界清晰吗？有没有"v1 功能"用→"v2 才有的组件"？ |
| 性能指标 | 技术选型 | 选型能撑住指标吗？（QPS、延迟、数据量）                           |
| 安全要求 | 数据模型 | 加密字段、脱敏规则、审计日志——哪层负责？                          |

### 实操步骤

1. **先把这两个章节并排打开**（对比阅读，不要来回滚动）
2. **建一个"承诺清单"**：章节 A 承诺了什么操作/能力
3. **逐一在章节 B 里找对应**
4. **找不→ → 矛盾或缺定义**；找→但不一致 → 矛盾

### 示例

**spec 片段 A（功能需求）：**

> v1 支持用户批量导入（CSV、Excel）、单个创建、按角色搜索。

**spec 片段 B（API 契约）：**

> POST /api/v1/users — 创建单个用户
> GET /api/v1/users — 搜索用户（支持 role 过滤）

**交叉对比结果：**

- 功能说"批量导入",API 只有单条 POST,标为**矛盾**(必修)
- 功能说"CSV、Excel",API 没提文件上传格式,标为**缺定义**(应修)
- 角色搜索,API 有 `role` 过滤,标为**一致**

## 3 类问题的判断标准

### 矛盾（必修级）

**判断**：两个地方说的不一致，会导致实施时做不下去。

```
特征："v1 做 X" vs "v1 不做 X"
     "用 A 方案" vs "用 B 方案"（两处段说的不同）
     "管理员可以删除" vs "删除权限只给超级管理员"（角色边界打架）
```

**为什么矛盾几乎总该是必修？**
矛盾不修，后面做的决策都在自相矛盾的地基上。即使看起来是"小矛盾"（如措辞不同），很可能反映更深的认知不一致。

### 缺定义（应修级，但可能是必修）

**判断**：重要概念提了名但没定义，实施时**无法确定怎么做**。

```
特征："P0 必填"但字段类型没说
     "支持多语言"但没写存储方案
     "高性能"但没写 QPS/延迟目标
```

**升为必修的条件**：

- 缺定义阻塞了阶段 2/3 的判断（不知道字段类型，连数据库选型都定不了）
- 缺定义让两个实施方向都"通"但方向相反（"多语言支持"→→底按语言拆表还是 JSON 字段？）
- 缺定义有安全/合规影响（"加密传输"但没说 TLS 版本）

### 悬空引用（应修）

**判断**：引用了不存在的章节、表、API、接口。

在 grep 阶段（阶段 4）会全部暴露——阶段 1 可以先标记，阶段 4 再统一处理。

## 自审的深度自检

### "我审得够深吗？"的 5 个信号

1. 至少找→ **5 个问题**（如果少于 5 个，大概率是审得不够——真实文档很少有完美一致的）
2. 至少有 **1 个矛盾**（纯缺定义和模糊 → 可能只是表面审，没做交叉对比）
3. 每个问题都引用**两个具体位置**（交叉对比的标志）
4. 必修/应修/可选分类合理——至少三分之一应该是必修或应修
5. 自审结束后，你对 spec 的**理解比自审前更深**——你找→了 spec 作者可能都没意识→的矛盾

### 如果问题清单是空的

两种可能：

- spec 真的完美（极端罕见，只在极简 spec 中出现）
- 审得不够深（默认假设）

处理方式是：**用子代理再审一遍**。子代理没有主上下文的认知偏差（你刚才读了 spec，已经有了"它是对的"的印象），更能抓→矛盾。

在阶段 1 就用子代理审的技术和阶段 4 一样——子代理独立读取 spec，生成它的问题清单，然后和你的问题清单合并。

## 常见漏审的模式

这些是容易漏的地方——因为它们不在"核心内容"里，而在边角：

- **假设章节**："我们假设用户已经有了 X"——如果后续章节没验证这个假设，就是矛盾
- **migration/迁移说明**：如果说了"从旧系统迁移"但数据转换方案没写 → 缺定义
- **glossary/术语表**：如果两个术语定义打架（如"项目"和"workspace"本质是同一个概念） → 矛盾
- **非功能需求**：安全/性能/可维护性——往往写在单独段落，和功能章节的隐含前提不一致
