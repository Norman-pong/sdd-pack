# 阶段 1 深度参考:自审(Self-Review,基于 PRD 边界)

> 加载时机:用户卡在 PRD 边界识别、不确定"哪些矛盾会影响任务拆解"、需要填问题清单时。
> 本阶段是 sdd-phase 的"根基"——PRD 边界不清,任务拆解必错。

---

## 1. 为什么自审是 sdd-phase 的根基

阶段 2 的"任务分解"完全建立在 PRD 边界之上。如果 PRD 内部矛盾没找出来:

- 任务拆解时把矛盾两边都拆了——下游工程师不知道听哪边
- 任务依赖识别错——并行任务被串行化,工期翻倍
- 验收标准缺失——任务"完成"无客观判定,反复返工

核心洞察:**不是找新东西,是找 PRD 已写出来但互相打架的东西**。一段说 v1 做 A,另一段说 v1 不做 A——这就是矛盾,会直接影响任务拆解。

---

## 2. PRD 边界审查的 4 个维度

### 2.1 范围维度(影响任务清单)

**信号**:
- "v1 做 X" vs "v1 不做 X" 同时存在
- 功能清单 P0/P1/P2 分级不明确
- "可选功能"和"必须功能"混在一起

**示例**:
```markdown
PRD §3.1 功能清单:
| 功能模块 | 功能点 | 优先级 | 说明 |
| 用户系统 | 邮箱注册 | P0 | 主要方式 |
| 用户系统 | 手机注册 | P1 | 辅助方式 |

PRD §1.3 成功指标:
- "新用户注册转化率 ≥ 30%"

# 矛盾:§3.1 说 P0 是邮箱注册,但 §1.3 没说是否包含手机注册
# 影响:任务拆解时不知道是否要给手机注册排期
```

### 2.2 依赖维度(影响任务依赖链)

**信号**:
- PRD 提到外部系统但未说集成方式
- 跨章节引用的接口/字段定义不一致
- "假设已有 X" 但没说 X 由谁提供

**示例**:
```markdown
PRD §3.2.1 用户注册:
"调用邮件服务发送验证邮件"

PRD §7 集成需求:
(未列邮件服务)

# 缺定义:邮件服务由谁提供?SMTP 第三方?自建?
# 影响:任务拆解时无法规划"集成邮件服务"任务
```

### 2.3 验收维度(影响任务完成判定)

**信号**:
- "性能良好"、"界面友好"等不可证伪的形容词
- NFR 阈值缺失(无 P99 数字、无 SLO 数字)
- 验收标准散落在多章节,口径不一致

**示例**:
```markdown
PRD §4.1 性能要求:
"系统响应快速"

PRD §5 验收标准:
"页面加载时间不超过 3 秒"

# 缺定义:§4.1 无具体阈值;§8 的"3 秒"是哪个页?首屏还是完全加载?
# 影响:任务验收时无客观判定
```

### 2.4 引用维度(影响 Phase 文档引用完整性)

**信号**:
- PRD 引用了不存在的章节
- Phase 任务会引用 PRD 内容,但 PRD 内部就有矛盾引用

**示例**:
```markdown
PRD §3.2.1:
"详见 §3.2.5 异常处理"

# 实际无 §3.2.5 章节
# 影响:任务"实现异常处理"无法定位需求来源
```

## 3. 3 类问题的判断标准

### 3.1 矛盾(必修级)

**判断**:两处说的不一致,会导致任务拆解方向相反。

```
特征:
- "v1 做 X" vs "v1 不做 X"
- 任务粒度不同(同一功能被标 P0 和 P2)
- 验收口径冲突(性能"快速"vs"3 秒")
```

**为什么矛盾几乎总该是必修**:
矛盾不修,任务拆解在自相矛盾的地基上做。任务拆错,下游返工成本是阶段 1 的 5-10 倍。

### 3.2 缺定义(应修级,但可能是必修)

**判断**:重要概念提了名但没定义,任务无法开始。

```
特征:
- "邮件服务"但没说 SMTP 还是 SES
- "P0 必填"但字段类型没说
- "高可用"但没说几个 9
```

**升为必修的条件**:
- 缺定义阻塞了任务粒度切分(任务边界无依据)
- 缺定义让两个任务方向都"通"但方向相反
- 缺定义有安全/合规影响("加密传输"但没说 TLS 版本)

### 3.3 悬空引用(应修)

**判断**:Phase 写完后会引用不存在的章节/接口。

```
特征:
- "详见 §X"但 §X 不存在
- "调用 Y 接口"但 Y 无定义
- 引用了外部文档但项目内无该文档
```

## 4. 自审的深度自检

### 4.1 "我审得够深吗?"的 5 个信号

1. 至少找到 **5 个问题**(如果少于 5 个,大概率是审得不够)
2. 至少有 **1 个矛盾**(纯缺定义 → 可能只是表面审)
3. 每个问题都引用**两个具体位置**(PRD 章节 + 行号)
4. 必修/应修/可选分类合理——至少三分之一应该是必修或应修
5. 自审结束后,你**对任务拆解的边界比自审前更清晰**——能找到 PRD 作者没意识到的边界问题

### 4.2 如果问题清单是空的

两种可能:
- PRD 真的完美(极端罕见)
- 审得不够深(默认假设)

处理方式:**用子代理再审一遍**。子代理没有主上下文的认知偏差(你刚才读了 PRD,已经有了"它是对的"的印象),更能抓到矛盾。

## 5. 与 sdd-prd 阶段 1 的差异

| 维度 | sdd-prd 阶段 1 | sdd-phase 阶段 1 |
|---|---|---|
| 输入 | spec(用户故事/客户需求) | PRD(已写好) |
| 目标 | 找 spec 内部矛盾 | 找 PRD 影响任务拆解的矛盾 |
| 产物 | problem-list.md → ADR / 约束 | problem-list.md → 任务分解 |
| 重点 | 范围/数据/部署 | 范围/依赖/验收/引用 |

**关键差异**:sdd-prd 是从 spec 提纯,矛盾越早找越好;sdd-phase 是从 PRD 拆任务,矛盾影响的是"任务边界"和"依赖链"。

## 6. 常见漏审的模式

这些是容易漏的地方——因为它们不在"核心内容"里,而在边角:

- **PRD §1.3 成功指标**:往往用模糊词汇("提升效率"),与 §3 任务无直接对应
- **PRD §4 NFR 阈值**:经常散落,与 §3 任务验收不对应
- **PRD §7 集成需求**:常被忽略,但 Phase 任务依赖外部系统
- **PRD §10 风险与约束**:已被识别为风险但未说"应对任务由谁做"
