# 阶段 2 深度参考:任务分解(Task Breakdown)

> 加载时机:用户卡在任务粒度、不知道"多细的任务合适"、需要识别任务依赖时。
> 阶段 2 是 sdd-phase 的核心——任务清单质量决定 Phase 可执行性。

---

## 1. 任务分解的本质

阶段 2 做的事情可以用一句话概括:

> **把 PRD 边界的"功能需求"拆解为"可执行任务",每条任务 = 1 个工程师 1-3 天可完成 + 客观可验收。**

任务粒度不是越细越好,也不是越粗越好。**合适的粒度 = 1-3 天可完成 + 验收标准可判定**。

---

## 2. 任务粒度的 3 个级别

### 2.1 太粗(> 3 天)

**示例**:

```markdown
T001: 实现用户系统
```

**问题**:

- 无法追踪进度(做了 50% 算什么状态?)
- 无法识别依赖(数据库 schema?API 设计?前端?测试?没分)
- 无法判定"完成"(用户系统完成 = 什么?)

### 2.2 太细(< 0.5 天)

**示例**:

```markdown
T001: 写 users 表的 migration
T002: 写 users 表的 schema
T003: 写 users 表的索引
```

**问题**:

- 文档臃肿(10 个任务合并为 1 个更易管理)
- 工程师疲于"切任务"而非"做任务"
- 依赖链过度细化,无意义

### 2.3 合适(1-3 天)

**示例**:

```markdown
T001: 实现用户注册 API + 单元测试

- 包含:POST /api/v1/users 端点、参数校验、密码哈希、邮箱唯一性检查
- 验收:201/400/409 + bcrypt + 单元测试覆盖 ≥ 80%
- 工时:2 人天
```

**优势**:

- 进度可追踪(0%/50%/100%)
- 依赖可识别(数据库 schema 已完成 / [外部:邮件服务已就绪])
- 验收可判定(201/400/409 是客观 HTTP 状态码)

## 3. 任务分解的方法

### 3.1 方法 1:从 PRD §3 功能需求映射

**步骤**:

1. 列出 PRD §3 全部 P0/P1 功能点
2. 每个 P0 功能 → ≥1 个任务
3. 每个 P1 功能 → 1 个或多个任务(视复杂度)
4. P2 功能可选拆,若拆则单独标记

**示例**:

```markdown
PRD §3.1 功能清单:
| 用户注册 | 邮箱注册 | P0 |
| 用户注册 | 手机注册 | P1 |
| 用户登录 | 密码登录 | P0 |
| 用户登录 | 第三方登录 | P2 |

→ 任务:
T001: 实现邮箱注册 API(2 人天,P0)
T002: 实现手机注册 API(2 人天,P1)
T003: 实现密码登录 API(1 人天,P0)
T004: 实现用户登出 API(0.5 人天,P0)
T005: 第三方登录调研 + 原型(3 人天,P2)
```

### 3.2 方法 2:从技术层次切分

有些功能需要横切多个层次(数据库、API、前端、测试)。**任务按层次切分**比按功能切分更清晰。

**示例**:

```markdown
PRD §3.2.1 用户注册:
→ 任务(按层次):
T001: 设计 users 表 schema + migration(0.5 人天)
T002: 实现邮箱唯一性索引(0.5 人天,依赖 T001)
T003: 实现 POST /api/v1/users 端点(1.5 人天,依赖 T002)
T004: 实现邮件验证邮件发送(1 人天,依赖 T003)
T005: 实现前端注册表单(1 人天,依赖 T003)
T006: 集成测试 + E2E 测试(1 人天,依赖 T004, T005)
```

### 3.3 方法 3:从依赖链倒推

**步骤**:

1. 列出"必须最后做"的任务(部署、集成测试、上线)
2. 倒推"必须先做"的任务
3. 中间填充"可并行"的任务

**示例**:

```markdown
末梢任务:T020 上线(0.5 人天)
倒推:T019 E2E 测试 → T015-T018 全部 P0 功能 → T001-T010 基础设施
```

## 4. 任务属性设计

每条任务必须有 6 个属性(对齐 sdd-core §4.1 阶段文档模板):

| 属性         | 取值               | 示例                             |
| ------------ | ------------------ | -------------------------------- |
| **任务 ID**  | `T001`、`T002`...  | `T001`                           |
| **任务名称** | 动词开头           | "实现用户注册 API"               |
| **负责人**   | 占位(待定)         | `@待分配`                        |
| **预估工时** | 人天               | `2 人天`                         |
| **依赖**     | 任务 ID / 外部依赖 | `T001` / `[外部:邮件服务已就绪]` |
| **状态**     | sdd-core §4.4 三态 | `未开始`                         |

任务详情(对齐 sdd-core §4.1):

```markdown
#### T001: 实现用户注册 API

**任务描述**:
基于 PRD §3.2.1 用户注册功能,实现 POST /api/v1/users 端点。

**验收标准**:

- [ ] POST /api/v1/users 返回 201/400/409
- [ ] 密码使用 bcrypt 哈希(对齐 §4.2 NFR)
- [ ] 邮箱重复返回 409
- [ ] 单元测试覆盖率 ≥ 80%
- [ ] 集成测试通过

**涉及文档**:

- [PRD §3.2.1 用户注册](../prd/2026-06-23-user-authentication.md)
- [ADR-001 前后端选型](../architecture/decisions.md#adr-001)

**备注**:
密码哈希成本因子 = 12(对齐 §4.2 安全要求)
```

## 5. 任务依赖识别

### 5.1 依赖类型

| 类型         | 示例                   | 标识                    |
| ------------ | ---------------------- | ----------------------- |
| **强依赖**   | DB schema 必须先于 API | `T001 → T003`           |
| **弱依赖**   | 文档可后于实现         | `T003 → T010`           |
| **外部依赖** | 邮件服务、第三方 API   | `[外部:邮件服务已就绪]` |
| **并行任务** | 前端/后端各自独立      | `T005 ‖ T003`           |

### 5.2 依赖识别清单

- **数据库 schema** 必须先于 API
- **API 实现** 必须先于前端调用
- **单元测试** 可与实现并行,但**集成测试**必须后于实现
- **部署脚本** 必须后于所有功能完成
- **文档更新** 可后于实现(但归档前必须完成)

### 5.3 依赖闭环检查

每条任务的依赖必须:

- 存在(任务 ID 在清单中)
- 合理(无循环依赖 A→B→A)
- 显性化(写在属性中,不靠"心照不宣")

## 6. 任务粒度的反模式

### 6.1 任务粒度不统一

```markdown
T001: 设计数据库 schema(0.5 人天)
T002: 实现用户系统(5 人天)
T003: 写一个 if 分支(0.1 人天)
```

**问题**:无法估算整体工期;无法识别关键路径。

**修复**:统一粒度在 1-3 天,T002 拆为 5 个子任务。

### 6.2 验收标准不可测

```markdown
T001: 优化用户系统性能
验收:系统性能良好
```

**问题**:"良好"无客观判定;工程师和 reviewer 认知不同。

**修复**:

```markdown
T001: 优化用户系统性能
验收:

- [ ] POST /api/v1/users P99 延迟 ≤ 200ms(对齐 PRD §4.1)
- [ ] 100 并发用户下,无 5xx 错误
- [ ] 数据库查询 P99 ≤ 50ms
```

### 6.3 任务依赖未识别

```markdown
T001: 实现用户注册 API
T002: 编写注册接口的单元测试
```

**问题**:T002 依赖 T001,但未显性化——下游工程师不知道要先等 T001 完成。

**修复**:

```markdown
T002: 编写注册接口的单元测试
依赖:T001
```

### 6.4 任务覆盖度不够

```markdown
PRD §3 功能需求:8 个 P0 功能
任务清单:5 个任务
```

**问题**:8 个功能只拆了 5 个任务——必有遗漏。

**修复**:阶段 2 完成前,**必须交叉验证** PRD §3 全部 P0/P1 都有对应任务。

## 7. 与 sdd-prd 阶段 2(深审)的差异

| 维度 | sdd-prd 阶段 2        | sdd-phase 阶段 2    |
| ---- | --------------------- | ------------------- |
| 输入 | spec + 矛盾清单       | PRD + 边界问题清单  |
| 目标 | 质疑继承选型,产出 ADR | 拆任务,产出任务清单 |
| 产物 | adr-set.md            | task-breakdown.md   |
| 关注 | 选型是否最优          | 任务粒度/依赖/验收  |
| 涉及 | 架构/技术决策         | 工程实施            |

**关键差异**:sdd-prd 阶段 2 选"做什么技术",sdd-phase 阶段 2 选"怎么拆任务"。两者正交,不重复。
