# 阶段 3 深度参考:增量设计(里程碑、风险、依赖)

> 加载时机:用户卡在"里程碑怎么分"、"风险怎么识别"、"协作需求怎么列"。
> 阶段 3 在任务清单之上加**节奏感**——把零散任务组织为可推进的阶段。

---

## 1. 增量设计的本质

阶段 3 做的事情可以用一句话概括:

> **把"任务清单"组织为"可推进的阶段节奏"——里程碑给出时间锚点,风险暴露不确定性,协作明确责任。**

任务清单回答"做什么";阶段 3 回答"什么节奏做、谁配合、有什么风险"。

---

## 2. 里程碑设计

### 2.1 里程碑的本质

**里程碑 = 一组任务完成的检查点 + 可交付物**。

不是任务的简单聚合,而是**节奏锚点**——团队在某个时间点可以"暂停"并评估进度。

### 2.2 里程碑数量

**推荐 3-5 个**:

| 数量 | 适用                            |
| ---- | ------------------------------- |
| < 3  | 阶段任务极少(警告:可能粒度太粗) |
| 3-5  | **标准**(推荐)                  |
| > 5  | 节奏碎片化(警告:可能粒度太细)   |

### 2.3 里程碑的典型结构

```markdown
| 里程碑             | 日期       | 交付物                       | 状态   |
| ------------------ | ---------- | ---------------------------- | ------ |
| M0:环境就绪        | 2026-06-25 | DB schema + 部署脚本 + CI/CD | 未达成 |
| M1:后端 API 完成   | 2026-07-05 | 全部 P0 API 端点 + 单元测试  | 未达成 |
| M2:前端联调完成    | 2026-07-15 | 前端集成后端完成             | 未达成 |
| M3:集成测试 + 上线 | 2026-07-20 | E2E 通过 + 生产部署          | 未达成 |
```

### 2.4 里程碑设计的方法

**方法 1:从依赖链关键路径推**

```markdown
任务依赖图:
T001(T0+0.5) → T003(T0+2) → T006(T0+4) → T010(T0+6) → T020(上线 T0+7)

→ 关键路径 = 7 天
→ 里程碑:
M0(T0+0.5): T001 完成
M1(T0+4): T006 完成(后端 API ready)
M2(T0+6): T010 完成(集成测试 ready)
M3(T0+7): T020 完成(上线)
```

**方法 2:从交付物角度推**

```markdown
交付物:

- DB schema ready
- 后端 API ready
- 前端 ready
- 集成测试 ready
- 上线

→ 每个交付物 = 1 个里程碑
```

## 3. 风险识别

### 3.1 风险的 3 个类型

| 类型         | 示例                                    | 应对                             |
| ------------ | --------------------------------------- | -------------------------------- |
| **技术风险** | 关键人员未掌握 Rust / 第三方 API 不稳定 | 培训 / 备选方案 / 缓冲时间       |
| **资源风险** | 关键人员离职 / 工期冲突                 | 备份人员 / 任务并行 / 优先级重排 |
| **业务风险** | PRD 业务规则理解偏差                    | 早期 demo / 评审会议 / 文档澄清  |

### 3.2 风险属性

每条风险必须有 4 个属性(对齐 sdd-core §4 风险表):

```markdown
| 风险                          | 影响          | 概率 | 应对措施                            | 责任人           |
| ----------------------------- | ------------- | ---- | ----------------------------------- | ---------------- |
| 邮件服务集成延迟              | 高(阻塞 T004) | 中   | 提前 1 周联系供应商 / 准备 SES 备选 | @PM              |
| 关键后端工程师 @张三 离职     | 高(M1 延期)   | 低   | 任务双人配对 / 文档化               | @Tech Lead       |
| 用户注册流程与 PRD 理解不一致 | 中(可能返工)  | 中   | M1 前 demo 评审                     | @PM + @Tech Lead |
```

### 3.3 风险识别的反模式

**反模式 1:空泛风险**

```markdown
| 可能延期 | 高 | 中 | 加快进度 |
```

**问题**:"可能延期"不可操作;"加快进度"无具体动作。

**修复**:

```markdown
| 邮件服务集成延迟 | 高(阻塞 T004) | 中 | 提前 1 周联系供应商 / 准备 SES 备选 | @PM |
```

**反模式 2:风险无责任人**

```markdown
| 第三方 API 故障 | 高 | 低 | 联系供应商 |
```

**问题**:无责任人,风险无人跟进。

**修复**:加 `@Tech Lead` 或具体人名。

## 5. 阶段验收设计(对齐 sdd-core §4.1 §5)

### 5.1 阶段级完成标准

阶段验收回答"**整个阶段**是否完成",而非"单任务是否完成"。

```markdown
## 5. 验收

### 5.1 验收清单

- [ ] 全部 P0 任务已完成且验收标准通过
- [ ] 全部 P1 任务已完成或明确推迟理由
- [ ] 集成测试(E2E)通过率 100%
- [ ] 性能指标达到 PRD §4 NFR 阈值
- [ ] 安全审计通过(对齐 §4.2)
- [ ] 文档(API 文档、用户手册)已发布
- [ ] 部署到生产环境并灰度通过

### 5.2 与 PRD 验收的对应关系

- 阶段验收 → PRD §5 验收标准的子集
- 阶段未涵盖的验收项 → 需新阶段或下个 Phase
```

### 5.2 验收记录表

```markdown
| 验收项      | 验收人     | 验收日期   | 结果        | 备注                |
| ----------- | ---------- | ---------- | ----------- | ------------------- |
| P0 任务完成 | @Tech Lead | YYYY-MM-DD | 通过/不通过 |                     |
| 集成测试    | @QA        | YYYY-MM-DD | 通过/不通过 |                     |
| 性能 NFR    | @SRE       | YYYY-MM-DD | 通过/不通过 | P99 = 180ms ≤ 200ms |
```

## 6. 与 sdd-prd 阶段 3(增量设计)的差异

| 维度 | sdd-prd 阶段 3    | sdd-phase 阶段 3     |
| ---- | ----------------- | -------------------- |
| 输入 | spec + ADR        | PRD + 任务清单       |
| 目标 | 补全约束          | 组织里程碑/风险/协作 |
| 产物 | constraint-set.md | milestone-set.md     |
| 关注 | 实施卡点          | 节奏锚点             |
| 涉及 | 机制/约束         | 协作/时间            |

**关键差异**:sdd-prd 阶段 3 选"用什么机制";sdd-phase 阶段 3 选"什么节奏做"。
